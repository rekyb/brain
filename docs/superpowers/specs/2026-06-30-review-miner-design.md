# review-miner — Design Spec

**Date:** 2026-06-30
**Status:** Approved design, pending spec review
**Location:** `20-projects/review-miner/`

## Purpose

A product-agnostic Python CLI that **collects** product reviews from multiple
sources into a single clean CSV for manual analysis. It is a *collection* tool,
not an analysis tool — clustering, sentiment, and synthesis are done by the user
afterward in Claude/ChatGPT. This boundary is deliberate and keeps the tool
small, cheap, and low-maintenance.

The tool has zero product-specific knowledge baked in. Everything specific to a
project lives in an interactively-built, saved config. Reusing it on a new
project means answering a short set of prompts (or reusing a saved config) —
no code changes.

## Non-goals (out of scope for v1)

- Analysis, sentiment, clustering, or summarization of any kind
- Scheduling / continuous monitoring (run on demand only)
- A database (CSV files on disk are the storage)
- Incremental "only-new-since-last-run" pulls (each run is a fresh full pull)
- Any dashboard or GUI (CLI only)

## User flow

```
$ review-miner run

?  Project name:                 acme-lms

# If config/acme-lms.yaml already exists:
?  Reuse "acme-lms" config? [Y/n]    → Y skips all remaining prompts and runs

# Otherwise (or if "n"), the adaptive interactive wizard continues:
?  Output folder:                20-projects/acme-lms/research/review-mining
                                 (default derived from project name)
?  Which sources? [space toggle] ◉ App Store  ◉ Reddit  ◯ Google Play  ◯ G2  ◯ Capterra
   # only the selected sources prompt for their details:
?  App Store app IDs (csv):      123456789
?  Country:                      us
?  Reddit subreddits:            edtech, elearning
?  Reddit search queries:        acme lms, acme learning
?  Max reviews per source:       200

→ Saving config/acme-lms.yaml
→ Collecting…  App Store ✓ 187   Reddit ✓ 64
→ Wrote reviews_2026-06-30.csv + run-summary_2026-06-30.md
```

The wizard is **adaptive**: it only asks follow-up questions for the sources the
user selected. Assembled answers are validated by pydantic before any scraping
starts, so bad input (e.g. an empty app-ID list) fails at the prompt, not
mid-run.

## Architecture

The core idea is **pluggable, isolated connectors** behind one interface, so a
broken connector (markup change, IP block, timeout) never takes down the others,
and adding a source later is a new file with no changes elsewhere.

```
20-projects/review-miner/
├── pyproject.toml
├── .env.example               # REDDIT_CLIENT_ID / SECRET / USER_AGENT
├── config/
│   └── <project>.yaml         # auto-generated by the wizard, one per project
├── reviewminer/
│   ├── cli.py                 # Typer `run` command; orchestrates wizard → collect → write
│   ├── wizard.py              # questionary prompts, adaptive to selected sources
│   ├── config.py              # load/save/validate YAML (pydantic models)
│   ├── models.py              # Review schema — single source of truth
│   ├── normalize.py           # map each source's fields onto the shared schema
│   ├── dedupe.py              # drop near-duplicate reviews by text hash
│   ├── output.py              # write CSV + markdown run-summary
│   └── connectors/
│       ├── base.py            # Connector ABC: fetch(spec) -> list[Review]
│       ├── appstore.py        # app-store-scraper (library)
│       ├── googleplay.py      # google-play-scraper (library)
│       ├── reddit.py          # PRAW (official API)
│       ├── g2.py              # Playwright (DIY scrape)
│       └── capterra.py        # Playwright (DIY scrape)
└── output/                    # only if no vault output_dir is set
```

### Data flow

```
1. WIZARD / LOAD   build or load a validated RunConfig
2. COLLECT         for each selected source, call its connector
                     - clean libs/API: appstore, googleplay, reddit
                     - DIY Playwright:  g2, capterra
                     - a failing connector logs a warning, returns [], run continues
3. NORMALIZE       map every source's fields onto the shared Review schema
4. DEDUPE          drop near-identical reviews (same text across pulls)
5. WRITE           reviews_<date>.csv + run-summary_<date>.md into output_dir
```

### Connector interface

Every connector implements the same ABC:

```python
class Connector(ABC):
    name: str
    @abstractmethod
    def fetch(self, spec: SourceSpec) -> list[Review]: ...
```

`fetch` must never raise out to the orchestrator: it catches its own errors,
logs a warning, and returns what it has (possibly an empty list). The
orchestrator records per-connector status for the run-summary.

## Data model

`Review` (single normalized schema across all sources) → CSV columns:

```
product, source, review_id, date, rating, title, body,
author, url, helpful_count, language, scraped_at
```

- `source` ∈ {appstore, googleplay, reddit, g2, capterra}
- `rating` normalized to a number where available; null for sources without one
  (e.g. some Reddit posts)
- `review_id` source-native id; used with `body` hash for dedupe

## Output artifacts

Written into `output_dir` (default `20-projects/<project>/research/review-mining/`):

1. **`reviews_<date>.csv`** — one row per review, schema above. The deliverable.
2. **`run-summary_<date>.md`** — date, config used, and a per-source count table,
   e.g. `App Store: 187 ✓ | Reddit: 64 ✓ | G2: 0 ✗ blocked`. Gives the vault a
   readable record and surfaces silent scrape failures at a glance.

## Configuration & secrets

- **Per-project config** (`config/<project>.yaml`) is auto-generated by the
  wizard on first run and reused on later runs. Example:

  ```yaml
  project: acme-lms
  output_dir: "20-projects/acme-lms/research/review-mining"
  max_per_source: 200
  sources:
    appstore:   { apps: ["123456789"], country: "us" }
    reddit:     { subreddits: ["edtech"], queries: ["acme lms"] }
  ```

- **Reddit API credentials** are a reusable secret, not project-specific. They
  live in `.env` (`REDDIT_CLIENT_ID`, `REDDIT_CLIENT_SECRET`,
  `REDDIT_USER_AGENT`), read via env vars. `.env.example` documents them. The
  wizard checks they exist when Reddit is selected and warns if missing.

## Error handling

- **Per-connector isolation:** a connector failure is logged and skipped; the run
  always completes and writes whatever was collected.
- **Config validation:** pydantic rejects malformed input at the wizard / on load,
  with a clear message, before any network calls.
- **Politeness on scraped sites (G2/Capterra):** `time.sleep` + random jitter
  between requests and rotating user-agents to reduce blocks. DIY scrapers are
  expected to be the brittle surface; they are isolated so one site can be patched
  without touching the rest.

## Tech stack

| Layer | Choice | Why |
|-------|--------|-----|
| Runtime | Python 3.11+, `uv` | Fast, simple dependency management |
| CLI | Typer | Clean command/flags, good help |
| Prompts | questionary | Checkboxes, selects, validated text input |
| Validation | pydantic | Fail-fast config validation |
| App Store (iOS) | app-store-scraper | Maintained library, not brittle DIY |
| Google Play | google-play-scraper | Maintained library |
| Reddit | PRAW | Official API, ToS-clean |
| G2 / Capterra | Playwright | Handles JS-heavy pages (the DIY part) |
| Data wrangling | pandas | Normalize, dedupe, write CSV |

Only G2/Capterra are true DIY scraping; App Store, Google Play, and Reddit use
clean libraries/APIs, so the brittle, high-maintenance surface is small and
isolated.

## Testing

- **Connectors:** unit tests with recorded/fixture responses (no live network in
  CI); each connector's normalize step asserted against the shared schema.
- **normalize / dedupe:** pure-function unit tests on representative messy input.
- **config:** round-trip save/load and validation-failure cases.
- **wizard:** logic for which prompts appear given a source selection (the
  adaptive behavior) tested with questionary's prompt objects stubbed.
- Live scraping is verified manually, not in CI, since it depends on third-party
  sites.

## Legal / maintenance note

DIY scraping of G2/Capterra is ToS-sensitive and breaks when sites change markup
or block IPs. This is an accepted, user-chosen tradeoff. The design contains the
risk to two isolated files and makes failures visible in the run-summary rather
than letting them corrupt or silently shrink the dataset.
