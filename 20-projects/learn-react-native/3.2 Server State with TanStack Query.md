# 3.2 Server State with TanStack Query

> [!abstract] TL;DR
> TanStack Query (v5+) replaces local state managers with a declarative server-state cache. It manages network data-fetching, loading/error states, automatic caching, garbage collection, and background synchronization, allowing React Native developers to avoid syncing remote data into local state stores.

## Digest

In client-side development, **client state** (e.g., UI theme, route parameters, modal open states) and **server state** (e.g., lists of habits fetched from a database, user profile details) are fundamentally different. 

Server state is asynchronously fetched, owned remotely, mutable by external agents, and can easily go out-of-sync. When using general-purpose state managers like Redux or Zustand for server state, developers end up writing excessive boilerplate—managing loaders, handle-error actions, caching logic, and request-retry behaviors manually. 

TanStack Query treats remote data as a cache. It provides simple hooks that handle caching, automatic retries, background updates, and request deduplication out-of-the-box.

### TanStack Query vs. Global State Managers

| Feature | Zustand / Redux | TanStack Query (v5+) |
| :--- | :--- | :--- |
| **Primary Focus** | Client State (local UI configuration). | Server State (cached remote database). |
| **Async Operations** | Requires thunks, sagas, or manual async actions. | Declarative hooks wrapping fetch/mutation promises. |
| **Cache Invalidation** | Manual state updates or action dispatches. | Declarative key invalidation triggers automatic background refetch. |
| **Status Helpers** | Explicit boolean states (e.g. `isLoading`, `error`) must be coded manually. | Native state properties like `isPending`, `isFetching`, `isError`, `error`. |
| **GC / Lifecycles** | Retained in memory until the application is closed or manually cleared. | Automatically garbage-collects unused data after a configurable threshold. |

### Query Keys Design

Query keys are the foundation of caching in TanStack Query. In v5, they **must be arrays**. They act as unique hashes to index the cache.

Keys should be structured **hierarchically**, moving from general resources to specific records:

- **List query**: `['habits', 'list']`
- **Filtered list query**: `['habits', 'list', { filter: 'Health' }]`
- **Single item query**: `['habits', 'detail', habitId]`

This hierarchical design enables targeted cache invalidations. For instance, calling `queryClient.invalidateQueries({ queryKey: ['habits'] })` will invalidate *all* query keys starting with `['habits']`. Conversely, invalidating `['habits', 'list']` will refresh list data while leaving individual habit detail views intact.

### Fetching with `useQuery`

The `useQuery` hook connects a query key to an asynchronous fetch function. 

```tsx
// Concept Example: Fetching user habits
import { useQuery } from '@tanstack/react-query';

interface Habit {
  id: string;
  name: string;
}

const fetchHabits = async (): Promise<Habit[]> => {
  const response = await fetch('https://api.example.com/habits');
  if (!response.ok) throw new Error('Network error');
  return response.json();
};

export function useHabits() {
  return useQuery({
    queryKey: ['habits', 'list'],
    queryFn: fetchHabits,
    staleTime: 1000 * 60 * 5, // Data remains fresh for 5 minutes
    gcTime: 1000 * 60 * 10,   // Unused data remains in cache for 10 minutes (formerly cacheTime)
  });
}
```

#### Status Properties in v5

In TanStack Query v5, query states are represented by two primary statuses:
1. `status` (`'pending' | 'error' | 'success'`): Reflects the state of the *data*.
2. `fetchStatus` (`'fetching' | 'paused' | 'idle'`): Reflects the state of the *query execution*.

- **`isPending`**: True when there is no cached data yet and the query has not resolved.
- **`isFetching`**: True whenever the query function is executing, including background refetches when stale data is already shown.
- **`isLoading`**: Equivalent to `isPending && isFetching`. Represents the first-time fetch.
- **`isError` / `error`**: Triggered if the query function rejects/throws an exception.

### Cache Lifecycle: Stale vs. Garbage Collection

Understanding the cache lifecycle prevents unnecessary requests:

- **`staleTime`**: The duration (in milliseconds) before cached data is considered old. While data is "fresh," mounting a component will read from the cache *without* sending a network request. Once data is "stale," mounting the component will still serve the cached data immediately to keep the UI snappy, but it will trigger a background refetch.
- **`gcTime` (Garbage Collection Time)**: The duration (in milliseconds) that unused data remains in memory. When a screen containing a query unmounts, the query becomes "inactive." After `gcTime` expires, the cache entry is garbage-collected.

### Mutations and Cache Invalidation

To write or modify data, use `useMutation`. Once a mutation succeeds, you invalidate corresponding query keys to trigger a refetch and update the UI.

```tsx
// Concept Example: Mutating server state
import { useMutation, useQueryClient } from '@tanstack/react-query';

const addHabitApi = async (name: string) => {
  const response = await fetch('https://api.example.com/habits', {
    method: 'POST',
    body: JSON.stringify({ name }),
  });
  return response.json();
};

export function useCreateHabit() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: addHabitApi,
    onSuccess: () => {
      // Invalidate the cache to trigger automatic refetches
      queryClient.invalidateQueries({ queryKey: ['habits', 'list'] });
    },
  });
}
```

## Drill

Integrate TanStack Query to manage habit data dynamically, replacing local React state with server caching.

### Task Description

1. Refactor the custom hook `useHabitViewModel` created in the previous chapter to fetch list data using `useQuery`.
2. The query should call a mock REST endpoint (using a mock API helper class that returns promises with artificially delayed responses).
3. Implement habit creation inside `useHabitViewModel` using `useMutation`.
4. In the mutation's `onSuccess` callback, trigger an automatic cache invalidation using `queryClient.invalidateQueries` to refresh the habits list.
5. In your stateless UI component, handle and render the query states:
   - Display a spinner or placeholder when the query `isPending` is true.
   - Display an error message if the query `isError` is true.
   - Map the successful `data` array into the list rendering logic.
   - Disable the "Add Habit" input or button during mutation execution (`isPending` of the mutation).

> [!example] Success criteria
> - [ ] Query states (loading/pending, error, and success) are explicitly handled in the visual UI.
> - [ ] Additions are routed through `useMutation`, and successful calls invalidate the query keys.
> - [ ] The custom hook includes a structured explanation of its configured `staleTime` and `gcTime` settings.
> - [ ] No manual local state array (`useState`) is used to store the list of habits; the hook relies entirely on TanStack Query's cache.
