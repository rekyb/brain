# 2.1 App Structure and Stacks

> [!abstract] TL;DR
> Expo Router v3+ brings file-based routing to React Native, mapping your project's directory structure in `app/` directly to native mobile screens. By mirroring Next.js App Router conventions, it uses layout files (`_layout.tsx`) to manage native stack navigators and supports both declarative and imperative navigation with auto-generated type safety.

## Digest

In classic React Native development, navigation was configured imperatively using React Navigation, requiring developers to manually define screen maps, state configurations, and nesting containers in a monolithic root file. Expo Router replaces this with a file-based routing system built on top of React Navigation, bringing web-like route conventions to mobile.

### Comparing Expo Router to Next.js

If you have built applications with Next.js (App Router), the directory-based routing model will feel immediately familiar. However, because mobile applications do not reload HTML pages and instead manage a stack of native view controllers, the routing mechanics differ:

| Concept | Next.js App Router | Expo Router (v3+) | Mobile Significance |
| :--- | :--- | :--- | :--- |
| **Root Directory** | `app/` | `app/` | Contains all routes and layouts. |
| **Page Entry** | `page.tsx` | `index.tsx` (or `[name].tsx`) | Renders the primary UI for that route segment. |
| **Layouts** | `layout.tsx` | `_layout.tsx` | Wraps children, but in Expo Router it configures the *navigator container* (e.g. Stack or Tabs). |
| **History** | Browser URL history | Native Stack / History stack | Navigating pushes views onto a visual stack; going back pops them. |
| **NotFound** | `not-found.tsx` | `+not-found.tsx` | Fallback screen for missing routes. |

### Anatomy of the `app/` Directory

A typical Expo Router project structure looks like this:

```
my-app/
├── app/
│   ├── _layout.tsx        # Root layout (defines Stack navigator)
│   ├── index.tsx          # Home screen (path: /)
│   ├── details.tsx        # Details screen (path: /details)
│   ├── +not-found.tsx     # Fallback route
│   └── (tabs)/            # Route group (omitted from URL path)
│       ├── _layout.tsx
│       └── feed.tsx       # Path: /feed
├── app.json
└── package.json
```

### Layouts and the Stack Navigator

In Expo Router, `_layout.tsx` files are used to wrap sub-pages. Instead of rendering a simple container `<div />` wrapper like on the web, a layout in Expo Router instantiates a **Navigator component** that handles transitions and header rendering.

The most basic layout uses a `<Stack>` navigator, which mimics iOS `UINavigationController` and Android `Fragment` stacks. When you navigate to a new screen, it slides in from the right (iOS) or fades/slides up (Android).

Here is a typical root `app/_layout.tsx` using `<Stack>`:

```tsx
import { Stack } from 'expo-router';

export default function RootLayout() {
  return (
    <Stack
      screenOptions={{
        headerStyle: {
          backgroundColor: '#1c1c1e', // Dark header background
        },
        headerTintColor: '#ffffff',     // White back buttons and text
        headerTitleStyle: {
          fontWeight: 'bold',
        },
      }}
    >
      {/* 
        You can explicitly configure individual screens here, 
        or let Expo Router infer them automatically from the file structure.
      */}
      <Stack.Screen 
        name="index" 
        options={{ 
          title: 'Overview' 
        }} 
      />
      <Stack.Screen 
        name="details" 
        options={{ 
          title: 'Details',
          headerBackTitle: 'Back', // iOS-only back button text
        }} 
      />
    </Stack>
  );
}
```

### Configuring Headers Dynamically

Headers can be customized in two ways:
1. **Statically in the Layout**: Using `<Stack.Screen options={{...}} />` inside the layout file (as shown above).
2. **Dynamically inside the Screen Component**: Using the `<Stack.Screen />` component inside the page file itself, which allows you to tie header configuration directly to local screen state (e.g. displaying a loading indicator or adding action buttons):

```tsx
// app/details.tsx
import { useState } from 'react';
import { View, Text, Button } from 'react-native';
import { Stack } from 'expo-router';

export default function DetailsScreen() {
  const [count, setCount] = useState(0);

  return (
    <View style={{ flex: 1, justifyContent: 'center', alignItems: 'center' }}>
      {/* Configure header options dynamically based on component state */}
      <Stack.Screen 
        options={{
          title: `Details (Pressed ${count}x)`,
          headerRight: () => (
            <Button 
              onPress={() => setCount(c => c + 1)} 
              title="Add" 
              color="#0a84ff" 
            />
          ),
        }} 
      />
      <Text>Details Content Screen</Text>
    </View>
  );
}
```

### Navigating: `<Link>` and `router`

Expo Router provides two main mechanisms for navigating between screens:

#### 1. The Declarative `<Link>` Component
Perfect for static buttons, list rows, or text links. Under the hood, it wraps the child element in a touch responder.

```tsx
import { Link } from 'expo-router';
import { Text } from 'react-native';

export default function HomeScreen() {
  return (
    // Link component takes an href path
    <Link href="/details" asChild>
      {/* The 'asChild' prop ensures the touch handler is passed down to Pressable/TouchableOpacity */}
      <Text style={{ color: '#0a84ff', fontSize: 18 }}>View Details</Text>
    </Link>
  );
}
```

#### 2. The Imperative `router` Object
Use this inside functions, callbacks, or gesture handlers where declarative navigation is not possible:

```tsx
import { View, Button } from 'react-native';
import { router } from 'expo-router';

export default function HomeScreen() {
  const handleCheckout = () => {
    // Perform some logic (e.g. call api, validate data)
    const success = true;
    if (success) {
      router.push('/details'); // Push onto the navigation stack
      // router.replace('/details'); // Replace the current screen in-place
      // router.back(); // Go back to the previous screen
    }
  };

  return (
    <View>
      <Button title="Complete Order" onPress={handleCheckout} />
    </View>
  );
}
```

### Type-Safe Routing

One of Expo Router's most powerful features is auto-generated type safety. When your development server runs, it parses the `app/` directory structure and outputs TypeScript definitions to `.expo/types/`. 

This generates typed definitions for routes, meaning:
1. `Link` components and `router.push()` calls will enforce valid paths in IDE autocomplete.
2. Typos in paths (e.g., `router.push('/detals')`) will trigger TypeScript errors.
3. Path configurations are mapped inside `expo-env.d.ts`, which must be included in your project root.

To enable type-safety:
- Keep the `expo-env.d.ts` in your root.
- Ensure `"extends": "expo/tsconfig.base"` is in your `tsconfig.json`.
- Run `npx expo start` to compile the type registry.

## Drill

Set up a basic Stack navigation flow that links a summary list to a detailed item view, featuring customized native headers.

### Task Description
1. Create a root index screen that displays a list of mock items (e.g. habits or tasks).
2. Configure a `_layout.tsx` file that instantiates a `<Stack>` navigator with customized headers (e.g. specific background color and font styling).
3. Create a details screen that displays individual item info.
4. Implement navigation from the list screen to the details screen using:
   - Declarative `<Link>` component with `asChild` wrapping a touchable element.
   - Imperative `router` object inside a custom press handler.
5. Dynamically configure the details screen header options to show a custom title and a right header action button (like an edit or favorite icon) using `<Stack.Screen />` inside the details component.

> [!example] Success criteria
> - [ ] A root Stack navigator layout is set up correctly in `_layout.tsx`.
> - [ ] Clicking a list item successfully transitions/slides to the details screen.
> - [ ] Navigation is implemented using both the `<Link>` component and the `router` API.
> - [ ] Custom background styling is applied to the navigation header at the layout level.
> - [ ] The detail view uses `<Stack.Screen>` to dynamically customize its header title and render a custom right-aligned button.
> - [ ] Expo Router type-safe parameters are utilized to avoid arbitrary string routes.
