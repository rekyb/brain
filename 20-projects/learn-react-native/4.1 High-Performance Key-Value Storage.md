# 4.1 High-Performance Key-Value Storage

> [!abstract] TL;DR
> Modern React Native applications leverage JSI (JavaScript Interface) to perform fully synchronous, high-speed key-value persistence. By replacing the legacy asynchronous Bridge-based `AsyncStorage` with C++ backed libraries like `react-native-mmkv`, developers can eliminate layout flashes on app startup and access persistent storage synchronously on the main JavaScript execution thread.

## Digest

In React Native development, persisting user preferences, themes, configuration settings, or small authentication tokens requires a key-value store. Historically, the standard tool was `@react-native-async-storage/async-storage`. However, the old architecture introduces latency and state synchronization challenges.

### AsyncStorage (Legacy Bridge-Based)

`AsyncStorage` is built on React Native's legacy bridge architecture. Every storage operation must cross the bridge:

```mermaid
graph LR
    JS[JS Thread] -->|1. Serialize JSON| Bridge[React Native Bridge]
    Bridge -->|2. Asynchronous Queue| Native[Native OS SQLite/Preferences]
    Native -->|3. Read/Write Disk| Bridge
    Bridge -->|4. Deserialize JSON| JS
```

This design has significant drawbacks:
1. **Asynchronous API**: Everything returns a `Promise`, forcing components or hooks to manage loading states (e.g., `isLoading = true`) even for small booleans.
2. **Bridge Bottleneck**: High frequency read/write operations clog the single-threaded asynchronous queue, delaying UI rendering or touch handlers.
3. **App Boot Flashing**: On startup, the UI renders before `AsyncStorage` resolves the saved theme or preference. This causes a jarring visual layout flash or theme jump (e.g., light theme displaying for 100ms before dark theme is loaded).

---

### MMKV and the JavaScript Interface (JSI)

Modern React Native uses **JSI (JavaScript Interface)**. JSI is a lightweight API that allows JavaScript code to hold direct references to C++ host objects and invoke their methods synchronously.

`react-native-mmkv` is a C++ wrapper around WeChat's MMKV storage engine. It binds directly to the JS runtime via JSI:

```mermaid
graph LR
    JS[JS Runtime] -->|Direct Synchronous C++ Call| JSI[JSI Layer]
    JSI -->|In-Memory Map| MMKV[MMKV Engine]
    MMKV -->|Kernel mmap| Disk[Disk Storage]
```

#### Why JSI Enables Synchronous Access Safely
MMKV uses the operating system's `mmap` mechanism. `mmap` maps a file on disk directly into the virtual memory of the application process. 
- When the JS code calls `storage.getString('theme')`, JSI routes the call instantly to C++.
- C++ reads the memory-mapped space directly.
- The operating system kernel automatically manages loading blocks from disk and flushing dirty pages back to the storage device.
- Because memory reads are CPU-trivial (microsecond range), the operation executes synchronously without blocking the JS thread.

#### Comparison Table

| Feature | `AsyncStorage` | `react-native-mmkv` |
| :--- | :--- | :--- |
| **Architecture** | Legacy Bridge (JSON Messages) | Modern JSI (C++ Direct Calls) |
| **Access Pattern** | Asynchronous (`Promise`) | Synchronous (Immediate) |
| **Performance** | Slow (Serialization + Bridge overhead) | Extremely Fast (Native memory access) |
| **Data Types** | Strings only (Requires JSON serialization) | Strings, Booleans, Numbers, Buffers |
| **Startup Flashing** | Common (Requires loading screens to hide) | Avoided (Values read instantly on boot) |

### Usage Patterns

#### 1. Initialization
MMKV instances can be instantiated anywhere in the app, including globally:

```typescript
import { MMKV } from 'react-native-mmkv';

export const storage = new MMKV({
  id: 'app-storage',
  // Optional: encryptionKey: 'some-secure-key'
});
```

#### 2. Synchronous Reading and Writing
Writing and reading primitives is clean and does not require `async`/`await`:

```typescript
// Writing values
storage.set('username', 'alex_dev');
storage.set('isDarkMode', true);
storage.set('loginCount', 42);

// Reading values (instant, synchronous)
const username = storage.getString('username'); // 'alex_dev' | undefined
const isDark = storage.getBoolean('isDarkMode'); // true | false (default false if unset)
const count = storage.getNumber('loginCount'); // 42 | undefined

// Deleting keys
storage.delete('username');
```

#### 3. Storing Complex Objects
Although MMKV supports raw primitives, objects must be serialized to JSON strings. Since operations are synchronous, we can write helper accessors easily:

```typescript
interface UserProfile {
  id: string;
  email: string;
}

export function saveProfile(profile: UserProfile): void {
  storage.set('user_profile', JSON.stringify(profile));
}

export function getProfile(): UserProfile | null {
  const json = storage.getString('user_profile');
  if (!json) return null;
  try {
    return JSON.parse(json) as UserProfile;
  } catch {
    return null;
  }
}
```

## Drill

Implement a theme provider and user preference store using MMKV. Write code to verify preferences load instantly on app boot without layout flashes.

### Task Description

1. Install `react-native-mmkv` in a clean React Native / Expo environment.
2. Initialize an MMKV storage instance specifically designated for app preferences.
3. Design a `ThemeProvider` context and a `useTheme` hook. The provider should hold the current theme state (e.g., `'light'` or `'dark'`).
4. Read the initial theme value from MMKV directly inside the state initializer of your context provider (synchronously).
5. Expose a toggle method `toggleTheme()` that modifies the react state and updates the persisted value in MMKV simultaneously.
6. Verify that on application boot (or simulated refresh), the theme state is resolved on the first visual frame, showing that there is no layout jump or flash from a default fallback value.

> [!example] Success criteria
> - [ ] Theme settings and user preferences persist correctly across application restarts.
> - [ ] Read operations from MMKV are executed fully synchronously (no use of `await` or `.then` callbacks).
> - [ ] Documentation or code verification demonstrates how visual layout flashes are prevented on app startup by initializing state directly from the synchronous MMKV check.
