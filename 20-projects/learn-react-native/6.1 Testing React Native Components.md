# 6.1 Testing React Native Components

> [!abstract] TL;DR
> Testing in React Native leverages Jest as the runner and React Native Testing Library (RNTL) for component integration and hook unit testing. Unlike web testing, React Native testing requires specialized configurations to ignore native binary paths and mock native dependencies (like MMKV's JSI bindings, SQLite database contexts, and Expo Router navigation). Modern testing avoids deprecated hook libraries in favor of RNTL's native `renderHook` and uses query hierarchies like `getBy`, `queryBy`, and `findBy` combined with async utilities like `waitFor` to simulate user interactions accurately.

## Digest

For a web React developer, transitioning to testing React Native components involves moving from DOM-centric assertions (like checking elements by CSS selectors or HTML tags like `<div>` and `<button>`) to native-element assertions (like `<View>`, `<Text>`, and `<TextInput>`). Furthermore, because React Native projects include JSI (JavaScript Interface) modules and binary libraries that depend on mobile OS internals, tests will fail immediately in a Node.js/Jest test environment unless these native dependencies are systematically mocked.

---

### Jest Configuration for React Native

To execute React Native code within Node.js, Jest needs to know how to transform React Native syntax and ignore compiled native binary modules. Expo simplifies this with the `jest-expo` preset.

#### Configuration Structure (`jest.config.js`)
A standard modern configuration is configured as follows:

```javascript
module.exports = {
  preset: 'jest-expo',
  transformIgnorePatterns: [
    'node_modules/(?!((jest-)?react-native|@react-native(-community)?)|expo(nent)?|@expo(nent)?/.*|@expo-google-fonts/.*|react-navigation|@react-navigation/.*|@unimodules/.*|unimodules|sentry-expo|native-base|react-native-svg)',
  ],
  setupFilesAfterEnv: ['@testing-library/react-native/extend-expect'],
  setupFiles: ['./jest.setup.js'],
};
```

*   **`preset: 'jest-expo'`**: Directs Jest to load custom environment mock settings for Expo's native modules.
*   **`transformIgnorePatterns`**: A regex whitelist instructing Babel to transform ES6 code inside third-party React Native libraries while ignoring pure Node packages.
*   **`setupFiles`**: Runs custom scripts (like mocking APIs) before each test suite executes.

### Mocking Native Modules

React Native code that communicates via the JS-Native Bridge or JSI (C++ direct bindings) cannot run in a Node environment. If a component imports an active SQLite database or an MMKV store, it will crash Jest.

#### 1. Mocking `react-native-mmkv`
Because `react-native-mmkv` uses synchronous JSI bindings instead of the standard Bridge, Jest cannot find the native methods. We mock MMKV with an in-memory JS store:

```javascript
// jest.setup.js
jest.mock('react-native-mmkv', () => {
  return {
    MMKV: jest.fn().mockImplementation(() => {
      let store = new Map();
      return {
        set: jest.fn((key, value) => {
          store.set(key, String(value));
        }),
        getString: jest.fn((key) => store.get(key)),
        getNumber: jest.fn((key) => {
          const val = store.get(key);
          return val ? Number(val) : undefined;
        }),
        getBoolean: jest.fn((key) => {
          const val = store.get(key);
          return val ? val === 'true' : undefined;
        }),
        delete: jest.fn((key) => {
          store.delete(key);
        }),
        clearAll: jest.fn(() => {
          store.clear();
        }),
      };
    }),
  };
});
```

#### 2. Mocking Modern `expo-sqlite` (SDK 51+)
Modern SQLite uses synchronous and asynchronous execution APIs. Mocking it requires matching the object returned by `openDatabaseSync()`:

```javascript
// jest.setup.js
jest.mock('expo-sqlite', () => {
  return {
    openDatabaseSync: jest.fn(() => ({
      execSync: jest.fn(),
      runSync: jest.fn(() => ({ lastInsertRowId: 1, changes: 1 })),
      getFirstSync: jest.fn(),
      getAllSync: jest.fn(() => []),
      execAsync: jest.fn().mockResolvedValue(undefined),
      runAsync: jest.fn().mockResolvedValue({ lastInsertRowId: 1, changes: 1 }),
      getFirstAsync: jest.fn().mockResolvedValue(null),
      getAllAsync: jest.fn().mockResolvedValue([]),
    })),
  };
});
```

#### 3. Mocking `expo-router`
Components that contain buttons navigating to other screens need routing mocks to prevent route-resolution errors and allow assertion of navigation calls:

```javascript
// jest.setup.js
jest.mock('expo-router', () => {
  const actual = jest.requireActual('expo-router');
  return {
    ...actual,
    useRouter: () => ({
      push: jest.fn(),
      replace: jest.fn(),
      back: jest.fn(),
    }),
    useLocalSearchParams: () => ({}),
    useGlobalSearchParams: () => ({}),
  };
});
```

### Isolated Hook Testing with `renderHook`

Hooks encapsulate local state, async effects, and external dependencies. Rather than mounting a dummy visual component to inspect hook behavior, we test hooks in isolation.

> [!WARNING]
> Do not use `@testing-library/react-hooks`. It is **deprecated** and incompatible with React 18+. Modern hook testing is built directly into React Native Testing Library (`@testing-library/react-native`).

#### Example Hook Test
When testing hooks, state updates must be wrapped in `act()` to ensure that updates are applied before assertions are evaluated.

```tsx
import { renderHook, act } from '@testing-library/react-native';
import { useCounter } from './useCounter';

describe('useCounter hook', () => {
  it('should initialize with default value and increment count', () => {
    const { result } = renderHook(() => useCounter({ initialValue: 5 }));

    expect(result.current.count).toBe(5);

    act(() => {
      result.current.increment();
    });

    expect(result.current.count).toBe(6);
  });
});
```

### Component Integration Testing

RNTL exposes a `render` function that returns utilities to query the virtual tree, trigger user events, and await updates.

#### Query Methods Hierarchy
RNTL uses a prefix taxonomy to find native layout structures:

| Prefix | Behavior | Match Count | Throws Error? | Returns Promise? |
| :--- | :--- | :--- | :--- | :--- |
| **`getBy*`** | Finds matching element | Exact 1 | Yes (if 0 or >1 found) | No |
| **`queryBy*`** | Finds matching element | 0 or 1 | Yes (if >1 found) | No (returns `null` if 0) |
| **`findBy*`** | Resolves when element appears | Exact 1 | Yes (after timeout) | Yes |

*   **Preferred Finders**: Query elements using accessibility roles (`getByRole`) or user-facing text content (`getByText`). Use test identifiers (`getByTestId`) as an escape hatch for dynamic layouts.

#### Firing Events and Async Handling
Unlike the DOM where you fire generic native events, React Native uses proprietary event props (like `onPress` and `onChangeText`). RNTL's `fireEvent` handles this translation:

```tsx
import React from 'react';
import { render, fireEvent, waitFor } from '@testing-library/react-native';
import { HabitListScreen } from './HabitListScreen';

describe('<HabitListScreen />', () => {
  it('allows the user to type a habit name and add it to the list', async () => {
    const { getByPlaceholderText, getByText, queryByText } = render(
      <HabitListScreen />
    );

    // Assert initial state
    expect(queryByText('Read Books')).toBeNull();

    // Type text into the text field
    const input = getByPlaceholderText('Add new habit...');
    fireEvent.changeText(input, 'Read Books');

    // Press the submit button
    const submitButton = getByText('Submit');
    fireEvent.press(submitButton);

    // Await layout update or async DB persistence
    await waitFor(() => {
      expect(getByText('Read Books')).toBeTruthy();
    });
  });
});
```
