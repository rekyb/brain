# 0.1 Modern React Native and Expo landscape

> [!abstract] TL;DR
> Modern React Native development has shifted from bare CLI projects to managed Expo workflows. Through "prebuild," Expo dynamically generates ephemeral native project directories (`ios/` and `android/`) from a single configuration file (`app.json`), enabling rapid development, clean Git histories, and custom native extensions via Development Clients.

## Digest

For years, React Native developers faced a stark choice: use the rigid, feature-limited "Managed" Expo workflow (Expo Go) or eject to the "Bare" CLI workflow, managing complex `ios/` and `android/` directories containing millions of lines of generated, platform-specific code. 

Today, that dichotomy is obsolete. Modern React Native is **Managed Expo**. Expo SDK 51+ and Expo Router v3+ provide a unified workflow built around dynamic native code generation (prebuild) and custom Development Clients.

### The Magic of Expo Prebuild

In the classic bare workflow, you maintained the `/ios` and `/android` directories manually. Adding a package with native dependencies required modifying CocoaPods, Gradle files, and Android Manifests. If you upgraded React Native, you had to manually apply complex git diffs to these native folders.

With **Expo Prebuild** (`npx expo prebuild`), these directories are treated as ephemeral build targets—conceptually similar to the `/dist` or `/build` directory in web development. You do not check `ios/` and `android/` into version control. Instead, you specify your app configuration in `app.json` (such as native permissions, display name, bundle IDs, and third-party native configurations via **Config Plugins**). When you run prebuild, Expo generates the native folders on the fly. 

Upgrading the React Native version is as simple as bumping the Expo SDK version in `package.json` and running `prebuild` again.

### Expo Go vs. Development Clients

To run and debug your app locally, you need a native runner.

1. **Expo Go**: A pre-built native wrapper app available on the App Store and Google Play. It contains a fixed set of native SDKs. It is excellent for quick prototyping because you don't need to install Xcode or Android Studio. However, **you cannot add custom native modules, custom Swift/Kotlin code, or third-party libraries that require custom native setup**.
2. **Development Clients** (`expo-dev-client`): A custom native app built specifically for your project. By installing `expo-dev-client` into your project and building it locally (or in the cloud), you create a personalized version of Expo Go. It includes all the custom native dependencies specified in your `package.json`. You get the same fast feedback loop (Metro hot reloading) but with zero restrictions on custom native code.

### Introduction to EAS (Expo Application Services)

EAS is a suite of cloud services designed for React Native:
- **EAS Build**: Compiles your app into binaries (`.ipa` for iOS, `.apk`/`.aab` for Android) in the cloud. This means you can build iOS apps even if you are developing on a Linux or Windows machine.
- **EAS Submit**: Automates sending those binaries directly to the Apple App Store and Google Play Store.
- **EAS Update**: Enables Over-The-Air (OTA) updates to fix bugs instantly by sending new Javascript bundles to users without requiring a full App Store review.

### What changed (Web -> Mobile / Classic RN -> Modern Expo)

| Concept | Web Development | Classic React Native (CLI) | Modern Managed Expo |
| :--- | :--- | :--- | :--- |
| **Native Projects** | N/A (runs in browser) | Checked in `/ios` and `/android`; managed manually | Generated dynamically via `npx expo prebuild`; ignored in Git |
| **Local Runner** | Webpack / Vite / Next Dev | `react-native run-ios` (requires local compile) | Expo Go (instant) or Development Clients (custom native modules) |
| **Routing** | React Router / Next.js file-based | React Navigation (imperative JS configuration) | Expo Router (file-based, built on React Navigation) |
| **App Config** | `package.json` / `next.config.js` | Modifying `Info.plist`, `AndroidManifest.xml` | `app.json` + Config Plugins (transpiled to native config) |
| **Cloud Builds** | Vercel / Netlify (static bundle) | Custom Fastlane / CI/CD scripts | EAS Build & EAS Submit |

## Drill

Initialize a new Expo application, configure it for file-based routing, and explore its anatomy.

### Task Description
1. Create a new Expo project using the official template CLI.
2. Structure the directory to use Expo Router with a basic root index screen.
3. Review and document the generated config files.

> [!example] Success criteria
> - [ ] A new project is successfully generated using `npx create-expo-app@latest`.
> - [ ] The project structure utilizes the `/app` directory for Expo Router.
> - [ ] The file `app.json` is inspected and its main keys (`name`, `slug`, `scheme`, `ios`, `android`, `plugins`) are documented with their purposes.
> - [ ] The `.expo` and `.gitignore` configurations are analyzed to verify that platform-specific build folders are excluded from source control.
> - [ ] Learner can explain why `/ios` and `/android` are not checked into Git when using prebuild.
