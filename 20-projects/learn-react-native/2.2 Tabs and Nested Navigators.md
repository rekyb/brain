# 2.2 Tabs and Nested Navigators

> [!abstract] TL;DR
> Bottom tab navigation is the standard dashboard pattern for mobile apps. Expo Router simplifies tab configuration and navigator nesting through folder structure and layout hierarchies. Nesting stacks inside tabs is achieved by putting a nested stack layout folder inside a tab group, requiring careful header visibility configuration to avoid "double-header" layouts.

## Digest

Mobile interfaces favor flat, quickly accessible navigation hierarchies. While desktop websites rely on side menus or top navbar links, mobile applications lean heavily on bottom tab navigators (`Tabs`) to toggle between core hubs (e.g. Home, Inbox, Settings). 

In Expo Router, layouts can be nested inside other layouts. This maps directly to nesting one navigator inside another (e.g. a Stack navigator inside a Tab screen).

### Implementing Bottom Tabs

The `<Tabs>` component from `expo-router` wraps the bottom tab bar implementation from React Navigation. It accepts screen options to control active/inactive colors, labels, background styling, and tab bar icon rendering.

Here is a basic layout folder structure configuring a tab interface:

```
app/
├── (tabs)/
│   ├── _layout.tsx        # Configures <Tabs>
│   ├── index.tsx          # Home Tab
│   ├── profile.tsx        # Profile Tab
│   └── settings.tsx       # Settings Tab
└── _layout.tsx            # Root Stack navigator (if any)
```

> [!note]
> The directory name `(tabs)` is surrounded by parentheses. In Expo Router, **parentheses define route groups**. Route groups allow you to organize files without adding extra segments to the URL path. Therefore, the path for `app/(tabs)/index.tsx` is `/`, and `app/(tabs)/settings.tsx` is `/settings`.

Here is an example `app/(tabs)/_layout.tsx`:

```tsx
import { Tabs } from 'expo-router';
import { Ionicons } from '@expo/vector-icons'; // Built-in Expo vector icons

export default function TabLayout() {
  return (
    <Tabs
      screenOptions={{
        tabBarActiveTintColor: '#0a84ff',   // Color for selected tab
        tabBarInactiveTintColor: '#8e8e93', // Color for unselected tab
        tabBarStyle: {
          backgroundColor: '#1c1c1e',      // Tab bar background
          borderTopColor: '#38383a',
        },
        headerStyle: {
          backgroundColor: '#1c1c1e',
        },
        headerTintColor: '#ffffff',
      }}
    >
      <Tabs.Screen
        name="index"
        options={{
          title: 'Home',
          tabBarIcon: ({ color, focused }) => (
            <Ionicons 
              name={focused ? 'home' : 'home-outline'} 
              size={24} 
              color={color} 
            />
          ),
        }}
      />
      <Tabs.Screen
        name="profile"
        options={{
          title: 'Profile',
          tabBarIcon: ({ color, focused }) => (
            <Ionicons 
              name={focused ? 'person' : 'person-outline'} 
              size={24} 
              color={color} 
            />
          ),
        }}
      />
    </Tabs>
  );
}
```

### Nesting Stacks inside Tabs

Often, you want a tab to represent a stack-based flow. For instance, the **Habits Tab** should first display a list of habits. Clicking a habit should open a detail screen with transitions, but the bottom tab bar should remain visible.

To nest a Stack navigator within a Tab:
1. Create a subdirectory under your tab group (e.g. `app/(tabs)/habits/`).
2. Add a `_layout.tsx` inside that subdirectory configured with `<Stack>`.
3. Put the screens for that sub-navigation inside that folder.

#### Folder Structure for Nested Navigation

```
app/
└── (tabs)/
    ├── _layout.tsx          # Configures <Tabs>
    ├── index.tsx            # Home Tab (/index)
    ├── settings.tsx         # Settings Tab (/settings)
    └── habits/              # Habits Tab Folder
        ├── _layout.tsx      # Configures nested <Stack> for Habits tab
        ├── index.tsx        # Habits Dashboard list (Path: /habits)
        └── [id].tsx         # Dynamic Habit Detail screen (Path: /habits/123)
```

In the tab bar configuration (`app/(tabs)/_layout.tsx`), you map the tab screen option target to the `habits` directory:

```tsx
// app/(tabs)/_layout.tsx
import { Tabs } from 'expo-router';

export default function TabLayout() {
  return (
    <Tabs>
      <Tabs.Screen name="index" options={{ title: 'Home' }} />
      
      {/* Target the nested folder name */}
      <Tabs.Screen 
        name="habits" 
        options={{ 
          title: 'Habits',
          headerShown: false, // CRITICAL: Avoid double headers!
        }} 
      />
      
      <Tabs.Screen name="settings" options={{ title: 'Settings' }} />
    </Tabs>
  );
}
```

Then, configure the nested stack layout inside `app/(tabs)/habits/_layout.tsx`:

```tsx
// app/(tabs)/habits/_layout.tsx
import { Stack } from 'expo-router';

export default function HabitsLayout() {
  return (
    <Stack>
      <Stack.Screen name="index" options={{ title: 'My Habits' }} />
      <Stack.Screen name="[id]" options={{ title: 'Habit Details' }} />
    </Stack>
  );
}
```

### The Double-Header Problem

A common mistake in mobile navigation is leaving headers enabled on both the parent navigator and the child navigator. This creates stacked headers at the top of the device screen:

```
+-----------------------------+
|        Parent Header        |  <- Tab bar header
+-----------------------------+
|        Child Header         |  <- Nested Stack header
+-----------------------------+
|                             |
|        Screen Content       |
```

To resolve this issue:
- **Hand over header responsibility to the child**: If a Tab screen contains a nested Stack navigator, set `headerShown: false` on the `<Tabs.Screen>` within the Tab layout. This hides the Tab-level header and allows the Stack-level header to render and animate titles and back buttons seamlessly.
- **Hand over header responsibility to the parent**: Alternatively, if the parent layout should contain a fixed header and nested screens shouldn't have their own navigation bars, set `headerShown: false` on the `<Stack>` or individual `<Stack.Screen>` options inside the child navigator.

## Drill

Design a tab-based navigation shell that hosts an independent details navigation stack.

### Task Description
1. Create a `(tabs)` directory with three distinct tab views: `Home`, `Habits`, and `Settings`.
2. Implement vector icon styling for each tab using `@expo/vector-icons` (e.g. Ionicons) showing active (filled) vs. inactive (outlined) states depending on the tab's active state.
3. Nest an independent Stack navigator inside the `Habits` tab directory structure.
4. On the Habits main screen, create a button or list row that triggers a push navigation to a nested details screen.
5. Configure header rendering options so that navigating into the details screen displays a native Stack header (with a working back button) without creating a double-header UI.

> [!example] Success criteria
> - [ ] Bottom tab bar displays 3 active tabs (Home, Habits, Settings) with distinct icons.
> - [ ] Choosing a tab item changes active visual focus (color/icon fill state changes).
> - [ ] The `habits` tab points to a nested subdirectory containing its own `<Stack>` layout.
> - [ ] The double header issue is resolved by disabling the tab-level header for the Habits stack.
> - [ ] Navigating to the habit detail screen displays a back button and keeps the bottom tab bar visible (or pushes onto it, depending on structural preference, but here tabs remain visible because the Stack is nested inside the tab).
