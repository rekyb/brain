# 1.1 Core Components and Layouts

> [!abstract] TL;DR
> React Native replaces browser DOM nodes with native primitives mapped directly to iOS and Android UI views. Layouts are constructed using a customized, column-first implementation of Flexbox, styled strictly via `StyleSheet.create` without global cascading or CSS stylesheets.

## Digest

Web React developers are accustomed to relying on the browser's DOM. In React Native, there is **no DOM and no HTML**. When you write `<View>`, React Native instantiates a `UIView` on iOS and a `android.view.ViewGroup` on Android.

### Web to Mobile Primitive Mapping

React Native provides direct primitives to construct interfaces. Here is the web-to-mobile mapping:

| Web equivalent | React Native Primitive | Purpose |
| :--- | :--- | :--- |
| `<div>` | `<View>` | The basic container. Supports layout (Flexbox) and styling, but does not display text. |
| `<span>`, `<p>`, `<h1>` | `<Text>` | All text must be wrapped in this component. Unwrapped raw text causes compilation crashes. |
| `<img>` | `<Image>` or `<ImageBackground>` | Renders local or remote images. Requires a `source` prop containing a file import or a URI object. |
| `div` with `overflow: scroll` | `<ScrollView>` | A scrollable container. Unlike web `overflow`, views do NOT automatically scroll when content overflows; you must explicitly wrap them. Renders all children at once (not optimized for long lists). |
| Virtualized Lists | `<FlatList>` | Renders large datasets efficiently by only instantiating visible items on the screen (windowing). |
| `<input type="text">` | `<TextInput>` | Accepts user text input. Relies on native keyboards. |

### Styling via StyleSheet

React Native uses a CSS-in-JS style system. Instead of style sheets, you define style objects and pass them to the `style` prop. For performance and validation, you should use `StyleSheet.create`:

```tsx
import { StyleSheet, View, Text } from 'react-native';

export default function MyCard() {
  return (
    <View style={styles.card}>
      <Text style={styles.title}>Hello Mobile World</Text>
    </View>
  );
}

const styles = StyleSheet.create({
  card: {
    backgroundColor: '#ffffff',
    padding: 16,
    borderRadius: 8,
    shadowColor: '#000',
    shadowOpacity: 0.1,
    shadowRadius: 4,
    elevation: 3, // Required for Android shadow rendering
  },
  title: {
    fontSize: 18,
    fontWeight: 'bold',
    color: '#333333',
  },
});
```

### Critical Layout Rules: Flexbox Differences

React Native's Flexbox engine (written in C++ as Yoga) mimics CSS Flexbox but has critical differences:

1. **Flex Direction Defaults to Column**: Web flexbox defaults to `row`. On mobile, screens are vertical by default, so `flexDirection` defaults to `column`.
2. **All Elements are Flex by Default**: You cannot change `display` to `block`, `inline`, or `grid`. The only valid values are `flex` and `none`.
3. **No Units**: Widths, heights, margins, and paddings are defined as raw numbers. They represent **density-independent points** (logical pixels), not physical pixels. Percentage strings (e.g. `'50%'`) are supported, but `em`, `rem`, `vh`, and `vw` are not.
4. **No Cascading**: Styles do not cascade. Setting `color: 'red'` on a outer `<View>` does not make the inner `<Text>` components red. The only exception is nested `<Text>` components: `<Text style={{ color: 'red' }}><Text>This text is red</Text></Text>`.
5. **No Flex Shrink by Default**: On the web, `flexShrink` defaults to `1`. In React Native, it defaults to `0`, meaning elements will not shrink to fit the screen unless explicitly given `flexShrink: 1` or `flex: 1`.

### Pixel Density and Scaling

Mobile devices have varying pixel densities. For instance, an iPhone "Retina" screen might pack 3 physical pixels into 1 logical point (a scale factor of `@3x`). 

React Native handles this dynamically:
- If you supply a width of `100`, it occupies `100` logical points (which maps to 300 physical pixels on an `@3x` device).
- For images, you can provide resolution-specific assets (e.g., `avatar.png`, `avatar@2x.png`, `avatar@3x.png`) in your project directories, and the packager will load the correct asset based on device density.

## Drill

Construct a profile card that renders consistently on iOS and Android.

### Task Description
1. Create a Profile Card component featuring:
   - An avatar image (circular).
   - User name and a short biography text.
   - A list of recent activities/posts.
2. The card should use a card-like style with clean borders, rounded corners, and a subtle shadow.
3. The list of recent activities must use a `<FlatList>` component to scroll vertically inside the profile screen.
4. Do not use external libraries (like Tailwind or styled-components). Rely entirely on `StyleSheet.create`.

> [!example] Success criteria
> - [ ] The profile card layout adapts properly to mobile viewport sizes.
> - [ ] Avatar image is cropped to a perfect circle using `borderRadius` (width and height must be equal, and `borderRadius` set to half that size).
> - [ ] Card layout uses `flexDirection: 'column'` (default) or `flexDirection: 'row'` explicitly to align sub-components.
> - [ ] The list of posts renders using `<FlatList>` with custom child components representing individual post cards.
> - [ ] Android shadow configuration uses `elevation`, while iOS uses `shadowColor`, `shadowOffset`, `shadowOpacity`, and `shadowRadius`.
> - [ ] Absolutely no HTML tags (`div`, `p`, etc.) are used in the codebase.
