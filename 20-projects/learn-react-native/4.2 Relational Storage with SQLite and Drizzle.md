# 4.2 Relational Storage with SQLite and Drizzle

> [!abstract] TL;DR
> Modern Expo applications utilize the rewritten, high-performance `expo-sqlite` driver combined with Drizzle ORM. This stack provides complete TypeScript type-safety for database operations, direct JSI bindings, declarative schema definitions, and reactive UI updates via live queries, eliminating legacy manual SQL string concatenation.

## Digest

For applications handling complex, structured, or relational data (such as habits with completion logs, or localized task management), flat key-value persistence is insufficient. Developers need a local relational database. 

React Native projects have transitioned away from legacy native binding libraries (like `react-native-sqlite-storage`) in favor of the modernized `expo-sqlite` library integrated with **Drizzle ORM**.

---

### The Evolution: Legacy vs. Modern SQLite in React Native

Historically, using SQLite in React Native was cumbersome and error-prone:
1. **Bridge Overhead**: Query parameters and large rows were serialized to JSON and pushed across the asynchronous React Native bridge.
2. **Untyped SQL Strings**: Developers wrote raw SQL strings directly in JS. A typo in a column name would only fail at runtime.
3. **Manual Migrations**: Changing database schemas required manual version tracking tables and raw SQL `ALTER TABLE` execution logic.

#### Modern Expo SQLite + Drizzle Stack
Modern `expo-sqlite` interacts directly with SQLite engine C++ APIs via JSI. When coupled with **Drizzle ORM**, the developer workflow changes dramatically:
- **Type-Safety**: Drizzle extracts compile-time types directly from your schema definitions.
- **SQL-Like Syntax**: Unlike heavy ActiveRecord ORMs (e.g., TypeORM), Drizzle matches SQL structure closely, keeping database queries transparent and highly performant.
- **Automated Migrations**: Drizzle Kit compiles schema changes into SQL migration scripts, which execute automatically when the app boots.

---

### Defining Schemas with Drizzle

In Drizzle, the TypeScript schema definition is the single source of truth. Here is how a schema is declared using SQLite column types:

```typescript
import { sqliteTable, text, integer } from 'drizzle-orm/sqlite-core';

export const habits = sqliteTable('habits', {
  id: text('id').primaryKey(),
  title: text('title').notNull(),
  frequency: text('frequency').default('daily').notNull(),
  createdAt: integer('created_at', { mode: 'timestamp' }).notNull(),
  isActive: integer('is_active', { mode: 'boolean' }).default(true).notNull(),
});

export const completions = sqliteTable('completions', {
  id: text('id').primaryKey(),
  habitId: text('habit_id')
    .notNull()
    .references(() => habits.id, { onDelete: 'cascade' }),
  completedAt: integer('completed_at', { mode: 'timestamp' }).notNull(),
});
```

### Database Initialization and Setup

To connect Drizzle with Expo SQLite, open or create the database using `expo-sqlite` and wrap it with Drizzle's database driver.

```typescript
import { openDatabaseSync } from 'expo-sqlite';
import { drizzle } from 'drizzle-orm/expo-sqlite';

// Open the physical SQLite database file
const expoDb = openDatabaseSync('app.db', { enableChangeListener: true });

// Instantiate the Drizzle instance
export const db = drizzle(expoDb);
```

#### Running Startup Migrations
Drizzle provides a migration helper to run migrations stored within your app assets. `drizzle-kit` compiles schema definitions into SQL migration scripts inside a migration folder (e.g., `./drizzle`). 

During development, these assets are compiled. At app startup, we run the migrations:

```typescript
import { useMigrations } from 'drizzle-orm/expo-sqlite/migrator';
import migrations from './drizzle/migrations'; // Generated by drizzle-kit
import { Text, View } from 'react-native';

export default function RootLayout() {
  const { success, error } = useMigrations(db, migrations);

  if (error) {
    return (
      <View>
        <Text>Migration failed: {error.message}</Text>
      </View>
    );
  }

  if (!success) {
    return (
      <View>
        <Text>Migrating database schemas...</Text>
      </View>
    );
  }

  return <AppContent />;
}
```

### Type-Safe CRUD Operations

Drizzle query builder uses TypeScript generics to enforce that inputs and outputs conform to your schema.

#### Inserting Data
```typescript
import { habits } from './schema';

async function createHabit(title: string) {
  await db.insert(habits).values({
    id: Math.random().toString(36).substring(7),
    title: title,
    createdAt: new Date(),
  });
}
```

#### Querying Data
```typescript
import { eq } from 'drizzle-orm';
import { habits } from './schema';

async function getActiveHabits() {
  return await db
    .select()
    .from(habits)
    .where(eq(habits.isActive, true));
}
```

### Reactive UI: Live Queries vs. TanStack Query

When the database is modified, the UI needs to reflect those changes instantly. There are two primary patterns:

#### 1. Drizzle's Live Query Hook
Drizzle provides a React hook `useLiveQuery` that subscribes to database modifications and re-renders components whenever affected tables change.

```typescript
import { useLiveQuery } from 'drizzle-orm/expo-sqlite';
import { db } from './db';
import { habits } from './schema';
import { FlatList, Text } from 'react-native';

export function HabitListScreen() {
  // Automatically updates when data is inserted/deleted/updated in the 'habits' table
  const { data } = useLiveQuery(db.select().from(habits));

  return (
    <FlatList
      data={data}
      keyExtractor={(item) => item.id}
      renderItem={({ item }) => <Text>{item.title}</Text>}
    />
  );
}
```

#### 2. TanStack Query Cache Invalidation
Alternatively, if your application relies on TanStack Query, perform invalidations on database mutations:

```typescript
import { useMutation, useQueryClient } from '@tanstack/react-query';

export function useCreateHabitMutation() {
  const queryClient = useQueryClient();

  return useMutation({
    mutationFn: async (title: string) => {
      return await db.insert(habits).values({
        id: Math.random().toString(36).substring(7),
        title,
        createdAt: new Date(),
      });
    },
    onSuccess: () => {
      // Force UI hooks reading habits to refetch from the local DB
      queryClient.invalidateQueries({ queryKey: ['habits'] });
    },
  });
}
```

## Drill

Create a habits table schema using Drizzle. Write queries to insert a habit and fetch the list of habits. Set up startup migrations.

### Task Description

1. Define a schema for a `habits` table containing `id` (text primary key), `title` (text, non-null), `status` (text/integer status indicator), and `updatedAt` (timestamp/integer indicating last change).
2. Configure a connection to an `expo-sqlite` database instance using the Drizzle ORM driver.
3. Write a type-safe function `insertHabit(title: string)` that inserts a new habit row into the SQLite database.
4. Write a type-safe query function `fetchHabits()` that retrieves all habits from the database.
5. Create a module or initialization script that loads and executes database migrations on application startup before the main interface renders.

> [!example] Success criteria
> - [ ] Schema is defined with correct relational columns and appropriate types.
> - [ ] Insert/update and fetch database operations are verified as type-safe compiled statements using Drizzle query syntax.
> - [ ] Startup migration flow is fully designed, ensuring schema consistency before user interactions can occur.
