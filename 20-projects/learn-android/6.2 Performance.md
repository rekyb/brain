# 6.2 Performance

> [!abstract] TL;DR
> Compose performance problems are almost always over-recomposition — caused by unstable types, un-remembered lambdas, or missing list keys — and the tool chain now gives you exact recomposition counts in the Layout Inspector to find them.

## Digest

### What triggers a recomposition

Compose skips re-running a composable when all its parameters are equal to the last call — but only when those parameters are **stable**. A stable type is one the compiler can trust: when the reference is the same (for object types), the value is also the same. For primitive types and most `data class`es with all-stable fields, the compiler infers stability automatically. For types it cannot prove stable, it conservatively marks them unstable and always re-runs the composable.

The practical consequence: a composable receiving an unstable parameter will recompose every time any ancestor recomposes, even if the actual data is unchanged.

### Finding over-recomposition with Layout Inspector

Android Studio's Layout Inspector (view the running app via the **Layout Inspector** tool window) has a **Recomposition** mode that shows a live count and highlight for each composable. A count that keeps climbing on a composable that should be inert after initial load is the clearest signal you can get. The Compose compiler can also emit stability metadata. Since Kotlin 2.0, the Compose compiler ships with Kotlin and is configured through the `composeCompiler {}` Gradle DSL (part of the Compose Compiler Gradle plugin). The old `freeCompilerArgs += ["-P", "plugin:androidx.compose.compiler.plugins.kotlin:..."]` approach no longer works. Add the following block to your module-level `build.gradle.kts` to generate per-composable stability reports:

```kotlin
composeCompiler {
    reportsDestination = layout.buildDirectory.dir("compose_compiler")
    metricsDestination = layout.buildDirectory.dir("compose_compiler")
}
```

### Stability: `@Stable`, `@Immutable`, and collections

The most common cause of instability is a plain `List<T>`, `Map<K, V>`, or `Set<T>` field on an otherwise-stable class. The Kotlin standard-library collection interfaces are mutable at runtime, so the Compose compiler does not mark them stable. A `data class` with a `val habits: List<Habit>` field is therefore **unstable** by default, and any composable accepting it recomposes unnecessarily.

Fix options:

1. **`kotlinx.collections.immutable`** — `ImmutableList<T>`, `PersistentList<T>`, etc. These are annotated `@Immutable` and the compiler infers stability correctly. Switch your `UiState` to use them and convert at the use-site with `.toImmutableList()`.
2. **`@Immutable` on your own class** — you guarantee the class will not change after construction. Correct for a snapshot data class; do not use if the object can mutate.
3. **`@Stable` on your own class** — you guarantee consistent equality: `a == b` implies `a.x == b.x` for all relevant properties. Correct for classes that may hold mutable state but report changes through Compose's snapshot system.

```kotlin
@Immutable
data class HabitListUiState(
    val habits: ImmutableList<Habit>,
    val isLoading: Boolean,
    val errorMessage: String?,
)
```

Lambdas inline-defined inside a composable are also recreated on every recomposition. In hot-path composables, wrap them in `remember`:

```kotlin
val onDelete = remember(habitId) { { viewModel.delete(habitId) } }
```

### `derivedStateOf`

When a value is derived from one or more state reads, wrapping it in `derivedStateOf` prevents the composable from recomposing every time the source state changes — it only recomposes when the **derived result** actually changes:

```kotlin
val showEmptyState by remember {
    derivedStateOf { habits.isEmpty() }
}
```

Without `derivedStateOf`, every `habits` update recomposes anything reading `showEmptyState`, even when the list was non-empty before and stays non-empty after. Use `derivedStateOf` when the derivation filters or reduces a high-frequency state to a low-cardinality value.

### Lazy-list keys

As covered in [[2.4 Lazy lists]], always set `key = { it.id }` on `items { }` blocks. Without a key, inserting a habit at the top of the list causes every visible item to recompose because Compose falls back to position-based identity. With a stable `Long` key, Compose recomposes only the newly inserted item and animates the rest.

### Baseline Profiles

Compose apps have a higher cold-start overhead than equivalent View-based apps because the Compose runtime and your composables must be JIT-compiled on first run. **Baseline Profiles** address this: they are hint files (`.prof`) that tell the Android runtime to AOT-compile a specified set of hot methods during app installation, buying back most of that startup time.

The `androidx.benchmark:benchmark-macro-junit4` library generates Baseline Profiles by running a Macrobenchmark test on a real device. The `baselineProfile` Gradle plugin (AGP 8+) wires generation into the build and inserts the `.prof` file into the release APK/AAB automatically. For a Compose app, typical results are a 20–40 % faster cold start and noticeably smoother first-frame rendering.

### Strong Skipping

Compose compiler 1.5.4+ enables **Strong Skipping** by default. Under Strong Skipping, composables with unstable parameters are still skippable when their lambdas are the same instance — the compiler automatically `remember`s lambdas that are created inside composables. This reduces (but does not eliminate) the need to manually remember lambdas. Stability annotations and `ImmutableList` remain best practice for data types passed across recomposition boundaries.

### What changed since 2021

- **The Layout Inspector now shows recomposition counts.** In 2021 there was no first-class IDE tool for seeing which composables were recomposing; you relied on `Log` statements or profiler traces.
- **Stability and `@Immutable`/`@Stable`** are concepts that have no analogue in the View world. View-based UI had no equivalent "skip draw if input is the same" optimisation path for arbitrary types.
- **`derivedStateOf`** was introduced with Compose. The closest View-world concept was `MediatorLiveData`, but it operated on LiveData not on the rendering pass itself.
- **Baseline Profiles** apply to both Compose and non-Compose apps, but the Compose cold-start penalty made them particularly impactful. The `baselineProfile` Gradle plugin (AGP 8+) made them easy to integrate — in 2021 the tooling was experimental.
- **Strong Skipping** (2023+) reduces the manual lambda-remembering burden that early Compose required.
- **Compose compiler metrics** are configured via the `composeCompiler {}` Gradle DSL (introduced with the Compose Compiler Gradle plugin in Kotlin 2.0). Setting `reportsDestination` and `metricsDestination` in that block generates per-composable stability output without guessing.

## Drill

Work in the capstone habit-tracker project.

**Task — find and fix an over-recomposing composable**

Introduce an over-recomposing composable deliberately: either pass the habit list as a plain `List<Habit>` through a non-`@Stable` wrapper class, or define a callback lambda inline (not remembered) on a composable that receives it. Trigger a repeating state update — a button that writes to a `MutableStateFlow` field — and open the Layout Inspector in recomposition highlight mode. Observe which composables light up and how many times.

Fix the over-recomposition using one or more techniques from the Digest: `@Immutable` on the wrapper class, `ImmutableList` for the list field, or `remember { }` for the lambda. Return to the Layout Inspector and confirm the affected composable's recomposition count no longer climbs on the same trigger.

> [!example] Success criteria
> - [ ] Before the fix, the Layout Inspector shows the composable recomposing on each trigger even though its meaningful data has not changed.
> - [ ] After the fix, the recomposition count for the affected composable drops to zero or near-zero during the same trigger action.
> - [ ] The fix uses at least one of: `@Immutable`, `@Stable`, `ImmutableList`, `remember`-ed lambda, or `derivedStateOf`.
> - [ ] You can explain in one sentence *why* the fix works (what property of the type or lambda changed to make Compose able to skip it).
