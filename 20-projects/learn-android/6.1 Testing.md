# 6.1 Testing

> [!abstract] TL;DR
> Testing a modern Android app means coroutine-aware test runners (`runTest`), Flow assertions via Turbine, fake repositories instead of Mockito mocks, and Compose UI tests that talk to the semantic tree — not layout IDs.

## Digest

### Unit testing coroutines and ViewModels

In 2021, testing async code usually meant `runBlocking { }` or the now-deprecated `runBlockingTest`. The current replacement is `runTest` from `kotlinx-coroutines-test`, which controls virtual time — `delay()` calls advance instantly unless you configure otherwise. Pair it with `StandardTestDispatcher` (manual time control via `advanceUntilIdle()` or `runCurrent()`) or `UnconfinedTestDispatcher` (eager, runs coroutines synchronously until suspension) depending on how much scheduling control you need.

For a ViewModel that dispatches to `Dispatchers.IO` or `Dispatchers.Main`, swap in a test dispatcher before the ViewModel is constructed. The cleanest way is a JUnit4 `TestWatcher` rule:

```kotlin
class MainDispatcherRule(
    val dispatcher: TestDispatcher = UnconfinedTestDispatcher()
) : TestWatcher() {
    override fun starting(description: Description) = Dispatchers.setMain(dispatcher)
    override fun finished(description: Description) = Dispatchers.resetMain()
}

class HabitListViewModelTest {
    @get:Rule val mainDispatcherRule = MainDispatcherRule()

    @Test
    fun `initial load populates habits`() = runTest { ... }
}
```

The rule installs the test dispatcher as `Main` before every test and restores it after, so `viewModelScope.launch` runs on a controlled scheduler.

### Flow testing with Turbine

A `StateFlow` emitting a sequence of states is awkward to collect manually — you need a background job, a list accumulator, and cancellation. **Turbine** wraps all that into a readable API:

```kotlin
@Test
fun `inserting a habit emits updated state`() = runTest {
    val viewModel = HabitListViewModel(FakeHabitRepository())
    viewModel.uiState.test {
        val empty = awaitItem()          // initial emission
        assertTrue(empty.habits.isEmpty())

        viewModel.onEvent(HabitListEvent.AddHabit("Morning run"))
        val updated = awaitItem()        // next emission after insert
        assertEquals("Morning run", updated.habits.first().name)

        cancelAndIgnoreRemainingEvents()
    }
}
```

`awaitItem()` suspends until an emission arrives. `cancelAndIgnoreRemainingEvents()` closes the subscription without failing on un-asserted downstream emissions. `expectNoEvents()` is useful when you want to assert nothing was emitted.

### Fakes over mocks

Mockito still works, but for repository interfaces a **fake** — a real implementation backed by in-memory state — is simpler and more honest. It tests the actual contract instead of mocking individual method calls, it plays well with `StateFlow`, and it gives you a controllable data source with no `verify()` gymnastics:

```kotlin
class FakeHabitRepository : HabitRepository {
    private val habits = MutableStateFlow<List<Habit>>(emptyList())
    override fun getHabits(): Flow<List<Habit>> = habits.asStateFlow()
    override suspend fun addHabit(name: String) {
        habits.update { it + Habit(id = System.currentTimeMillis(), name = name) }
    }
    override suspend fun deleteHabit(id: Long) {
        habits.update { list -> list.filterNot { it.id == id } }
    }
    override suspend fun toggleCompleted(id: Long) {
        habits.update { list -> list.map { if (it.id == id) it.copy(isCompleted = !it.isCompleted) else it } }
    }
    override suspend fun setReminder(id: Long, time: LocalTime?) {
        habits.update { list -> list.map { if (it.id == id) it.copy(reminderTime = time) else it } }
    }
}
```

Construct the ViewModel with `FakeHabitRepository()` in every test method that needs it. Each test starts from a clean slate without any `@Before` `reset()` calls.

### Compose UI tests

Compose UI tests drive the **semantic tree**, not view IDs or XML hierarchies. The entry point is a JUnit4 rule:

```kotlin
@get:Rule val composeTestRule = createComposeRule()
// or, for tests that need a real Activity context:
@get:Rule val composeTestRule = createAndroidComposeRule<MainActivity>()
```

Set content, find nodes by semantic properties, act, then assert:

```kotlin
@Test
fun `habit list screen displays habits`() {
    composeTestRule.setContent {
        HabitListScreen(viewModel = fakeViewModel)
    }
    composeTestRule.onNodeWithText("Morning run").assertIsDisplayed()
    composeTestRule.onNodeWithContentDescription("Add habit").performClick()
    composeTestRule.onNodeWithText("New habit").assertExists()
}
```

Finders — `onNodeWithText`, `onNodeWithContentDescription`, `onNodeWithTag` — return a `SemanticsNodeInteraction` chainable with `.performClick()`, `.performTextInput()`, `.assertIsDisplayed()`, `.assertIsEnabled()`, and more. When multiple matching nodes exist, use `onAllNodesWithText(...)[0]` to select by index, or add a `testTag` to the composable and use `onNodeWithTag`.

### What changed since 2021

- **`runBlockingTest` is removed.** Use `runTest` from `kotlinx-coroutines-test 1.7+`. Virtual time advances automatically for `delay`; use `advanceUntilIdle()` or `runCurrent()` when you need to control execution steps explicitly.
- **`StandardTestDispatcher` vs `UnconfinedTestDispatcher`.** `StandardTestDispatcher` (the `runTest` default) queues coroutines and requires manual advancement — useful when you need to assert intermediate states between steps. `UnconfinedTestDispatcher` runs coroutines eagerly — useful when you just want the final state.
- **Turbine is the standard for Flow assertions.** There was no clean idiom for asserting sequential emissions in 2021; today Turbine is the first answer.
- **Espresso is no longer the path for new UI tests in Compose projects.** `createComposeRule` and the semantic-tree API replace layout-ID-based Espresso interactions. Espresso interop exists for hybrid screens that still have Views, but it is a fallback.
- **Mockito is optional.** For repository interfaces, a fake implementation tests more of the real contract, runs faster, and is easier to read.

## Drill

Work in the capstone habit-tracker project.

**Part A — ViewModel unit test with a fake**

Write a unit test class for the habit-list ViewModel. Implement `FakeHabitRepository` (or equivalent) backed by a `MutableStateFlow`. Apply the `MainDispatcherRule` pattern. Write at least two tests using `runTest` and Turbine's `.test { awaitItem() }`:
1. Assert the initial `uiState` emission reflects an empty or loading state.
2. Assert that calling `onEvent(HabitListEvent.AddHabit(...))` (or equivalent) causes the next emission to include the new habit.

**Part B — Compose UI test**

Write one Compose UI test using `createComposeRule`. Set the screen's content using a pre-populated fake ViewModel (or inject the fake repository via Hilt test). Assert that at least one habit name is visible on screen using a semantic finder. Optionally add a second assertion: a user action (click) causes the screen to update.

> [!example] Success criteria
> - [ ] Both test classes compile and pass (Part A with `./gradlew test`, Part B with `./gradlew connectedAndroidTest` or IDE run).
> - [ ] The ViewModel test uses `FakeHabitRepository` (or equivalent) — no `Mockito.mock(...)` or `@Mock` anywhere in the test class.
> - [ ] The ViewModel test uses `runTest` and Turbine's `.test { awaitItem() }` — no `runBlocking` or manual `toList()`.
> - [ ] `Dispatchers.setMain(...)` is applied before the ViewModel is constructed (via rule or `@Before`) so background dispatchers are controlled.
> - [ ] The Compose UI test uses `createComposeRule`, a semantic finder (`onNodeWithText` / `onNodeWithContentDescription` / `onNodeWithTag`), and at least one assertion — no Espresso view matchers.

## Capstone step

Add a **test suite** to the habit tracker covering the core ViewModel and one screen.

The suite must include: (1) a unit-test class for the habit-list ViewModel backed by a `FakeHabitRepository`, using `runTest` and Turbine to assert emitted states for at least two scenarios (e.g., initial load, habit insertion); (2) a Compose UI test class for the habit-list screen asserting that the screen renders a list of habits and responds to at least one user action.

> [!example] Capstone acceptance criteria
> - [ ] Both test classes are runnable and green (`./gradlew test` for unit tests; `./gradlew connectedAndroidTest` for the UI test).
> - [ ] The unit-test class has zero real-database or real-network references — only `FakeHabitRepository`.
> - [ ] The Compose UI test manipulates the semantic tree — no view IDs, no XML matchers, no Espresso.
