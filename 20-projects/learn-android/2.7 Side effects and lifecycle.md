# 2.7 Side effects and lifecycle

> [!abstract] TL;DR
> Composable functions must be side-effect-free by default; Compose provides explicit effect APIs ŌĆö `LaunchedEffect`, `rememberCoroutineScope`, `DisposableEffect` ŌĆö for work that needs to outlive or react to composition, and `collectAsStateWithLifecycle` for lifecycle-aware flow collection.

## Digest

### Why side effects need special handling

A composable function can be re-executed at any time ŌĆö Compose may recompose it multiple times before it appears, skip it, or cancel it. Launching a coroutine or registering a listener directly in the function body would create leaks and duplicate side effects. Compose's effect APIs are the sanctioned escape hatches.

### LaunchedEffect

`LaunchedEffect(key)` launches a coroutine *scoped to the composition*. The coroutine starts when the composable enters composition and is **cancelled and restarted** whenever any key changes. It is cancelled for good when the composable leaves composition.

```kotlin
@Composable
fun SearchScreen(query: String) {
    var results by remember { mutableStateOf<List<Result>>(emptyList()) }

    LaunchedEffect(query) {              // restarts every time query changes
        results = searchApi(query)       // suspend function ŌĆö safe here
    }

    ResultsList(results)
}
```

Use `LaunchedEffect(Unit)` for a one-shot effect that runs once on first composition and never restarts. Use a meaningful key (the trigger value) when you want the effect to rerun in response to a specific change.

### rememberCoroutineScope

`rememberCoroutineScope` gives you a `CoroutineScope` tied to the composition but triggered from event handlers (which are not themselves composable ŌĆö you can't call `LaunchedEffect` inside a `Button` click lambda). It's the right tool when the user triggers the coroutine, not composition:

```kotlin
@Composable
fun SaveButton(onSave: suspend () -> Unit) {
    val scope = rememberCoroutineScope()
    Button(onClick = { scope.launch { onSave() } }) {
        Text("Save")
    }
}
```

The scope is cancelled when the composable leaves composition ŌĆö any in-flight coroutines are automatically cleaned up.

### DisposableEffect

`DisposableEffect(key)` is for effects that require explicit cleanup ŌĆö registering listeners, subscribing to platform APIs, attaching lifecycle observers. The `onDispose` block runs when the composable leaves composition or the key changes:

```kotlin
@Composable
fun LocationTracker(onLocationUpdate: (Location) -> Unit) {
    val context = LocalContext.current

    DisposableEffect(Unit) {
        val listener = LocationListener { onLocationUpdate(it) }
        locationManager.requestUpdates(listener)

        onDispose {
            locationManager.removeUpdates(listener)   // guaranteed cleanup
        }
    }
}
```

### collectAsStateWithLifecycle

`collectAsStateWithLifecycle()` (from `lifecycle-runtime-compose`) collects a `Flow` or `StateFlow` and exposes its latest value as Compose `State<T>`, **pausing collection when the app is in the background** (below `Lifecycle.State.STARTED`). This is the correct way to consume a ViewModel's `StateFlow` in a composable:

```kotlin
@Composable
fun HabitListScreen(viewModel: HabitViewModel = hiltViewModel()) {
    val habits by viewModel.habits.collectAsStateWithLifecycle()
    LazyColumn { items(habits, key = { it.id }) { HabitItem(it) } }
}
```

> *`hiltViewModel()` needs Hilt (Phase 3.3); until then use `viewModel()` ŌĆö the call site is the only thing that changes.*

The older `collectAsState()` collects continuously ŌĆö even when the screen is in the background ŌĆö wasting resources. Prefer `collectAsStateWithLifecycle` for any flow that feeds UI.

### Effect summary

| API | Trigger | Cleanup | Use for |
|---|---|---|---|
| `LaunchedEffect(key)` | Composition / key change | Auto (coroutine cancel) | Suspend work driven by composition |
| `rememberCoroutineScope` | Event handler | Auto (scope cancel) | Suspend work driven by user action |
| `DisposableEffect(key)` | Composition / key change | `onDispose { }` | Listener registration, platform subscriptions |
| `SideEffect` | Every successful recomposition | None | Sync Compose state to non-Compose code |
| `collectAsStateWithLifecycle` | Flow emission | Auto | Lifecycle-aware flow ŌåÆ State |

### What changed since 2021

- **`LaunchedEffect` / `DisposableEffect` are new.** There was no equivalent in Fragment/View world; coroutines in UI were launched in `viewLifecycleOwner.lifecycleScope` and cleaned up via `onDestroyView`.
- **`collectAsStateWithLifecycle` replaces `LiveData.observe`** at the UI layer. It provides the same lifecycle awareness but works with `StateFlow` and cold `Flow`.
- **No `lifecycleOwner.lifecycleScope` in composables.** Use `LaunchedEffect` instead; it integrates with the Compose lifecycle, not the Activity/Fragment lifecycle.

## Drill

Work in the Compose project with a `ViewModel` that exposes a `StateFlow<List<String>>` (can be a simple in-memory list that updates on a timer or button press).

**Part A ŌĆö collectAsStateWithLifecycle**

Collect the ViewModel's `StateFlow` in a composable using `collectAsStateWithLifecycle`. Display the current list in a `LazyColumn`. Confirm the UI updates when the list changes. Do not use `collectAsState()`.

**Part B ŌĆö LaunchedEffect with a key**

Add a search field (`TextField`) to the screen. Use `LaunchedEffect(searchQuery)` to delay 300 ms (`delay(300)`) after each keystroke before filtering the list ŌĆö a basic debounce. Each time the query changes, the previous `LaunchedEffect` coroutine should be cancelled and a new one started. Log a message at the start of each `LaunchedEffect` execution so you can observe the restart behaviour in Logcat.

> [!example] Success criteria
> - [ ] Part A: the composable uses `collectAsStateWithLifecycle`, not `collectAsState()` or `LiveData.observe`.
> - [ ] Part A: the `LazyColumn` visibly updates when the backing `StateFlow` emits a new list.
> - [ ] Part B: rapidly typing several characters produces only one Logcat message per "settled" query (previous effects are cancelled mid-delay).
> - [ ] Part B: changing the search query after the 300 ms window shows filtered results; clearing it shows all results.
> - [ ] No coroutines launched directly in composable function bodies (outside effect APIs).

## Capstone step

**Habit tracker UI shell**

Build the navigable UI shell for the habit-tracker app. There is no persistence in this step ŌĆö all data is static in-memory sample data, hoisted using Compose state. `ViewModel` and repository wiring come in the Phase 3 capstone (3.4).

**Feature:** A two-screen flow ŌĆö a habit list and an add-habit form ŌĆö wired with type-safe `NavHost` routes. The list is seeded with a handful of sample `Habit` values and grows when the user saves a new one. All state lives in a hoisted state holder (e.g. `remember { mutableStateListOf<Habit>() }` lifted to a level both screens can reach), never inside individual composable bodies.

> [!example] Capstone acceptance criteria
> - [ ] Habit list screen renders the sample habits in a `LazyColumn`; each row shows at minimum a name and streak count (static sample data at this phase ŌĆö real check-off logic and persistence arrive in [[4.4 Offline-first and caching]]).
> - [ ] Tapping the FAB navigates to the add-habit screen using a type-safe route.
> - [ ] Submitting the add-habit form appends a new `Habit` to the in-memory list and navigates back; the new habit is visible immediately without a restart.
> - [ ] System back from the add-habit screen returns to the list without adding a duplicate.
> - [ ] No `NavController` reference is passed into `HabitListScreen` or `AddHabitScreen` ŌĆö navigation is driven by lambdas.
> - [ ] No XML layouts, no `RecyclerView`, no `Fragment`, no `LiveData` anywhere in the project.
> - [ ] *(Note: replacing this hoisted state with a real `ViewModel` + `StateFlow` + `collectAsStateWithLifecycle` is the Phase 3 capstone task.)*
