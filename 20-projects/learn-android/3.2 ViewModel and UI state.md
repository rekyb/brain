# 3.2 ViewModel and UI state

> [!abstract] TL;DR
> The modern ViewModel exposes a single `StateFlow<UiState>` — one immutable data class capturing everything the screen needs — and replaces the multiple `LiveData` fields you used in 2021.

## Digest

### One UiState to rule them

In 2021 you likely had a ViewModel with several independent `LiveData` fields — `val habits: LiveData<List<Habit>>`, `val isLoading: LiveData<Boolean>`, `val error: LiveData<String?>` — that the UI observed separately. The modern pattern consolidates all of that into **one immutable state class**:

```kotlin
data class HabitListUiState(
    val habits: List<Habit> = emptyList(),
    val isLoading: Boolean = false,
    val errorMessage: String? = null,
)
```

The screen renders entirely from this single object. When anything changes — habits load, an error appears, a spinner starts — the ViewModel produces a new copy of the state via `.copy(...)`. The UI re-renders from that copy. There is no question of which `LiveData` to observe or in what order observers fire; one state object is the only source of truth for the screen.

### Exposing StateFlow\<UiState\>

The canonical pattern uses a private `MutableStateFlow` and an exposed read-only `StateFlow`:

```kotlin
@HiltViewModel
class HabitListViewModel @Inject constructor(
    private val repository: HabitRepository
) : ViewModel() {

    private val _uiState = MutableStateFlow(HabitListUiState())
    val uiState: StateFlow<HabitListUiState> = _uiState.asStateFlow()
}
```

The `private` / `public` split enforces the rule: only the ViewModel mutates state; the composable can only read it.

State updates always go through `.update { }`:

```kotlin
_uiState.update { current -> current.copy(isLoading = true) }
```

`.update { }` is a compare-and-set operation — safe under concurrent coroutine updates. Avoid direct `.value =` assignment on shared mutable state.

### Handling events

Composables send user intent to the ViewModel as events — not by calling methods that directly mutate state. A clean pattern is a sealed interface per screen, funnelled through a single `onEvent()`:

```kotlin
sealed interface HabitListEvent {
    data class AddHabit(val name: String) : HabitListEvent
    data class DeleteHabit(val id: Long) : HabitListEvent
    object Refresh : HabitListEvent
}

// in ViewModel
fun onEvent(event: HabitListEvent) {
    when (event) {
        is HabitListEvent.AddHabit    -> addHabit(event.name)
        is HabitListEvent.DeleteHabit -> deleteHabit(event.id)
        HabitListEvent.Refresh        -> loadHabits()
    }
}
```

The composable holds no logic — it passes events up and renders state down.

### viewModelScope

Long-running work (loading from the repository, write operations) runs in `viewModelScope` — a `CoroutineScope` tied to the ViewModel's lifetime. When the ViewModel is cleared, `viewModelScope` cancels all its children automatically:

```kotlin
private fun loadHabits() {
    viewModelScope.launch {
        _uiState.update { it.copy(isLoading = true) }
        repository.getHabits().collect { habits ->
            _uiState.update { it.copy(habits = habits, isLoading = false) }
        }
    }
}
```

You do not need to manage coroutine lifecycle manually or cancel jobs in `onCleared()` for work launched in `viewModelScope`.

### Collecting in the composable

In the composable, collect `uiState` with `collectAsStateWithLifecycle()` (covered in [[2.7 Side effects and lifecycle]]):

```kotlin
@Composable
fun HabitListScreen(viewModel: HabitListViewModel = hiltViewModel()) {
    val state by viewModel.uiState.collectAsStateWithLifecycle()
    // render from `state`
}
```

`hiltViewModel()` (from `androidx.hilt:hilt-navigation-compose`) retrieves the Hilt-injected ViewModel without any factory boilerplate.

### What changed since 2021

- **`LiveData` is no longer recommended for new code.** Architecture samples and official guides moved to `StateFlow` + `collectAsStateWithLifecycle()`. `LiveData` still compiles; it is just not where investment is happening.
- **One `UiState` data class replaces multiple `LiveData` fields.** No more registering separate observers for loading, data, and errors — one state object covers all of it, and its snapshot is always consistent.
- **`@HiltViewModel` + `hiltViewModel()` replaces `ViewModelProvider` factory boilerplate.** Constructor arguments are injected automatically.
- **Sealed interface** (Kotlin 1.5+) is the idiomatic event type — preferred for event/state hierarchies because implementations may live anywhere in the same package (Kotlin 1.5+; previously same-file only), and unlike a sealed class an implementation can still extend another class, preserving composability while keeping the hierarchy closed for exhaustive `when`.
- **`viewModelScope`** (lifecycle-viewmodel-ktx 2.1, 2019) is now the universal standard. There is no alternative — do not create scopes manually in a ViewModel.

## Drill

Work in the capstone project from Phase 2.

**Part A — define UiState**

Create a `HabitListUiState` data class with at least three fields: a `habits` list, an `isLoading` flag, and a nullable `errorMessage`. Give every field a default value so that `HabitListUiState()` (no arguments) represents a valid "idle, empty" state.

**Part B — expose StateFlow**

Write `HabitListViewModel`. Expose `uiState: StateFlow<HabitListUiState>` backed by a private `MutableStateFlow`. In `init { }`, add a placeholder load: launch a coroutine in `viewModelScope` that sets `isLoading = true`, waits 500 ms (simulating async work with `delay`), then sets `isLoading = false` and populates `habits` with two hardcoded `Habit` objects.

**Part C — add an event handler**

Define a `HabitListEvent` sealed interface with at least an `AddHabit(val name: String)` variant. Add `onEvent(event: HabitListEvent)` to the ViewModel. The `AddHabit` handler must update state immutably with `.update { it.copy(...) }`, appending the new habit to the existing list. Wire the Add action in `HabitListScreen` to call `viewModel.onEvent(HabitListEvent.AddHabit(...))`.

> [!example] Success criteria
> - [ ] `HabitListUiState` is a `data class` with all `val` fields and no mutable collections.
> - [ ] `uiState` is exposed as `StateFlow<HabitListUiState>` (read-only); the `MutableStateFlow` is `private`.
> - [ ] All state mutations use `.update { it.copy(...) }` — no direct `.value =` assignments.
> - [ ] The composable collects with `collectAsStateWithLifecycle()` and renders purely from `state`.
> - [ ] The Add action travels composable → `onEvent()` → ViewModel — no mutation logic lives in the composable.
> - [ ] No `LiveData` imports anywhere in the ViewModel or composable.
