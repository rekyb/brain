# 5.4 Lifecycle, process death and edge-to-edge

> [!abstract] TL;DR
> Android can kill your process at any moment; state you don't explicitly save is permanently lost. Use `rememberSaveable` for transient composable-owned state and `SavedStateHandle` in the ViewModel for screen-level state. Separately, Android 15 makes edge-to-edge the default — consume window insets or content will be obscured by system bars.

## Digest

### Activity and composition lifecycle

The Activity lifecycle (Created → Started → Resumed → Paused → Stopped → Destroyed) is unchanged from 2021, but in Compose you interact with it far less directly. The ViewModel survives configuration changes (rotation, language switch, multi-window resize) and continues to hold UI state across them. The composable is destroyed and recreated on every configuration change but immediately re-collects from the ViewModel's `StateFlow`, so the user sees no disruption.

The Compose *composition* lifecycle (entering composition → recomposing → leaving composition) is orthogonal to the Activity lifecycle. Effects that need to track it — `LaunchedEffect`, `DisposableEffect`, `collectAsStateWithLifecycle` — are covered in [[2.7 Side effects and lifecycle]].

### Process death and state restoration

Configuration change and process death are different events with different severity. A configuration change merely recreates the Activity; the ViewModel survives. **Process death** means the OS has killed the entire process — low memory, battery optimisation, explicit force-stop. All in-memory state is gone, including the ViewModel instance and its `StateFlow` values.

Android may recreate the last visible screen after process death using a saved Bundle, but only if you explicitly participated in saving state. Two tools handle this at different layers.

**`rememberSaveable`** serialises composable-owned state into the Bundle automatically, surviving both configuration changes and process death:

```kotlin
var inputText by rememberSaveable { mutableStateOf("") }
var selectedTab by rememberSaveable { mutableIntStateOf(0) }
```

Use it for ephemeral UI state the composable owns: text field contents, scroll position, toggle values, selected items. It uses a `Bundle` internally, so values must be `Bundle`-serialisable (primitives, `String`, `Parcelable`). For custom types, provide a `Saver`.

**`SavedStateHandle`** in the ViewModel is the equivalent for screen-level state that the ViewModel needs to access — navigation arguments, active filters, selected IDs:

```kotlin
@HiltViewModel
class HabitDetailViewModel @Inject constructor(
    savedStateHandle: SavedStateHandle,
    private val repository: HabitRepository
) : ViewModel() {

    val habitId: String = checkNotNull(savedStateHandle["habitId"])

    // Backed by SavedStateHandle — survives process death
    val filter = savedStateHandle.getStateFlow("activeFilter", FilterOption.ALL)

    fun setFilter(option: FilterOption) {
        savedStateHandle["activeFilter"] = option
    }
}
```

Hilt injects `SavedStateHandle` automatically into `@HiltViewModel` classes. The handle is restored from the Bundle *before* `init {}` runs — the ViewModel never sees the pre-death state as absent or `null` unless you designed it that way. `getStateFlow` wraps the handle value as a `StateFlow`, which the composable can collect with `collectAsStateWithLifecycle`.

**The boundary between the two tools:** state the ViewModel owns lives in `SavedStateHandle`. State the composable owns (with no ViewModel involvement) lives in `rememberSaveable`. Don't duplicate the same value in both.

### Configuration changes

The ViewModel survives them; any `remember { }` state does not (it's in-memory only). `rememberSaveable` fills the gap for composable-local state. For complex state that doesn't fit a Bundle — a large list loaded from the network, a decoded image — keep it in the ViewModel and let it re-emit through its `StateFlow` after the config change. The ViewModel's in-memory state is already there; only the composable was recreated.

### Edge-to-edge

From Android 15 (API 35), edge-to-edge is **enabled by default** for apps that target API 35. The app's window extends under the status bar and navigation bar. Content that doesn't account for system bar overlap will be partially hidden.

Call `enableEdgeToEdge()` in `Activity.onCreate`, before `setContent`, to adopt the behaviour consistently across API levels (it's a no-op below API 21 and adjusts status/navigation bar colours and appearance on supported APIs):

```kotlin
class MainActivity : ComponentActivity() {
    override fun onCreate(savedInstanceState: Bundle?) {
        super.onCreate(savedInstanceState)
        enableEdgeToEdge()          // before setContent
        setContent { HabitTrackerTheme { AppNavHost() } }
    }
}
```

Then consume insets in the layout. The simplest path is letting `Scaffold` handle it — its `contentPadding` lambda delivers a `PaddingValues` that accounts for status bar, navigation bar, and IME:

```kotlin
Scaffold { contentPadding ->
    LazyColumn(
        modifier = Modifier.fillMaxSize(),
        contentPadding = contentPadding   // items scroll under bars; padded at edges
    ) {
        items(habits) { HabitRow(it) }
    }
}
```

For screens without `Scaffold`, apply insets manually:

```kotlin
Column(
    modifier = Modifier
        .fillMaxSize()
        .windowInsetsPadding(WindowInsets.systemBars)
) { ... }
```

Never hardcode `dp` padding values for system bars — heights vary by device, orientation, and whether the gesture navigation bar is visible.

### Predictive back

Android 13+ introduced the predictive back gesture — swipe right from the edge previews the destination before the user commits the back action. Opting in requires two steps:

1. Add `android:enableOnBackInvokedCallback="true"` to the `<application>` element in `AndroidManifest.xml`.
2. Replace any `onBackPressed()` overrides or `KeyEvent` intercepts with the modern APIs: `BackHandler { }` in Compose, or `OnBackPressedCallback` with `OnBackPressedDispatcher` outside Compose.

Compose Navigation's `NavController` handles predictive back automatically once the manifest flag is set — no additional code needed for standard navigation. Custom back intercepts (e.g. "are you sure you want to discard?" dialogs) need to be migrated to `BackHandler`.

### What changed since 2021

- **`onSaveInstanceState` / `onRestoreInstanceState` are replaced.** `rememberSaveable` and `SavedStateHandle` handle the same concerns with far less boilerplate. The raw Bundle APIs still compile, but you'd only write them directly in edge cases that `rememberSaveable`'s `Saver` API doesn't cover.
- **Edge-to-edge is no longer opt-in on API 35.** In 2021 it required explicit `WindowCompat.setDecorFitsSystemWindows(window, false)` plus manual status/navigation bar colour management. Now targeting API 35 enables it unconditionally — any layout that doesn't handle insets ships broken.
- **`enableEdgeToEdge()` is the new setup call** (Activity KTX). It replaces the combination of `WindowCompat.setDecorFitsSystemWindows`, `window.statusBarColor`, and `window.navigationBarColor` that was the 2021 approach.
- **Predictive back is new (Android 13, 2022).** Any use of the deprecated `Activity.onBackPressed()` override will break the predictive-back animation — the preview will show but the intercept will be ignored or behave incorrectly.

## Drill

Work in the capstone.

**Part A — process death survival**

Pick a transient screen-level value that the ViewModel needs (a search query, an active filter, or the text of a draft habit name). Back it with `SavedStateHandle` in the ViewModel and expose it as a `StateFlow`. Collect it in the composable with `collectAsStateWithLifecycle`. Enable "Don't keep activities" in Developer Options. Type or set a value, press Home, then return to the app — confirm the value is restored exactly as it was. Test rotation too (both should pass with `SavedStateHandle`).

**Part B — edge-to-edge**

Call `enableEdgeToEdge()` in `MainActivity.onCreate`. Pick a screen that has a `LazyColumn` or a `Column` extending to the bottom of the window. Verify content is not hidden behind the navigation bar. Use `Scaffold`'s `contentPadding` or `Modifier.windowInsetsPadding` to pad correctly — do not hardcode any `dp` values for system bars.

> [!example] Success criteria
> - [ ] After simulated process death ("Don't keep activities" → Home → return), the `SavedStateHandle`-backed value is restored — the UI displays the value that was present before leaving.
> - [ ] Rotation does not lose the value (ViewModel survives rotation; `SavedStateHandle` also persists it across process death).
> - [ ] With `enableEdgeToEdge()` active, no content is obscured by the status bar or navigation bar on any tested screen.
> - [ ] Inset padding comes from `Scaffold` or `windowInsetsPadding` — no hardcoded status/nav bar pixel values in the layout.
> - [ ] No `onBackPressed()` override exists anywhere in the project; back handling uses `BackHandler` or NavController.
