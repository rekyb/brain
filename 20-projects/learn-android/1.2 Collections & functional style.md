# 1.2 Collections & functional style

> [!abstract] TL;DR
> Kotlin's collection pipeline — `map`, `filter`, `fold`, `groupBy`, `associate`, sequences — lets you process data without manual loops; mastering it now pays off immediately with Flow operators, Compose state transforms, and Room query results.

## Digest

Kotlin collections (`List`, `Set`, `Map`) are **read-only by default**. `listOf()`, `setOf()`, `mapOf()` produce read-only views; `mutableListOf()` and friends give you mutation. In practice you spend most of your time in the immutable API because functional operators return new collections.

### The essential pipeline operators

```kotlin
val names  = people.map { it.name }                    // transform each element
val adults = people.filter { it.age >= 18 }            // keep matching elements
val total  = amounts.fold(0.0) { acc, v -> acc + v }   // accumulate with a seed
val byCity = people.groupBy { it.city }                 // Map<String, List<Person>>
val byId   = people.associate { it.id to it }           // Map<Int, Person>
```

`reduce` is like `fold` without a seed value — it throws on an empty list, so prefer `fold` when that risk exists. `flatMap` flattens a list of lists:

```kotlin
val allTags = posts.flatMap { it.tags }
```

`runningFold` (added in Kotlin 1.4) produces a list of intermediate accumulator values — handy for running totals:

```kotlin
val running = amounts.runningFold(0.0) { acc, v -> acc + v }
// [0.0, first, first+second, ...]
```

### Lambdas and higher-order functions

A higher-order function takes or returns a function. The collection API is built on them:

```kotlin
fun <T, R> List<T>.myMap(transform: (T) -> R): List<R> {
    val result = mutableListOf<R>()
    for (item in this) result += transform(item)
    return result
}
```

**Trailing-lambda syntax**: when a lambda is the last argument, move it outside the parentheses. When it is the *only* argument, drop the parens entirely:

```kotlin
list.filter({ it > 0 })   // valid but verbose
list.filter { it > 0 }    // idiomatic
```

### Sequences for laziness

A sequence turns a pipeline lazy: each element flows through every step before the next element is fetched. For large collections with early termination (`first`, `take`) this avoids allocating intermediate lists:

```kotlin
val result = bigList.asSequence()
    .filter { it.active }
    .map { it.value * 2 }
    .take(10)
    .toList()   // only now does it iterate
```

Rule of thumb: reach for `asSequence()` when chaining three or more operators on a list of 1 000+ elements, or when you use `first` / `take`.

### Functional style and Flow

Every Flow operator you will use later (`map`, `filter`, `debounce`, `combine`) mirrors the collection API. The mental model is the same; the execution is asynchronous and lazy. Practising here makes the reactive layer feel familiar.

## Drill

Work in **Kotlin Playground** or a scratch `.kt` file. Start from the data below — do **not** modify the `Transaction` data class or the `transactions` list.

```kotlin
data class Transaction(
    val id: Int,
    val category: String,
    val amount: Double,   // positive = credit, negative = debit
    val merchant: String
)

val transactions = listOf(
    Transaction(1,  "food",      -12.50, "Kopi Kenangan"),
    Transaction(2,  "transport",  -3.00, "Grab"),
    Transaction(3,  "food",       -8.00, "Warung Padang"),
    Transaction(4,  "income",    500.00, "Salary"),
    Transaction(5,  "transport", -15.00, "Grab"),
    Transaction(6,  "food",      -22.00, "Pizza Hut"),
    Transaction(7,  "health",    -45.00, "Apotek"),
    Transaction(8,  "food",       -6.50, "Kopi Kenangan"),
    Transaction(9,  "income",    200.00, "Freelance"),
    Transaction(10, "transport",  -2.00, "Grab"),
)
```

Using **only** functional operators — no `for` loops, no `while`, no manual `mutableList` + `.add()` inside your computation logic — compute each of the following and print the result:

1. **Total spent:** sum of all negative amounts (as a positive number).
2. **Total by category:** a `Map<String, Double>` of the sum of amounts per category.
3. **Top 3 debit transactions** by absolute amount (largest spends first) as a list of merchant names.
4. **Running balance:** a `List<Double>` where each entry is the cumulative sum up to that transaction, in order. (Use `runningFold` or `scan`.)
5. **Repeat merchants:** unique merchant names that appear more than once, as a sorted list.

> [!example] Success criteria
> - [ ] All five computations produce the correct values (verify at least two by hand).
> - [ ] Zero `for` or `while` loops in your computation logic.
> - [ ] No manual `mutableListOf()` + `.add()` inside any computation.
> - [ ] `runningFold` (or `scan`) used for the running balance.
> - [ ] Code runs cleanly in Kotlin Playground with no exceptions.
