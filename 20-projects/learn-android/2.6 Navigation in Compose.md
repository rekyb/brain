# 2.6 Navigation in Compose

> [!abstract] TL;DR
> Compose Navigation replaces the Fragment back stack and XML nav graphs with `NavController` + `NavHost`: a Kotlin DSL that maps routes to composable destinations, with type-safe argument passing via `@Serializable` route objects.

## Digest

### NavController and NavHost

Add the dependency: `androidx.navigation:navigation-compose`. The two core concepts:

- **`NavController`** — manages the back stack and provides `navigate(route)`, `popBackStack()`, etc. Create it with `rememberNavController()`.
- **`NavHost`** — a composable that displays the current destination. It takes a `NavController`, a `startDestination`, and a DSL block defining each destination.

```kotlin
val navController = rememberNavController()

NavHost(navController, startDestination = HomeRoute) {
    composable<HomeRoute> {
        HomeScreen(onHabitClick = { id ->
            navController.navigate(DetailRoute(id = id))
        })
    }
    composable<DetailRoute> { backStackEntry ->
        val route: DetailRoute = backStackEntry.toRoute()
        DetailScreen(habitId = route.id, onBack = { navController.popBackStack() })
    }
}
```

### String routes vs type-safe routes

The original Navigation Compose API used string routes with string-encoded arguments (`"detail/{id}"`). It works but is stringly-typed — typos cause runtime crashes, not compile errors.

**Type-safe navigation** (navigation-compose 2.8+, kotlinx-serialization) replaces strings with `@Serializable` data classes or objects:

```kotlin
@Serializable
object HomeRoute

@Serializable
data class DetailRoute(val id: String)
```

Navigation becomes a Kotlin function call:

```kotlin
navController.navigate(DetailRoute(id = habit.id))
```

The library serializes the route to the back stack automatically. Argument extraction at the destination uses `backStackEntry.toRoute<DetailRoute>()`. This is the current recommended approach for all new code.

### Passing arguments

With type-safe routes, arguments are constructor parameters on the route data class — no string templates, no `arguments { navArgument(...) }` blocks:

```kotlin
@Serializable
data class HabitDetailRoute(
    val habitId: String,
    val readOnly: Boolean = false  // optional with default
)

// Navigate:
navController.navigate(HabitDetailRoute(habitId = "abc-123"))

// Receive:
composable<HabitDetailRoute> { entry ->
    val args = entry.toRoute<HabitDetailRoute>()
    HabitDetailScreen(habitId = args.habitId, readOnly = args.readOnly)
}
```

Supported argument types: primitives, strings, `@Serializable` classes, `Parcelable`, and `Enum`. `@Serializable` covers most cases.

### Back navigation

`navController.popBackStack()` navigates up one level. For the system back button/gesture, `NavHost` handles it automatically — the system back action calls `popBackStack()` by default. If you need custom back behaviour on a specific screen, use `BackHandler`:

```kotlin
BackHandler(enabled = isEditing) {
    // show discard-changes dialog instead of navigating back
}
```

### Passing NavController down

Avoid passing `NavController` deep into composable trees — it couples composables to navigation. The pattern is to pass navigate callbacks (lambdas) instead:

```kotlin
@Composable
fun HomeScreen(onHabitClick: (String) -> Unit) { /* ... */ }

// At the NavHost level:
composable<HomeRoute> {
    HomeScreen(onHabitClick = { id ->
        navController.navigate(HabitDetailRoute(habitId = id))
    })
}
```

This keeps `HomeScreen` testable and reusable without a `NavController` dependency.

### What changed since 2021

- **Fragments and the XML nav graph are gone.** The `Navigation` component's XML `<navigation>` files, `<fragment>` destinations, and `FragmentManager` transactions are replaced by the Compose DSL. `NavHostFragment` in XML is replaced by the `NavHost` composable.
- **No `SafeArgs` Gradle plugin.** The old type-safe argument approach required a code-generation plugin; now `@Serializable` data classes give the same safety as a first-class Kotlin language feature.
- **`BackHandler` replaces `OnBackPressedCallback`.** The API surface is simpler.
- **Shared ViewModel scoped to the nav graph** still works: `hiltViewModel()` or `viewModel()` with a back-stack entry argument lets you share state across destinations without a global singleton.

## Drill

Add `androidx.navigation:navigation-compose` to your project. You will also need `org.jetbrains.kotlinx:kotlinx-serialization-json` and the Kotlin serialization plugin.

**Task:** Build a two-screen flow: a list screen and a detail screen.

1. Define `@Serializable` route objects: `ListRoute` (object) and `DetailRoute(val itemId: Int)` (data class).
2. The list screen displays at least 5 tappable items. Tapping an item navigates to the detail screen, passing the item's `Int` id.
3. The detail screen displays the received id and a "Back" button that calls `popBackStack()`.
4. The system back gesture (or emulator back button) must also return to the list.

Do not pass `NavController` into `ListScreen` or `DetailScreen` — use lambda callbacks at the `NavHost` level.

> [!example] Success criteria
> - [ ] Tapping an item on the list screen navigates to the detail screen.
> - [ ] The correct `itemId` is displayed on the detail screen (verify with at least two different items).
> - [ ] Both the "Back" button and the system back gesture return to the list screen, preserving its scroll state.
> - [ ] Routes are `@Serializable` — no string route templates anywhere.
> - [ ] Neither `ListScreen` nor `DetailScreen` has a `NavController` parameter — navigation is driven by lambdas.
> - [ ] No XML nav graph files exist in the project.
