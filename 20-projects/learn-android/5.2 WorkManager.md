# 5.2 WorkManager

> [!abstract] TL;DR
> WorkManager is the standard solution for deferrable, guaranteed background work — work that must complete even if the app is killed or the device restarts. Use `CoroutineWorker` for suspend-friendly workers, attach `Constraints` to defer until conditions are met, and use unique work names to prevent duplicate jobs.

## Digest

### What WorkManager is for

WorkManager handles **deferrable, guaranteed work**: a sync that must upload a draft even after the app is swiped away, a periodic cleanup that should run whenever the device is idle and charging. The guarantees come from the OS — WorkManager delegates to `JobScheduler` (API 23+) under the hood and survives app process death and device reboots. Work that is enqueued but not yet run when the device powers off is rescheduled automatically on reboot.

It is *not* the right tool for everything:

- **Immediate in-process work** (user taps Save, you call the API): launch a coroutine in `viewModelScope`. WorkManager's overhead and scheduling delay are unnecessary.
- **Clock-exact reminders** (an alarm at precisely 07:30): use `AlarmManager` with exact alarm APIs, accepting the permission overhead they carry (see [[5.3 Notifications]]).
- **Long-running foreground work** (music playback, file download with a progress notification): use a `ForegroundService` — which a `Worker` can promote itself into via `setForeground()`, but that's a separate concern.

### CoroutineWorker

`CoroutineWorker` wraps a worker in a coroutine so that `doWork()` is a `suspend` function running on `Dispatchers.IO` by default. Return `Result.success()`, `Result.retry()`, or `Result.failure()`:

```kotlin
class SyncWorker(
    appContext: Context,
    params: WorkerParameters
) : CoroutineWorker(appContext, params) {

    override suspend fun doWork(): Result {
        return try {
            val count = syncRepository.fetchAndPersist()
            Result.success(workDataOf("synced_count" to count))
        } catch (e: IOException) {
            if (runAttemptCount < 3) Result.retry()
            else Result.failure()
        }
    }
}
```

`runAttemptCount` tracks how many times WorkManager has retried this particular execution. `workDataOf` attaches key-value output data that observers can read after the worker succeeds.

### Constraints

Attach a `Constraints` object to defer the work until system conditions are favourable:

```kotlin
val constraints = Constraints.Builder()
    .setRequiredNetworkType(NetworkType.CONNECTED)
    .setRequiresBatteryNotLow(true)
    .build()

val request = OneTimeWorkRequestBuilder<SyncWorker>()
    .setConstraints(constraints)
    .build()
```

Constraints are checked **before** a worker starts; a worker only begins once all its constraints are met. If a constraint **stops being met while the worker is running**, WorkManager **stops** the worker — `isStopped()` becomes `true` and, in a `CoroutineWorker`, the `doWork` coroutine is cancelled. The work is then re-enqueued and retried once the constraints are satisfied again. Design workers to be idempotent — safe to run more than once. For long-running work, periodically check `isStopped()` so the worker can clean up promptly when WorkManager signals cancellation.

### One-time vs periodic work

| Type | Builder | Minimum interval |
|---|---|---|
| One-time | `OneTimeWorkRequestBuilder<T>()` | — |
| Periodic | `PeriodicWorkRequestBuilder<T>(15, TimeUnit.MINUTES)` | 15 minutes |

The 15-minute floor is imposed by the OS. On battery-optimised devices, the OS may batch periodic work further to align with other wake windows (Doze mode). Periodic workers are for approximate timing — don't rely on them firing at a precise clock time.

### Unique work

Enqueue the same logical work twice and you get two queued workers. Prevent it by naming the work and choosing a conflict policy:

```kotlin
// Periodic
WorkManager.getInstance(context).enqueueUniquePeriodicWork(
    "daily_habit_reminder",                    // unique name
    ExistingPeriodicWorkPolicy.KEEP,           // or UPDATE, CANCEL_AND_REENQUEUE
    periodicRequest
)

// One-time
WorkManager.getInstance(context).enqueueUniqueWork(
    "initial_sync",
    ExistingWorkPolicy.KEEP,
    oneTimeRequest
)
```

`KEEP` leaves any existing pending or running work untouched. `UPDATE` (periodic only, WorkManager 2.8+) updates the work spec without resetting run history. `CANCEL_AND_REENQUEUE` stops the current worker and restarts from scratch.

### Observing WorkInfo

WorkManager exposes `Flow<List<WorkInfo>>` for reactive observation:

```kotlin
WorkManager.getInstance(context)
    .getWorkInfosByTagFlow("sync_tag")
    .collect { infos ->
        val info = infos.firstOrNull() ?: return@collect
        when (info.state) {
            WorkInfo.State.SUCCEEDED -> {
                val count = info.outputData.getInt("synced_count", 0)
                // update UI
            }
            WorkInfo.State.FAILED -> { /* show error */ }
            else -> { /* ENQUEUED, RUNNING, BLOCKED, CANCELLED */ }
        }
    }
```

Tag a request with `.addTag("sync_tag")` during building to query by tag. You can also query by `UUID` (the request ID returned at enqueue time) using `getWorkInfoByIdFlow(id)`.

### What changed since 2021

- **WorkManager is now the standard; `AsyncTask` is gone.** `AsyncTask` was deprecated in API 30 (2020) and removed in API 33. `JobScheduler` still exists but WorkManager wraps it with a far better API. There is no reason to write a raw `JobScheduler` job for new background work.
- **`CoroutineWorker` matured.** The suspend-based worker was introduced in 2019 and is now the default choice. `ListenableWorker` and `RxWorker` are niche alternatives for non-coroutine codebases.
- **`ExistingPeriodicWorkPolicy.UPDATE`** was added in WorkManager 2.8 (2022). Previously, updating a pending periodic worker's parameters required `CANCEL_AND_REENQUEUE`, resetting the run interval counter.
- **`getWorkInfosByTagFlow`** (returning a Kotlin `Flow`) was stabilised in WorkManager 2.8 alongside the `ListenableFuture`-based APIs that existed in 2021. Prefer the Flow version in Kotlin codebases.

## Drill

Work in the capstone or a scratch module.

**Part A — one-time worker**

Create a `CoroutineWorker` that logs a message to Logcat and returns `Result.success()` with at least one output data value (e.g. a current timestamp as a `Long`). Enqueue it as a `OneTimeWorkRequest`. Observe `WorkInfo` state from the UI — display the worker's state progression (`ENQUEUED` → `RUNNING` → `SUCCEEDED`) and show the output data value once the worker succeeds.

**Part B — periodic worker with constraint**

Convert the worker into a `PeriodicWorkRequest` at the minimum 15-minute interval. Attach a `Constraints` requiring at least one condition (network connectivity or battery not low). Enqueue it as unique periodic work under a stable name using `KEEP` policy so repeated enqueues during development do not stack up. Force-stop the app and reopen — verify in Logcat that the work runs without needing the app in the foreground.

> [!example] Success criteria
> - [ ] The `CoroutineWorker`'s `doWork()` is a `suspend` function — no raw `Thread`, `Executor`, or `AsyncTask` usage.
> - [ ] One-time work output data (e.g. a timestamp) is visible in the UI after the `SUCCEEDED` state is observed.
> - [ ] The periodic work is enqueued under a unique name; calling the enqueue code twice does not create two workers (Logcat shows only one run per interval).
> - [ ] The constraint is respected — placing the device in airplane mode (for a network constraint) prevents the worker from running until connectivity returns.
> - [ ] After a force-stop and relaunch, the periodic work continues to execute without being re-registered.
