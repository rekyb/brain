# 6.3 Accessibility and adaptive UI

> [!abstract] TL;DR
> Good accessibility in Compose means explicit semantics on every interactive element; good adaptive UI means reading `WindowSizeClass` to pick a layout rather than hard-coding phone layouts and hoping they hold up on tablets and foldables.

## Digest

### Semantics and content descriptions

Compose builds an accessibility (semantic) tree alongside the visual UI. Standard composables â€” `Button`, `Text`, `Checkbox`, `Switch` â€” populate the tree automatically. Where they don't, you annotate with `Modifier.semantics { }`.

Icon-only buttons are the most common gap. Passing a `contentDescription` to the `Icon` is the minimum fix; TalkBack reads it when the button is focused:

```kotlin
Icon(
    painter = painterResource(R.drawable.ic_delete),
    contentDescription = "Delete habit"
)
```

For a custom composable that represents a cohesive unit of content â€” say, a habit row that shows a name and a streak count â€” merging its children's semantics into one node produces a single TalkBack announcement instead of two separate ones:

```kotlin
Row(
    modifier = Modifier.semantics(mergeDescendants = true) {}
) {
    Text(habit.name)
    Text("${habit.streak} day streak")
}
```

Use `Modifier.semantics { contentDescription = "..." }` to override auto-generated semantics when the visual label is absent or misleading. Use `Modifier.clearAndSetSemantics { }` to replace the entire subtree's semantics with a single custom description at the parent.

### Touch targets

The minimum touch target for an interactive element is **48 Ă— 48 dp**. `IconButton` enforces this by default. For custom clickable composables where the visual element is smaller (e.g., a 24 dp icon wrapped in a `Box`), apply `Modifier.minimumInteractiveComponentSize()` from Material3 â€” it expands the touch target without changing the visual footprint:

```kotlin
Box(
    modifier = Modifier
        .minimumInteractiveComponentSize()
        .clickable { onDelete() }
) {
    Icon(Icons.Default.Delete, contentDescription = "Delete habit")
}
```

### Font scaling and dynamic type

Never hard-code text sizes in `dp` â€” use `sp` so the system font-scale setting takes effect. More importantly, never clip text by placing it in a fixed-height container without an overflow strategy:

```kotlin
// wrong â€” clips at 200 % scale
Box(modifier = Modifier.height(48.dp)) {
    Text(habit.name)
}

// right â€” expands or truncates gracefully
Text(
    text = habit.name,
    maxLines = 2,
    overflow = TextOverflow.Ellipsis
)
```

Test at 200 % font scale in the emulator (Settings â†’ Accessibility â†’ Font size and style, or via the Quick Settings tile). Material3's typography system â€” covered in [[2.5 Material 3 and theming]] â€” uses `sp` throughout; custom text styles should match.

### Window size classes and adaptive layouts

`WindowSizeClass` provides three named breakpoints â€” `Compact`, `Medium`, `Expanded` â€” for both width and height, covering phones in portrait (Compact width), large phones / small tablets in landscape (Medium), and tablets / desktop windows (Expanded). Retrieve it in your Activity and pass it into the composable tree:

```kotlin
// in Activity.onCreate, or via LocalContext + rememberWindowSizeClass:
val windowSizeClass = calculateWindowSizeClass(this)
setContent {
    MyApp(windowSizeClass = windowSizeClass)
}
```

Branch on `windowSizeClass.widthSizeClass` to choose a layout strategy:

```kotlin
@Composable
fun HabitApp(windowSizeClass: WindowSizeClass) {
    if (windowSizeClass.widthSizeClass == WindowWidthSizeClass.Expanded) {
        HabitListDetailLayout()   // two-pane side-by-side
    } else {
        HabitSinglePaneNavHost()  // standard nav stack
    }
}
```

For a structured list-detail pattern, `androidx.compose.material3.adaptive` provides `NavigableListDetailPaneScaffold` and `ListDetailPaneScaffold`. These composables manage pane visibility, back navigation, and focus automatically across all three breakpoints â€” including foldables where the hinge sits between panes.

For foldables specifically, the `FoldingFeature` API (Jetpack WindowManager) reports hinge position and fold state (flat vs. half-open). The Material3 adaptive scaffolds absorb most foldable cases implicitly when you use `WindowSizeClass` breakpoints, so you rarely need to read `FoldingFeature` directly for a list-detail layout.

### TalkBack testing

Enable TalkBack on a physical device or emulator (Settings â†’ Accessibility â†’ TalkBack, or the volume-key shortcut). Navigate your UI using swipes â€” right swipe moves to the next accessibility node, left swipe moves back, double-tap activates. TalkBack reads the semantic tree aloud; if it announces "unlabelled button" or reads a raw number instead of meaningful text, there is a semantics gap.

Common issues to look for:
- Icon-only buttons with no `contentDescription`.
- Custom composables exposing internal implementation strings rather than a human-readable description.
- Interactive elements so close together that TalkBack focus rectangles overlap (fix: add padding, or merge semantics).
- Decorative images with a non-null `contentDescription` (should be `null` to be skipped by TalkBack).

### What changed since 2021

- **`WindowSizeClass` replaces manual display-metrics breakpoints.** In 2021 you likely computed breakpoints from `Resources.displayMetrics.widthPixels` converted to dp. `WindowSizeClass` is the supported API, and it handles multi-window, desktop windowing, and foldables correctly.
- **`NavigableListDetailPaneScaffold` (material3-adaptive) is new.** Adaptive list-detail previously required maintaining two separate navigation graphs and swapping between them based on screen size â€” significant boilerplate. The scaffold does this automatically.
- **`Modifier.minimumInteractiveComponentSize()`** is a Material3 Compose addition. The View-world equivalent was a custom `TouchDelegate` â€” considerably more code.
- **`Modifier.semantics { }`** is the Compose-native accessibility API. In the View world you used `ViewCompat.setAccessibilityDelegate(view, ...)` and `AccessibilityNodeInfoCompat` â€” imperative, brittle, and hard to compose. `Modifier.semantics` is declarative and composable like everything else.
- **`WindowSizeClass`** now comes from `androidx.window` / `androidx.window:window-core` â€” the old `androidx.compose.material3:material3-window-size-class` artifact is deprecated and should not be used in 2026 projects. The Compose-friendly entry points (`currentWindowAdaptiveInfo()` / `rememberWindowSizeClass()`) are provided by `androidx.compose.material3.adaptive`. Neither existed in 2021.

## Drill

Work in the capstone habit-tracker project.

**Part A â€” semantics audit**

Audit the habit-list screen: every icon-only button (delete, add, mark-done) must have a meaningful `contentDescription`. Any row composable that groups a habit name and streak text should use `mergeDescendants = true` so TalkBack reads the row as one cohesive announcement. Enable TalkBack on a device or emulator and navigate the entire screen â€” every interactive element must be reachable and announce a useful label. Fix any "unlabelled" or unintended announcements.

**Part B â€” font-scale test**

Run the app at 200 % font scale. Scan each screen for clipped or hidden text inside fixed-height containers. Fix at least one instance using `overflow = TextOverflow.Ellipsis`, `maxLines`, or by allowing the container to size to content.

**Part C â€” adaptive layout**

Read `WindowSizeClass` in your main Activity (or via `rememberWindowSizeClass()`) and thread it down to your root composable. On `WindowWidthSizeClass.Expanded` (test with a resizable emulator set to tablet width), show a two-pane layout with the habit list on the left and a habit detail view on the right. On `Compact`, keep the existing single-pane navigation.

> [!example] Success criteria
> - [ ] TalkBack navigates the habit-list screen without announcing "unlabelled" for any interactive element.
> - [ ] At 200 % font scale, no meaningful text is clipped or hidden by a fixed-height container.
> - [ ] On `WindowWidthSizeClass.Expanded`, the app displays a two-pane layout; on `Compact`, it displays the single-pane layout.
> - [ ] The layout decision is driven by `WindowSizeClass` â€” no manual pixel or raw dp breakpoint calculations.
> - [ ] Icon-only touch targets use `Modifier.minimumInteractiveComponentSize()` or are at least 48 Ă— 48 dp.
