# 2.3 Layouts and modifiers

> [!abstract] TL;DR
> Compose replaces XML layout files and `ConstraintLayout` with three core layout composables — `Column`, `Row`, `Box` — plus a `Modifier` chain that controls appearance and behaviour, where **order matters**.

## Digest

### Column, Row, Box

The three fundamental layout composables cover the cases XML needed multiple layout types for:

- **`Column`** stacks children vertically (replaces `LinearLayout orientation="vertical"`).
- **`Row`** stacks children horizontally (replaces `LinearLayout orientation="horizontal"`).
- **`Box`** layers children on top of each other, last child on top (replaces `FrameLayout`).

Nested combinations handle most real layouts without `ConstraintLayout`. For genuinely complex constraint-based arrangements, `ConstraintLayout` is available as a Compose composable too, but reach for it only when nesting becomes deeply awkward.

```kotlin
@Composable
fun UserRow(name: String, subtitle: String) {
    Row(
        verticalAlignment = Alignment.CenterVertically,
        horizontalArrangement = Arrangement.spacedBy(12.dp)
    ) {
        AvatarPlaceholder()
        Column {
            Text(name, style = MaterialTheme.typography.bodyLarge)
            Text(subtitle, style = MaterialTheme.typography.bodySmall)
        }
    }
}
```

`Arrangement` controls how children are *distributed* along the main axis (`SpaceBetween`, `SpaceAround`, `spacedBy`, `Start`, `End`, `Center`). `Alignment` controls positioning on the *cross* axis (`Top`, `CenterVertically`, `Bottom` for `Row`; `Start`, `CenterHorizontally`, `End` for `Column`).

### The Modifier chain

A `Modifier` is an ordered list of transformations applied to a composable. Pass it via the `modifier` parameter (every standard composable accepts one):

```kotlin
Text(
    text = "Hello",
    modifier = Modifier
        .fillMaxWidth()
        .padding(horizontal = 16.dp, vertical = 8.dp)
        .background(MaterialTheme.colorScheme.surfaceVariant)
        .clickable { /* ... */ }
)
```

Common modifiers:
- `fillMaxWidth()` / `fillMaxHeight()` / `fillMaxSize()` — expand to parent's constraint.
- `size(dp)` / `width(dp)` / `height(dp)` — fixed size.
- `padding(dp)` — inner spacing (applied before or after background, depending on order).
- `weight(Float)` — inside `Row`/`Column`, distributes remaining space proportionally.
- `background(Color)` — fill with a colour.
- `clip(shape)` — clip to a shape (e.g. `CircleShape`, `RoundedCornerShape(8.dp)`).
- `clickable { }` — make the composable interactive.
- `semantics { }` — accessibility metadata.

### Order is load-bearing

Unlike XML attributes (which are unordered), modifier functions are applied **left-to-right / top-to-bottom** in sequence, and the order visibly changes the result:

```kotlin
// background() is outermost — fills the full composable area (including the 16dp inset zone) with red;
// padding() then insets the content inward. The 16dp border IS red.
Modifier.background(Color.Red).padding(16.dp)

// padding() is outermost — creates a 16dp transparent ring first;
// background() then fills only the inner content area. The outer ring is transparent, not red.
Modifier.padding(16.dp).background(Color.Red)
```

This is one of the most common sources of confusion for developers coming from XML. The mental model: each modifier wraps the thing to its right.

### Spreading space with weight

`Modifier.weight(1f)` inside a `Row` or `Column` gives a child a proportional share of *remaining* space after fixed-size siblings are measured:

```kotlin
Row {
    Text("Label", modifier = Modifier.weight(1f))  // expands to fill remaining width
    Switch(checked = true, onCheckedChange = {})    // fixed size, measured first
}
```

### What changed since 2021

- **No XML layout files.** `res/layout/` is gone; layouts are Kotlin code.
- **No `ConstraintLayout` by default.** Simple nesting of `Column`/`Row`/`Box` handles most cases that needed `ConstraintLayout` in XML. This is less verbose and more readable.
- **`Modifier` replaces XML attributes and `LayoutParams`.** Padding, gravity, weight, background, click — all in one ordered chain rather than spread across XML attributes and programmatic calls.
- **No `ViewGroup` subclassing.** Custom layout logic uses the `Layout` composable with a measurement/placement lambda, not a custom `ViewGroup`.

## Drill

Work in the Compose project. Do not create any XML.

**Part A — settings row**

Reproduce this settings row: a leading icon placeholder (24×24 `Box` with a background colour), a `Column` with a title and subtitle text, and a `Switch` at the trailing edge. The icon and the `Switch` should be at fixed sizes; the text column should fill remaining width (`weight`). The entire row should have 16 dp horizontal padding and 12 dp vertical padding, and a ripple on tap.

**Part B — modifier order**

Create two side-by-side `Box` composables (or two separate previews). Apply `background(Color.Blue).padding(24.dp)` to one and `padding(24.dp).background(Color.Blue)` to the other. Put them in a `@Preview`. Be able to explain, in a comment in the file, exactly why they look different.

> [!example] Success criteria
> - [ ] Part A: the row renders with icon on the left, title+subtitle in the middle (expanding), switch on the right — all vertically centred.
> - [ ] Part A: the row responds to tap (ripple visible on emulator).
> - [ ] Part A: no hardcoded sizes except the icon placeholder; text column uses `weight`.
> - [ ] Part B: both modifier-order variants render visibly differently in the preview (blue area differs).
> - [ ] Part B: a Kotlin comment in the file correctly explains why the results differ.
> - [ ] No XML layout files exist in the project.
