# 3.3 Dependency injection with Hilt

> [!abstract] TL;DR
> Hilt is Dagger with all Android boilerplate pre-written: annotate your classes and Hilt constructs and wires them; your ViewModel receives its repository without ever calling `new` or writing a factory.

## Digest

### Why dependency injection

A class that constructs its own dependencies is hard to test: you cannot swap the real database for a fake without changing the class itself. Dependency injection inverts that â€” the class declares what it needs and callers supply it. With Hilt, *the framework* supplies dependencies at the right scope; you declare intent with annotations.

### Gradle setup

Hilt requires a Gradle plugin and two dependencies. For projects using KSP (the preferred annotation processor for new projects):

```kotlin
// build.gradle.kts (project level)
plugins {
    id("com.google.dagger.hilt.android") version "2.51.1" apply false
}

// build.gradle.kts (app module)
plugins {
    id("com.google.dagger.hilt.android")
    id("com.google.devtools.ksp")
}

dependencies {
    implementation("com.google.dagger:hilt-android:2.51.1")
    ksp("com.google.dagger:hilt-android-compiler:2.51.1")
}
```

If your project still uses kapt, replace `ksp(...)` with `kapt(...)` and the KSP plugin with `kotlin("kapt")`. KSP is faster at incremental builds â€” prefer it for new projects.

### @HiltAndroidApp â€” the application root

Every Hilt app starts here. Annotate a custom `Application` subclass; this triggers Hilt's code generation and establishes the top-level DI container:

```kotlin
@HiltAndroidApp
class HabitApp : Application()
```

Register it in `AndroidManifest.xml`:

```xml
<application android:name=".HabitApp" ...>
```

Without this, Hilt cannot resolve any injections and the app will crash on startup.

### @AndroidEntryPoint â€” injecting into Android entry points

Activities, Fragments, Services, and BroadcastReceivers that need injected fields must be annotated with `@AndroidEntryPoint`:

```kotlin
@AndroidEntryPoint
class MainActivity : ComponentActivity() { ... }
```

In a pure Compose app `MainActivity` is often the only class needing this annotation, because ViewModels (not Activities) receive the real dependencies, and Compose composables are not Android entry points.

### @Inject â€” constructor injection

Annotate a class constructor with `@Inject` and Hilt will know how to instantiate it:

```kotlin
class HabitRepositoryImpl @Inject constructor(
    private val habitDao: HabitDao
) : HabitRepository { ... }
```

Hilt resolves `HabitDao` transitively â€” as long as Hilt also knows how to build `HabitDao`. Constructor injection is the cleanest form: no boilerplate, no factory, the dependency graph is self-documenting from the constructor signatures.

For types you do not own â€” third-party libraries, types built with a factory or builder, interfaces â€” you cannot annotate the constructor. That is where modules come in.

### @Module, @Provides, and @Binds

A **module** tells Hilt how to satisfy a type it cannot figure out from constructor annotations alone. Every module carries `@InstallIn(SomeComponent::class)`, which tells Hilt which **component (container)** to install the module in â€” that determines the lifecycle the module participates in, but does not by itself cause bindings to be shared. Sharing requires a **scope annotation** on the individual binding: `@Singleton` on a `@Provides` or `@Binds` method tells Hilt to create one instance and reuse it for every injection. At this stage, `@Singleton` installed in `SingletonComponent` is the one scope you need â€” a single app-wide instance for your repository. The broader system of components and custom scopes (ActivityComponent, ViewModelScoped, etc.) is Module 2 territory.

`@Provides` is for types where you call a factory or builder:

```kotlin
@Module
@InstallIn(SingletonComponent::class)
object NetworkModule {

    @Provides
    @Singleton
    fun provideRetrofit(): Retrofit =
        Retrofit.Builder()
            .baseUrl("https://api.example.com/")
            .build()
}
```

`@Binds` is the leaner alternative when you want to map an interface to a concrete implementation that already has `@Inject` on its constructor:

```kotlin
@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {

    @Binds
    @Singleton
    abstract fun bindHabitRepository(
        impl: HabitRepositoryImpl
    ): HabitRepository
}
```

`@Binds` generates less code than `@Provides` â€” prefer it for interface-to-implementation mappings. The abstract class (not object) is required because `@Binds` functions must be abstract.

### @HiltViewModel â€” injecting into ViewModels

Annotate the ViewModel with `@HiltViewModel` and its constructor with `@Inject`:

```kotlin
@HiltViewModel
class HabitListViewModel @Inject constructor(
    private val repository: HabitRepository
) : ViewModel() { ... }
```

In the composable, retrieve it with `hiltViewModel()` from `androidx.hilt:hilt-navigation-compose`:

```kotlin
@Composable
fun HabitListScreen(viewModel: HabitListViewModel = hiltViewModel()) { ... }
```

Hilt resolves `HabitRepository` (via the `@Binds` module above), constructs `HabitRepositoryImpl`, and passes it to the ViewModel. You never instantiate the ViewModel or its dependencies directly.

### What changed since 2021

- **Manual DI and raw Dagger are no longer the recommended path.** Before Hilt, the options were: pass dependencies by hand (manual DI â€” viable for tiny projects, painful at scale) or set up raw Dagger with Components, Subcomponents, `@Component.Builder`, and manually written inject methods. Hilt (released 2020, mainstream by 2022) is Dagger with all of that pre-built for standard Android components.
- **`@HiltViewModel` + `hiltViewModel()` replaces `ViewModelProvider.Factory` boilerplate** â€” no more implementing a factory just to pass a repository constructor argument.
- **KSP replaces kapt** as the preferred annotation processor (Dagger/Hilt added stable KSP support in 2023). Faster incremental builds, better IDE integration.
- **`hiltViewModel()` replaces the `viewModels()` KTX delegate** in Compose + Hilt projects. It looks up the Hilt-generated factory automatically.

## Drill

Work in the capstone project.

**Part A â€” add Hilt to the project**

Add the Hilt Gradle plugin and dependencies to your project and app `build.gradle.kts` files (use KSP). Create a custom `Application` subclass annotated with `@HiltAndroidApp` and register it in the manifest. Annotate `MainActivity` with `@AndroidEntryPoint`. Sync and verify the project builds with no errors.

**Part B â€” create the repository binding**

Define a `HabitRepository` interface if it does not exist yet. Create `HabitRepositoryImpl` implementing it, with `@Inject` on its constructor (it can accept no parameters for now â€” in-memory impl). Write a Hilt module that uses `@Binds` to map `HabitRepository` to `HabitRepositoryImpl`, installed in `SingletonComponent`.

**Part C â€” inject into the ViewModel**

Annotate `HabitListViewModel` with `@HiltViewModel` and its constructor with `@Inject`, accepting `HabitRepository` as a parameter. Remove any hardcoded construction of the repository inside the ViewModel. Update the composable to obtain the ViewModel via `hiltViewModel()`. Run the app and confirm it launches without a crash and the ViewModel receives the repository.

> [!example] Success criteria
> - [ ] The app builds and launches without a crash or Hilt-related exception.
> - [ ] `HabitListViewModel` constructor declares `HabitRepository` (the interface) as a parameter â€” not `HabitRepositoryImpl`.
> - [ ] The ViewModel is never instantiated with `HabitListViewModel(...)` anywhere in UI code.
> - [ ] The Hilt module uses `@Binds` (not `@Provides`) for the interfaceâ†’impl binding.
> - [ ] `MainActivity` is annotated with `@AndroidEntryPoint`.
> - [ ] No `ViewModelProvider.Factory` implementation exists anywhere in the project.
