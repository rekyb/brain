# 1.4 Coroutines fundamentals

> [!abstract] TL;DR
> Kotlin coroutines give you sequential-looking async code without raw threads or nested callbacks; understanding `suspend`, `CoroutineScope`, and the Dispatcher trio unlocks every async pattern in the rest of this path.

## Digest

### The core idea

A `suspend` function can pause without blocking the thread. The calling coroutine is suspended, the thread is returned to the pool, and when the result is ready the coroutine resumes ‚Äî possibly on a different thread. From the caller's perspective it reads like synchronous code:

```kotlin
suspend fun fetchUser(id: Int): User {
    return userService.getUser(id)   // suspends here, doesn't block the thread
}

suspend fun loadAndDisplay(id: Int) {
    val user = fetchUser(id)         // sequential-looking, non-blocking
    println(user.name)
}
```

You can only call a `suspend` function from another `suspend` function or from inside a coroutine builder.

### Coroutine builders

`launch` starts a coroutine and returns a `Job` (fire-and-forget). `async` starts a coroutine and returns a `Deferred<T>` ‚Äî call `.await()` to suspend until the value is ready.

```kotlin
// Fire-and-forget ‚Äî use for side-effects
scope.launch {
    saveHabit(habit)
}

// Parallel work ‚Äî use when you need the result
val deferred: Deferred<User> = scope.async { fetchUser(42) }
val user = deferred.await()
```

To run two operations concurrently, start both with `async` before calling either `.await()`:

```kotlin
val priceA = async { fetchPriceA() }
val priceB = async { fetchPriceB() }
val total = priceA.await() + priceB.await()   // runs in parallel
```

Calling `.await()` one after the other (without starting both first) runs them sequentially.

### Dispatchers

Dispatchers control which thread pool a coroutine runs on:

- `Dispatchers.Main` ‚Äî the Android UI thread. Use for anything touching Compose or Views.
- `Dispatchers.IO` ‚Äî a large pool for blocking I/O (network, disk, Room queries).
- `Dispatchers.Default` ‚Äî CPU-bounded pool for computation (sorting, parsing, image decoding).

Switch dispatchers with `withContext` ‚Äî it's a `suspend` function that runs its block on the given dispatcher and resumes on the caller's dispatcher when done:

```kotlin
suspend fun loadHabits(): List<Habit> = withContext(Dispatchers.IO) {
    database.habitDao().getAll()   // runs on IO thread pool
}   // automatically returns to the caller's dispatcher
```

### Structured concurrency

Coroutines have a parent-child relationship through `CoroutineScope`. Cancelling a scope cancels all its children. A parent does not finish until all its children finish. A child exception propagates to the parent (unless you use `SupervisorJob`). This makes leaks and orphaned work much harder to produce than with raw threads.

In Android, `viewModelScope` is a `CoroutineScope` tied to the ViewModel's lifecycle ‚Äî it's cancelled automatically when the ViewModel is cleared. You rarely need to cancel or manage coroutines manually.

### Cancellation

`Job.cancel()` requests cancellation; it's cooperative ‚Äî `suspend` functions from the kotlinx.coroutines library check for cancellation at suspension points. CPU-heavy loops need to opt in:

```kotlin
launch {
    repeat(1_000_000) { i ->
        ensureActive()   // throws CancellationException if cancelled
        heavyWork(i)
    }
}
```

`CancellationException` is special ‚Äî it's always rethrown even when caught by a bare `catch (e: Exception)`. Never swallow it.

### What changed since 2021

- **`AsyncTask` is gone.** Deprecated in API 30, removed from the SDK in API 33. Coroutines replaced it entirely ‚Äî there is no migration path other than rewriting with `launch`/`async`.
- **`viewModelScope` and `lifecycleScope`** ship in `androidx.lifecycle` ‚Äî you no longer create or manage a `CoroutineScope` manually inside a ViewModel, Fragment, or Activity.
- **`Dispatchers.IO.limitedParallelism(n)`** (coroutines 1.6+) lets you cap parallelism for a specific task without building a custom executor.
- **`suspendCancellableCoroutine`** is now preferred over `suspendCoroutine` when wrapping callbacks ‚Äî it properly propagates cancellation to the underlying callback source.
- **`runBlocking` in production:** never use it on the Main thread or inside an existing coroutine scope. It's fine for `main()` functions and tests.
- **Kotlin 2.0:** stricter lint warnings when `Deferred` is returned from `async` and `.await()` is never called.

## Drill

Work in **Kotlin Playground** or a scratch `.kt` file. You will need `kotlinx-coroutines-core` (Playground includes it by default).

**Part A ‚Äî concurrent async**

Write a `main()` using `runBlocking` that calls these two stub functions concurrently and prints their combined result and the elapsed time.

```kotlin
suspend fun fetchPriceA(): Double {
    delay(500)      // simulates network latency ‚Äî do not modify
    return 19.99
}

suspend fun fetchPriceB(): Double {
    delay(500)      // simulates network latency ‚Äî do not modify
    return 34.50
}
```

The concurrent version should complete in ~500 ms. A sequential version would take ~1 000 ms. Print the elapsed time alongside the combined result to prove the difference.

**Part B ‚Äî wrapping a callback**

This legacy API uses a callback. You cannot change it:

```kotlin
fun interface PriceCallback {
    fun onResult(price: Double?, error: String?)
}

fun fetchPriceLegacy(symbol: String, callback: PriceCallback) {
    Thread {
        Thread.sleep(300)
        if (symbol == "ERR") callback.onResult(null, "Symbol not found")
        else callback.onResult(42.00, null)
    }.start()
}
```

Write `suspend fun fetchPriceSuspend(symbol: String): Double` that wraps `fetchPriceLegacy` using `suspendCancellableCoroutine`, throwing an exception when `error` is non-null.

Call it from `runBlocking` ‚Äî once with `"BTC"` (expect `42.00`) and once with `"ERR"` (catch and print the error message).

> [!example] Success criteria
> - [ ] Part A: both `async` blocks are started before either `.await()` is called; elapsed time is ~500 ms, not ~1 000 ms.
> - [ ] Part A: swapping to sequential execution (two separate `fetchPriceX()` calls awaited in order) doubles the elapsed time.
> - [ ] Part B: `fetchPriceSuspend("BTC")` returns `42.00`; `fetchPriceSuspend("ERR")` throws and the error message is printed.
> - [ ] `suspendCancellableCoroutine` used, not `suspendCoroutine`.
> - [ ] No `Thread.sleep` in your own code (only inside the provided stub).
> - [ ] `runBlocking` used only at the `main` entry point, not nested inside another coroutine.
