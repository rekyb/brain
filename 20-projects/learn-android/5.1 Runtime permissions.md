# 5.1 Runtime permissions

> [!abstract] TL;DR
> Since Android 6, dangerous permissions are granted at runtime by the user, not at install time. In Compose, the Activity Result APIs replace the old `onRequestPermissionsResult` callback — use `rememberLauncherForActivityResult` with the `RequestPermission` or `RequestMultiplePermissions` contract.

## Digest

### The runtime permission model

Android classifies permissions into *normal* (granted at install) and *dangerous* (location, camera, microphone, storage, contacts…). Any dangerous permission must be explicitly granted by the user at runtime; attempting to use a protected API without it throws a `SecurityException`. The check is always `ContextCompat.checkSelfPermission(context, permission) == PackageManager.PERMISSION_GRANTED`.

If the minimum SDK for a modern project is 26 or higher — which it should be — you always operate in the runtime-permission world. There is no install-time grant path to fall back on.

### Requesting with the Activity Result APIs

The old flow — `ActivityCompat.requestPermissions(activity, permissions, requestCode)` followed by `onRequestPermissionsResult` — required routing a result through an overridden method on the Activity or Fragment. In Compose this means the result arrives in a completely different part of the code from where the request was triggered, with manual request-code matching in between.

The Activity Result APIs (AndroidX) replace this with typed contracts registered before the composable's first composition. In Compose, `rememberLauncherForActivityResult` remembers the launcher across recompositions:

```kotlin
@Composable
fun NotificationPermissionRequest(onResult: (Boolean) -> Unit) {
    val launcher = rememberLauncherForActivityResult(
        contract = ActivityResultContracts.RequestPermission(),
        onResult = onResult          // Boolean: true = granted
    )
    Button(onClick = {
        launcher.launch(Manifest.permission.POST_NOTIFICATIONS)
    }) {
        Text("Enable notifications")
    }
}
```

For multiple permissions at once, swap in `ActivityResultContracts.RequestMultiplePermissions()` — the callback receives `Map<String, Boolean>` instead of a single `Boolean`.

The launcher must be registered at composition time (not inside a click handler), which is why `rememberLauncherForActivityResult` is a composable call rather than a plain function call.

### Rationale UI

The Android system can signal when you should explain *why* you need a permission before asking: `ActivityCompat.shouldShowRequestPermissionRationale(activity, permission)` returns `true` after the user has denied once without checking "Don't ask again". At that point, showing a brief in-app explanation before re-requesting meaningfully improves grant rates.

The typical decision tree for a single permission:

1. Already granted → proceed directly.
2. `shouldShowRequestPermissionRationale` is `true` → show rationale text, then surface a "Request" button that launches the system dialog.
3. Neither condition → launch the system dialog immediately.
4. Denied with "Don't ask again" → the launcher returns `false` and `shouldShowRequestPermissionRationale` is also `false`. You can no longer show the system dialog; the only recourse is directing the user to app settings via an `Intent` to `Settings.ACTION_APPLICATION_DETAILS_SETTINGS`.

In Compose, this is naturally expressed as a sealed class or enum in `UiState` — `PermissionState.Granted`, `Denied`, `NeedsRationale`, `PermanentlyDenied` — rather than ad-hoc `Boolean` flags in the composable body.

### Android 13 and `POST_NOTIFICATIONS`

Android 13 (API 33) introduced `POST_NOTIFICATIONS` as a runtime dangerous permission. Apps targeting API 33 or higher must request it before any notification is visible to the user. Calling `NotificationManagerCompat.notify()` without the grant silently drops the notification — there is no exception, no crash, no Logcat warning at the call site.

Declare it in the manifest:

```xml
<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />
```

Then request it at runtime using `Manifest.permission.POST_NOTIFICATIONS` (requires `compileSdk 33+`; use the string literal `"android.permission.POST_NOTIFICATIONS"` if your module compiles against an older SDK).

Apps targeting API 32 or lower receive a one-time automatic grant on first launch after upgrading the OS to Android 13 — existing users are not disrupted. New installs targeting 33+ must always go through the runtime request.

> Accompanist Permissions (`com.google.accompanist:accompanist-permissions`) provided a `rememberPermissionState` Compose wrapper around the same Activity Result API. It predates the stabilisation of the platform API surface and is now largely superseded — the raw platform APIs are direct enough that the wrapper adds little value for new projects.

### What changed since 2021

- **`onRequestPermissionsResult` is superseded.** The Activity Result APIs landed in AndroidX in 2020 and are now the standard. The old callback still compiles but is wrong for Compose — results would need to be routed from the Activity into the composable via side channels.
- **`POST_NOTIFICATIONS` is new (Android 13, API 33).** It did not exist in 2021. Any code that posts notifications without requesting it fails silently on Android 13+ devices — the notification just never appears.
- **`shouldShowRequestPermissionRationale` semantics are unchanged**, but surfacing rationale as composable `UiState` rather than an imperative dialog call in the Activity is the right pattern now.

## Drill

Work in the capstone or a scratch module. Do not use an `onRequestPermissionsResult` override — use the Activity Result contract throughout.

**Part A — single permission request**

Add a screen (or a section of an existing screen) that reads the current `POST_NOTIFICATIONS` grant status on composition. If not granted, show a button labelled "Enable notifications". Tapping it launches `RequestPermission` for `POST_NOTIFICATIONS`. The UI updates to reflect the new state immediately after the user responds.

**Part B — rationale**

Before launching the system dialog, check `shouldShowRequestPermissionRationale`. If it returns `true`, show a brief explanation text above the button ("We need notification permission to send you habit reminders" — paraphrase freely). On first launch, before any denial, skip the explanation and go straight to the dialog. After "Don't ask again", replace the button with text that invites the user to enable the permission in system settings.

> [!example] Success criteria
> - [ ] Permission state (granted / denied / needs-rationale / permanently denied) is represented in ViewModel `UiState`, not as ad-hoc local `Boolean` variables in the composable.
> - [ ] Granting the permission updates the UI immediately without a relaunch or navigation.
> - [ ] Denying the permission does not crash the app — a "denied" state is displayed with appropriate messaging.
> - [ ] After one denial, the rationale text is shown before the system dialog appears on the next request attempt.
> - [ ] After "Don't ask again" (second denial), the UI shows a prompt to open system settings rather than re-launching the system dialog.
> - [ ] No `onRequestPermissionsResult` override anywhere in the project.
