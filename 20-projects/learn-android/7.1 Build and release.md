# 7.1 Build and release

> [!abstract] TL;DR
> Producing a release `.aab` involves three interacting systems: build variants and signing config, R8 shrinking with keep rules, and the app bundle format that unlocks Play App Signing. This chapter walks each step from a Kotlin DSL `build.gradle.kts` to an upload-ready artifact.

## Digest

### Version catalogs and Kotlin DSL

In 2021 you likely had a root `build.gradle` with an `ext {}` block scattering version strings, and per-module Groovy files pulling `ext.kotlin_version` and friends. That still compiles, but there is no IDE autocomplete and no compile-time checking — a typo in a version string fails silently until build time or, worse, at runtime.

New projects generate **Kotlin DSL** (`build.gradle.kts`) and a **version catalog** (`gradle/libs.versions.toml`). The catalog separates concerns: one place to declare versions, another for library coordinates, another for plugin IDs. Typed accessors — `libs.compose.ui`, `libs.room.ktx` — give autocomplete and catch renames at sync time:

```toml
# gradle/libs.versions.toml
[versions]
room = "2.7.1"

[libraries]
room-runtime = { group = "androidx.room", name = "room-runtime", version.ref = "room" }
room-ktx     = { group = "androidx.room", name = "room-ktx",     version.ref = "room" }
```

Bumping the single `room` version entry updates every consumer across all modules. No more grep-and-replace across multiple `build.gradle` files. See [[0.1 Modern Android landscape]] for the broader context on why version catalogs became the default.

### Build types and signing config

Debug and release build types have always been the core split. At release time the key additions are `isMinifyEnabled`, `isShrinkResources`, and a **signing config** pointing to your release keystore — without one, the `.aab` cannot be submitted to Play.

```kotlin
// app/build.gradle.kts
android {
    signingConfigs {
        create("release") {
            storeFile     = file(System.getenv("KEYSTORE_PATH") ?: "release.keystore")
            storePassword = System.getenv("KEYSTORE_PASS") ?: ""
            keyAlias      = System.getenv("KEY_ALIAS") ?: ""
            keyPassword   = System.getenv("KEY_PASS") ?: ""
        }
    }
    buildTypes {
        release {
            signingConfig     = signingConfigs.getByName("release")
            isMinifyEnabled   = true
            isShrinkResources = true
            proguardFiles(
                getDefaultProguardFile("proguard-android-optimize.txt"),
                "proguard-rules.pro"
            )
        }
    }
}
```

The golden rule: **never commit keystore credentials to VCS.** Use environment variables (as above) or a local `keystore.properties` file that is listed in `.gitignore`. CI reads them from secrets; local developers set them in their shell environment or a local properties file that is never staged.

### R8: shrinking, obfuscation, and keep rules

R8 is the default shrinker and optimizer, replacing standalone ProGuard. It runs automatically as part of the compile step when `isMinifyEnabled = true` — there is nothing extra to install or invoke. R8 removes unused classes and methods, renames identifiers to short symbols, and performs additional optimizations (constant folding, inlining, dead-code elimination) in a single pass.

The risk is the same as it always was: **reflection-based and serialization-based code breaks under aggressive shrinking.** R8 cannot statically see which classes are accessed via reflection, so it removes them — and the app crashes at runtime with `ClassNotFoundException` or a silent Gson deserialization failure.

Keep rules in `proguard-rules.pro` prevent this:

```
# Keep data classes used by Gson / Moshi / Retrofit
-keep class com.example.habittracker.data.model.** { *; }

# Keep Parcelable implementations
-keep class * implements android.os.Parcelable {
    public static final ** CREATOR;
}
```

The practical workflow: build release → do a quick smoke test on the minified build → add keep rules for any crash or silent corruption → repeat until clean. This is much faster done locally before submission than debugged from a production crash report.

### App bundle vs APK and Play App Signing

In 2021 you produced a **signed APK** and uploaded it to Play. That path still exists for side-loading but is no longer the Play distribution path. **App bundles (`.aab`)** are required for new apps on Google Play.

The difference matters for signing. When you upload an APK, Play distributes it signed with your key. When you upload an `.aab`, the signing model has two layers:

1. You sign the `.aab` with your **upload key** — a keystore you own and keep locally.
2. **Play App Signing** holds the actual **app signing key** — the key used to sign the delivery APKs sent to devices. Google manages this key; you do not have it on disk.

Play re-signs delivery artifacts after optimizing them per device (screen density, ABI, language). The practical benefit: if your upload key is lost or compromised, you can request a new one from the Play Console without losing your app. The app signing key is stored by Google and cannot be lost.

Generate the bundle via Android Studio's "Generate Signed Bundle / APK" wizard or directly from Gradle:

```
./gradlew bundleRelease
```

The output lands at `app/build/outputs/bundle/release/app-release.aab`.

### What changed since 2021

- **Build files:** Kotlin DSL (`build.gradle.kts`) is the default for new projects; Groovy still works. The practical gain is IDE autocomplete and compile-time checking on build config.
- **Version catalogs:** `gradle/libs.versions.toml` replaced scattered `ext {}` version blocks. Typed accessors (`libs.room.ktx`) are checked at sync time, not at runtime.
- **Shrinking:** R8 is the shrinker — no separate ProGuard invocation. It runs when `isMinifyEnabled = true`. Keep rules in `proguard-rules.pro` remain the fix for reflection/serialization crashes.
- **Distribution format:** `.aab` is required for new apps on Play; APK upload is for side-loading only. Play App Signing separates your upload key (which you manage) from the app signing key (which Google manages).

## Drill

Your capstone habit-tracker app is built and working. Prepare it for submission.

1. **Signing**: generate a release keystore using `keytool` or Android Studio's wizard. Add a `signingConfigs.release` block to `app/build.gradle.kts` that reads credentials from environment variables or a gitignored `keystore.properties` file — not hardcoded strings. Confirm the keystore file itself is also listed in `.gitignore`.

2. **R8**: in your `buildTypes.release` block, set `isMinifyEnabled = true` and `isShrinkResources = true`. Run `./gradlew bundleRelease`. Install the resulting bundle on a device or emulator (use bundletool or export a universal APK from the Studio wizard) and do a quick smoke test: launch, create a habit, navigate, trigger the reminder flow. If you get `ClassNotFoundException`, add keep rules to `proguard-rules.pro` and rebuild.

3. **Verify output**: confirm the `.aab` file exists at `app/build/outputs/bundle/release/app-release.aab` and that the build was signed with the release signing config (not the debug key).

> [!example] Success criteria
> - [ ] A release keystore exists; its credentials are not committed to VCS (confirmed by `git status` showing nothing new staged).
> - [ ] `build.gradle.kts` reads signing credentials from environment variables or a gitignored properties file.
> - [ ] `isMinifyEnabled = true` and `isShrinkResources = true` are set in the release build type.
> - [ ] `./gradlew bundleRelease` completes without error.
> - [ ] The minified release build runs on a device or emulator with no `ClassNotFoundException` crashes.
> - [ ] `app-release.aab` exists on disk and is signed with the release config.
