# 1.3 Classes the Kotlin way

> [!abstract] TL;DR
> `data class`, `sealed interface`, and scope functions are the core vocabulary of Kotlin domain modeling; mastering them now pays off immediately in ViewModel UI state, repository layers, and every `when` branch you write in Compose.

## Digest

### data class

`data class` generates `equals`, `hashCode`, `toString`, `copy`, and component functions automatically:

```kotlin
data class Habit(val id: Long, val name: String, val streak: Int = 0)

val morning = Habit(1L, "Morning run")
val updated = morning.copy(streak = morning.streak + 1)
val (id, name, _) = updated   // destructuring via component functions
```

`copy` returns a new instance with only the named fields changed — the primary way to produce updated immutable state.

### sealed class / sealed interface

A `sealed` hierarchy is closed: all direct subclasses must live in the same package. The compiler then verifies that a `when` over a sealed type is exhaustive — no `else` branch needed, and adding a new subtype causes compile errors everywhere it was not handled.

```kotlin
sealed interface UiState<out T> {
    data object Loading : UiState<Nothing>
    data class Success<T>(val data: T) : UiState<T>
    data class Error(val message: String) : UiState<Nothing>
}
```

`sealed interface` is preferred over `sealed class` when you don't need common state or behavior at the parent level — it's lighter and can be combined with other interfaces. `data object` (Kotlin 1.9+) is the modern way to express a singleton variant in a sealed hierarchy; it gives `toString` and `equals` without boilerplate, replacing `object` for this purpose.

### object and companion object

`object` is a top-level singleton — declared once, instantiated automatically:

```kotlin
object HabitValidator {
    fun isValid(name: String) = name.isNotBlank() && name.length <= 100
}
```

`companion object` lives inside a class and provides "static" members:

```kotlin
class Habit private constructor(val id: Long, val name: String) {
    companion object {
        fun create(name: String): Habit = Habit(generateId(), name)
    }
}
```

### Extension functions

You can add functions to any type without touching its source code:

```kotlin
fun String.toSlug(): String = trim().lowercase().replace(Regex("\\s+"), "-")
fun Habit.isOnStreak(): Boolean = streak >= 7
```

Extensions are resolved statically (not virtual), but they read like instance methods. The Kotlin standard library is mostly implemented as extension functions on basic types.

### Scope functions

These five look similar but differ in receiver and return value — pick by intent:

| Function | Receiver in block | Returns       | Typical use |
|----------|-------------------|---------------|-------------|
| `let`    | `it`              | block result  | Null-safe transform; one-off rename |
| `run`    | `this`            | block result  | Initialise + compute result |
| `with`   | `this` (argument) | block result  | Configure an object you pass in |
| `apply`  | `this`            | the receiver  | Builder / object initialisation |
| `also`   | `it`              | the receiver  | Side-effects (logging) without breaking the chain |

```kotlin
val intent = Intent(context, DetailActivity::class.java).apply {
    putExtra("id", habit.id)
    addFlags(Intent.FLAG_ACTIVITY_CLEAR_TOP)
}

habit?.let { log("Processing ${it.name}") }
```

### Interface delegation with `by`

`by` generates the boilerplate of forwarding every interface method to a delegate, so you only override what you want to change:

```kotlin
interface Logger { fun log(msg: String) }

class ConsoleLogger : Logger {
    override fun log(msg: String) = println(msg)
}

class TimestampLogger(private val inner: Logger) : Logger by inner {
    override fun log(msg: String) = inner.log("[${System.currentTimeMillis()}] $msg")
}
```

## Drill

Work in **Kotlin Playground** or a scratch `.kt` file.

Start from these type signatures — implement only the bodies; do not change the signatures:

```kotlin
data class Note(
    val id: Long,
    val title: String,
    val body: String,
    val isPinned: Boolean = false
)

sealed interface AsyncResult<out T> {
    data object Loading : AsyncResult<Nothing>
    data class Success<T>(val data: T) : AsyncResult<T>
    data class Failure(val reason: String, val retryable: Boolean) : AsyncResult<Nothing>
}
```

**Tasks:**

1. Write `fun <T> AsyncResult<T>.display(): String` as an **extension function** that returns:
   - `"Loading…"` for `Loading`
   - `"OK: $data"` for `Success`
   - `"Error ($reason)"` for `Failure`
   Use an exhaustive `when` — no `else` branch.

2. Write `fun summarize(result: AsyncResult<List<Note>>): String` that:
   - For `Success`, returns `"${data.size} notes, ${data.count { it.isPinned }} pinned"`.
   - For `Failure` with `retryable == true`, returns `"Failed — will retry: $reason"`.
   - For `Failure` with `retryable == false`, returns `"Failed permanently: $reason"`.
   - For `Loading`, returns `"Fetching…"`.
   Again, no `else` branch.

3. Write `fun Note.toPreview(maxLength: Int = 80): Note` that returns a `copy` of the note with `body` truncated to `maxLength` characters, appending `"…"` if truncated. Use `copy` and at least one scope function.

4. Write a `main()` that calls `display()`, `summarize()`, and `toPreview()` with at least one `AsyncResult` of each variant, and prints all results.

> [!example] Success criteria
> - [ ] `AsyncResult` has exactly three subtypes; `when` in `display()` and `summarize()` compile without an `else` branch.
> - [ ] Adding a fourth subtype to `AsyncResult` causes a compile error in both `when` expressions.
> - [ ] `toPreview()` uses `copy` and at least one scope function (`apply`, `let`, `run`, `also`, or `with`).
> - [ ] All functions produce the expected output in `main()`.
> - [ ] No `!!` anywhere.
