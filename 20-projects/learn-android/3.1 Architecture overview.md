# 3.1 Architecture overview

> [!abstract] TL;DR
> Google's recommended architecture stacks three layers — UI, domain (optional), and data — connected by strict unidirectional data flow; each boundary is a plain interface or data class, making layers independently testable and swappable.

## Digest

### The three-layer model

Google's [app architecture guide](https://developer.android.com/topic/architecture) defines three layers with explicit responsibilities:

```
┌────────────────────────────────────┐
│            UI layer                │
│  Composables + ViewModels          │
│  Renders state; sends events up    │
└──────────────┬─────────────────────┘
               │ (optional layer)
┌──────────────▼─────────────────────┐
│          Domain layer              │
│  Use cases — reusable business     │
│  logic shared across ViewModels    │
└──────────────┬─────────────────────┘
               │
┌──────────────▼─────────────────────┐
│           Data layer               │
│  Repositories + data sources       │
│  (Room, Retrofit, DataStore…)      │
└────────────────────────────────────┘
```

**UI layer** contains the composables that render what the user sees and the ViewModels that produce the state those composables render. The ViewModel is the boundary: it survives configuration changes, knows nothing about the UI toolkit, and depends only on the layer below.

**Domain layer** is optional. Add it when business logic is complex enough to be reused across multiple ViewModels, or when a single ViewModel grows unwieldy. The unit of organization here is a *use case* — a single-method class like `AddHabitUseCase` or `GetActiveHabitsUseCase`. Skip the domain layer in apps where logic fits neatly in the ViewModel.

**Data layer** contains repositories and their underlying data sources (a Room DAO, a Retrofit service, a DataStore). Repositories are the **single source of truth**: callers upstream never touch a DAO or API service directly. A repository's job is to coordinate sources and expose a clean interface to the layers above.

### Unidirectional data flow

Data flows in one direction only:

```
user action
    │
    ▼
ViewModel receives event → updates internal state
    │
    ▼  StateFlow<UiState>
Composable re-renders
    │
    └──► user action (cycle repeats)
```

Events travel upward (composable → ViewModel → repository); state travels downward (repository → ViewModel → composable). The UI never writes to ViewModel state directly; the ViewModel never calls a composable. Breaking this contract in either direction creates hidden couplings that are difficult to test.

### Separation of concerns

Each layer depends only on the layer below it — never above:

- **Composables** know only `UiState` — a plain `data class` with no Android framework types. They import nothing from the domain or data packages.
- **ViewModels** depend on repository interfaces, not on Room DAOs, Retrofit services, or Android `Context`. This makes them testable with plain JVM unit tests.
- **Repositories** coordinate data sources and expose results as `Flow` streams. They know nothing about ViewModels or composables.

Because each boundary is a data class or interface, swapping an implementation — in-memory fake ↔ real Room database — requires changing only the binding in one place (the Hilt module from chapter 3.3), not the ViewModel or composables.

### What changed since 2021

- **The layered model is now formal, documented guidance.** The architecture guide was substantially rewritten around 2022; it now names layers, assigns responsibilities, and explicitly flags anti-patterns (e.g., "don't access data sources from the UI layer directly").
- **ViewModel is unambiguously the state holder.** Earlier guidance entertained Fragments and Activities as legitimate state holders. The current guide does not — ViewModel is the answer.
- **Fragments are not the primary abstraction for new Compose screens.** Fragment-to-Fragment navigation wired through XML nav graphs is replaced by Navigation Compose. The three-layer model itself is unchanged; what changed is what sits inside each layer.
- **Use cases are a named, first-class concept** rather than an informal "helper class" pattern you might have invented on your own.

## Drill

Open a blank Markdown file (or paper). This is a design exercise — write no code.

**Part A — layer diagram**

Draw boxes for the three layers of the habit tracker. Inside each box, name the specific class (or composable) that will participate in the "add habit" feature. For each class, add a one-phrase description of what it owns.

**Part B — event and state trace**

Trace a single action — "user types a habit name and taps Save" — through your diagram. For each arrow between boxes, write one sentence: what crosses that boundary, and in which direction?

**Part C — responsibility audit**

Identify three responsibilities that would violate the architecture if placed in the ViewModel: one that belongs in the UI layer, one in the data layer, and one that is simply the wrong abstraction level. For each, name the concrete harm it would cause (e.g., testability, coupling, reuse).

> [!example] Success criteria
> - [ ] Part A: all three layers have at least one named, concrete class each.
> - [ ] Part A: the data source (in-memory list or Room DAO) appears only in the data layer box.
> - [ ] Part B: all arrows are unidirectional — no upward data dependencies.
> - [ ] Part B: `UiState` travels from ViewModel to composable, never the reverse.
> - [ ] Part C: each misplaced responsibility is paired with a concrete harm.
> - [ ] No code written — diagram and prose only.
