# 7.2 Play Store launch

> [!abstract] TL;DR
> Submitting to Play is not just uploading a bundle — it involves a store listing, a mandatory Data safety declaration, content rating, and a choice of release track. This chapter covers each checkpoint in order, from first console setup to post-launch monitoring.

## Digest

### Play Console basics

The Play Console (play.google.com/console) is where you manage your app's presence: create the app shell, manage releases across tracks, maintain the store listing, handle policy compliance, and watch post-launch health metrics. If you haven't used it since 2021, the dashboard has been redesigned — sidebar-based navigation, and policy gates (Data safety, content rating) are now hard blockers, not advisory items.

First step: create the app. Choose default language, declare app or game, free or paid. This creates the shell; nothing is visible to users yet. Most of the remaining work — store listing, Data safety, content rating — can be done before uploading a single `.aab`.

### Store listing assets

Before any track review, the store listing must be complete. Required assets:

- **App name** — up to 30 characters.
- **Short description** — up to 80 characters; appears on the Play Store app card in search results.
- **Full description** — up to 4,000 characters; the full listing page.
- **Screenshots** — at least 2 phone screenshots (each side between 320 px and 3,840 px; aspect ratio between 9:21 and 21:9 — portrait and landscape are both fine). Tablet and foldable screenshots are optional but improve discoverability on those form factors.
- **Feature graphic** — 1,024 × 500 px banner; required for any editorial featuring opportunity.
- **App icon** — 512 × 512 px; this is the Play Store icon, separate from the launcher icon inside the `.aab`.

Store listing quality feeds into ranking signals. A missing feature graphic makes the app ineligible for editorial placements entirely.

### Data safety form

This is new since 2021 and is **mandatory** — every app must complete the Data safety section as part of its store presence, and it is a hard gate before Closed testing, Open testing, and Production review or rollout. (Internal testing has no Google review step, so the form does not block it, but completing it early is good practice.) The form declares what data the app collects, how it is used, whether it is shared with third parties, and whether users can request deletion.

For an offline habit tracker with no analytics SDK, no crash reporter, no backend, and no network calls — the answers are minimal:

- **Data collected**: none that leaves the device. On-device storage (Room, DataStore) does not count as data collection under Play's definition.
- **Data shared with third parties**: none.
- **Security practices**: "Data is encrypted in transit" applies only if you have network calls — for an offline app, this is not applicable. "Users can request that data be deleted" applies if you provide an account deletion or data wipe flow.

Completing this accurately matters. Google audits declarations against app behavior and can remove apps where the form contradicts reality. If you later add analytics or a backend, you must update the form before shipping the update.

### Content rating questionnaire

Also mandatory before production. The Play Console questionnaire asks about violence, sexual content, gambling, controlled substances, user-generated content, and social features. For a habit tracker the answers are uniformly low or none, which yields an "Everyone" rating in most regions. The questionnaire takes around five minutes; completing it is a hard gate on publishing.

### Release tracks and staged rollout

Play offers four release tracks, intended to form a progression from narrow to wide:

| Track | Audience |
|---|---|
| **Internal testing** | Up to 100 testers invited by email. No review step — available to invited testers within minutes. Best for early smoke tests. |
| **Closed testing (alpha)** | Larger invited groups. Still fast review. |
| **Open testing (beta)** | Any user who opts in via a public link. Review-gated. |
| **Production** | All users in your target countries. |

For a first launch, the typical path is **internal testing → production** — alpha and beta are useful for gathering structured pre-release feedback but add time. Each promotion from one track to the next requires its own review cycle.

**Staged rollout** releases to a percentage of production users — for example, start at 10%, watch Play vitals for 24–48 hours, expand to 50%, then 100%. The Console lets you halt mid-rollout if metrics degrade. This is standard practice for significant updates to an existing user base. For an initial release (no existing users), going to 100% immediately is common since there is no installed base to protect.

### Play vitals and post-launch health

Once the app is live, the **Android vitals** dashboard surfaces health signals that Google uses in ranking:

- **ANR rate** — "Application Not Responding" dialogs, triggered when the main thread is blocked for more than 5 seconds. Google's bad-behavior threshold is roughly 0.47% of daily active users experiencing an ANR.
- **Crash rate** — unhandled exceptions that terminate the process. Threshold: roughly 1.09% of daily active users.
- **Excessive wakeups** — background alarms or wake locks firing more than 10 times per hour per device.

Breaching these thresholds can suppress your app's Play Store ranking or trigger a policy notification. For a habit tracker the main risk is the daily reminder: if the `AlarmManager` or `WorkManager` job fires excessively or blocks the main thread, it will surface here. Check vitals a day after each release before widening a staged rollout.

### Target API level requirements

Google enforces a minimum `targetSdk` for all submissions. As of mid-2026, both new apps and updates to existing apps must target **API 35** (Android 15). Google's cadence: the required target API level steps up roughly every year (around 31 August) to the prior year's Android release — check the Play Console target-API-level policy page for the exact current requirement, as it changes annually.

Targeting a high API level is not just compliance — it opts the app into behavioral changes: edge-to-edge enforcement from API 35, stricter background limits, predictive back gesture handling. Set `targetSdk` in `app/build.gradle.kts`, test on a matching emulator, and confirm the app behaves correctly under those constraints before submitting.

### What changed since 2021

- **App bundles required:** New apps must submit `.aab`. The Play Console will reject an APK upload for a new app.
- **Data safety form:** New and mandatory since 2022. Did not exist in the 2021 Play Console. Required as part of the app's store presence; a hard gate before Closed testing, Open testing, and Production review or rollout. (Internal testing has no review step and is not blocked by the form.)
- **Content rating:** The questionnaire existed before but is now a hard gate — you cannot progress to any release track without completing it.
- **Target API level enforcement:** Annual deadline-based minimums that step up each year. In 2021 there was no enforcement; now updates to existing apps can be blocked from reaching new users if `targetSdk` is below the minimum.
- **Play vitals enforcement:** Bad-behavior thresholds now actively suppress ranking. The signals existed in 2021 but enforcement was lighter.
- **Staged rollout UX:** Available for years, but now the explicitly recommended workflow for significant updates; the Console surfaces a halt button prominently when vitals degrade.

## Drill

Your signed `.aab` from [[7.1 Build and release]] is ready. Before uploading, prepare the submission package in writing.

1. **Store listing draft**: write a complete store listing for the habit tracker — app name (≤ 30 chars), short description (≤ 80 chars), and a full description (≤ 4,000 chars, written out in full). List the screenshots and feature graphic dimensions you would prepare.

2. **Data safety questionnaire**: answer each section for an offline habit tracker with no analytics, no backend, and no crash-reporting SDK:
   - Data collected: what, if anything?
   - Data shared with third parties: what?
   - Security practices: which of the listed practices apply?
   - Is user-requested data deletion available?

3. **Rollout plan**: write a short staged rollout plan for releasing version 1.0 to production. Include: which track you would use first; your initial rollout percentage (or why you'd go to 100%); at least two Play vitals signals you'd watch before expanding; and your criterion for halting the rollout.

> [!example] Success criteria
> - [ ] Store listing is drafted with app name (≤ 30 chars), short description (≤ 80 chars), and full description written out.
> - [ ] Required asset dimensions are listed correctly (screenshots, feature graphic, high-res icon).
> - [ ] Data safety answers cover all four sections; answers are consistent with an offline tracker that has no third-party SDKs and no network calls.
> - [ ] Rollout plan names a starting track, an initial percentage (or justification for 100%), at least two Play vitals signals to monitor, and an explicit halt criterion.

## Capstone step

With the habit tracker built, tested, and accessible (Phases 1–6 complete), this is the final milestone: **a submission-ready release package**.

**Feature**: produce a signed `.aab` built from the release variant, with R8 minification enabled, ready to upload to the Play Console.

> [!example] Capstone acceptance criteria
> - [ ] The `.aab` is signed with the upload key — not the debug key. Confirm by checking the signing config used in the build, or by inspecting the certificate via Android Studio's **Build > Analyze APK…** (works for `.aab` too; shows the signing certificate and fingerprint). Alternatively, extract an APK set with `bundletool` and run `apksigner verify --verbose` on an extracted APK. (Avoid `jarsigner -verify` on `.aab` files — it only checks v1/JAR signatures and reports "jar is unsigned" for modern v2/v3-signed bundles even when signing is correct.)
> - [ ] The release build runs cleanly on a device or emulator — no crash on cold launch, habit creation, navigation, or reminder trigger.
> - [ ] `targetSdk` in `app/build.gradle.kts` is set to API 35 (the 2026 Play Store minimum for new apps).
> - [ ] The store listing draft from the Drill is written and ready (in notes, or entered in the Play Console if you have a developer account).
> - [ ] The Data safety questionnaire answers are written and truthfully reflect the app's behavior.
