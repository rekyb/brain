# 5.3 Notifications

> [!abstract] TL;DR
> Every notification must belong to a channel (mandatory since Android 8) and on Android 13+ the app must hold the `POST_NOTIFICATIONS` runtime permission before any notification appears. Use `NotificationCompat` to build backwards-compatible notifications and a `PendingIntent` to deep-link the user into the right screen on tap.

## Digest

### Notification channels

Since Android 8 (API 26), every notification must be assigned to a **channel**. Users control sound, importance, and badge behaviour at the channel level. If you post a notification with a channel ID that doesn't exist in the `NotificationManager`, the notification is silently dropped â€” no crash, no Logcat error at the call site.

Create channels once, early in the app lifecycle. `Application.onCreate` is a reliable place:

```kotlin
val channel = NotificationChannel(
    "habit_reminders",                          // channel ID â€” stable, never change
    "Habit Reminders",                          // user-visible name
    NotificationManager.IMPORTANCE_DEFAULT
).apply {
    description = "Daily reminders to complete your habits"
}
NotificationManagerCompat.from(context).createNotificationChannel(channel)
```

Creating a channel that already exists is a no-op â€” safe to call on every launch. Deleting and re-creating a channel resets user-configured settings, so avoid it.

### Building and posting

`NotificationCompat.Builder` (from `androidx.core`) is the backwards-compatible way to construct notifications. The minimum meaningful notification specifies a small icon, a title, and the channel ID:

```kotlin
val notification = NotificationCompat.Builder(context, "habit_reminders")
    .setSmallIcon(R.drawable.ic_habit)
    .setContentTitle("Time to track")
    .setContentText("Don't forget your morning habits")
    .setPriority(NotificationCompat.PRIORITY_DEFAULT)   // for pre-26 devices
    .setAutoCancel(true)                                // dismiss on tap
    .setContentIntent(pendingIntent)
    .build()

NotificationManagerCompat.from(context)
    .notify(NOTIFICATION_ID, notification)
```

Use distinct integer IDs per notification type. Reusing an ID updates the existing notification rather than posting a new one, which is correct for reminders.

### Android 13+ `POST_NOTIFICATIONS` permission gate

From Android 13 (API 33), posting any notification requires the user to grant `POST_NOTIFICATIONS` at runtime. Without the grant, `notify()` silently does nothing â€” no exception, no Logcat warning at the call site, no visible notification. This is the same class of silent failure as posting to a non-existent channel, but for a different reason and on a per-app basis.

The full flow:

1. Declare in the manifest: `<uses-permission android:name="android.permission.POST_NOTIFICATIONS" />`.
2. Before posting, check the grant with `ContextCompat.checkSelfPermission`.
3. If not granted, request via `rememberLauncherForActivityResult(ActivityResultContracts.RequestPermission())` â€” see [[5.1 Runtime permissions]] for the full rationale-and-denial flow.
4. Only call `notify()` after confirming the grant.

Apps targeting API 32 or lower receive a temporary automatic grant on first launch after upgrading to Android 13 â€” existing installs are not disrupted. New installs targeting API 33+ must always go through the runtime request.

### Deep-linking with PendingIntent

A tap on a notification should navigate the user to a specific in-app destination, not just open the launcher Activity. Construct a `PendingIntent` that encodes the destination:

```kotlin
val intent = Intent(context, MainActivity::class.java).apply {
    action = "OPEN_HABIT_DETAIL"
    putExtra("habitId", habitId)
    flags = Intent.FLAG_ACTIVITY_SINGLE_TOP or Intent.FLAG_ACTIVITY_CLEAR_TOP
}
val pendingIntent = PendingIntent.getActivity(
    context,
    requestCode,
    intent,
    PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE   // IMMUTABLE required on API 31+
)
```

`FLAG_IMMUTABLE` is required from Android 12 (API 31) â€” omitting it causes a crash on modern devices. In `MainActivity`, inspect the incoming intent and navigate to the appropriate Compose destination. For more complex back-stack construction (so Back returns to a meaningful screen, not the empty launcher), use `TaskStackBuilder`:

```kotlin
val stackBuilder = TaskStackBuilder.create(context).apply {
    addNextIntentWithParentStack(intent)
}
val pendingIntent = stackBuilder.getPendingIntent(
    requestCode,
    PendingIntent.FLAG_UPDATE_CURRENT or PendingIntent.FLAG_IMMUTABLE
)
```

### Scheduling: WorkManager vs AlarmManager

For most product reminders â€” daily habit nudges, weekly summaries â€” **WorkManager** is the right scheduler. It survives process death and device reboots, respects battery restrictions automatically, and integrates naturally with the rest of the Android background-work model. Post the notification inside `CoroutineWorker.doWork()`.

Use **AlarmManager** when exact-time delivery is a hard product requirement â€” a countdown timer, a calendar event at a precise moment. From Android 12 (API 31), exact alarms require `SCHEDULE_EXACT_ALARM` (API 31+, user-grantable and revocable); from Android 13 (API 33), `USE_EXACT_ALARM` is additionally available but is restricted to apps whose primary purpose is alarms or calendar events. Acquiring either permission adds Play Store review overhead. For habit reminders, the right WorkManager pattern is a **`OneTimeWorkRequest`** with an `initialDelay` computed to the next occurrence of the habit's `reminderTime`. When the worker fires it posts the notification and re-enqueues itself for the next day using `WorkManager.enqueueUniqueWork` keyed on the habit's id â€” so each habit's reminder chain is independently cancellable. The approximate, battery-friendly delivery of WorkManager is exactly what the OS expects for non-urgent daily nudges; exact-to-the-minute delivery requires AlarmManager with the permissions described above.

### What changed since 2021

- **Channels existed in 2021, but `POST_NOTIFICATIONS` is new (Android 13).** Any code written before 2023 that posts notifications without first requesting the runtime permission will silently fail on Android 13+ devices â€” the notification never appears.
- **`PendingIntent.FLAG_IMMUTABLE` is now required** on API 31+ (Android 12). In 2021 it was a best-practice recommendation; today omitting it throws an `IllegalArgumentException` at `PendingIntent.getActivity()`.
- **Exact alarms are significantly restricted.** Android 12â€“14 progressively tightened `setExact` and `setAlarmClock` behind permissions that require user action and Play Store justification. The practical upshot: WorkManager is the right default unless the product explicitly requires clock-precise delivery.

## Drill

Work in the capstone or a scratch module.

**Part A â€” channel and notification**

Create a notification channel during app startup. Add a button to any screen that builds and posts a notification on that channel. On a device or emulator running API 33+, ensure you have requested `POST_NOTIFICATIONS` first (reuse the request flow from [[5.1 Runtime permissions]] or add a minimal inline check). Confirm the notification appears in the shade.

**Part B â€” deep link**

Attach a `PendingIntent` to the notification that navigates to a specific screen when the notification is tapped â€” not just the launcher. Test from both a running app (notification appears while app is foregrounded, tap it) and a cold-start (dismiss the app from recents, tap the notification from the shade). Verify the destination screen is correct in both cases.

> [!example] Success criteria
> - [ ] The channel is created before the first `notify()` call â€” no silent drop due to a missing channel.
> - [ ] On API 33+ (device or emulator), `POST_NOTIFICATIONS` is requested before `notify()` is attempted; denying it suppresses the notification without a crash.
> - [ ] Tapping the notification opens the correct in-app destination screen, not just the app's launcher activity.
> - [ ] The `PendingIntent` is built with `FLAG_IMMUTABLE`.
> - [ ] No exact `AlarmManager` alarm is used for this drill.

## Capstone step

**Habit reminders via WorkManager and notifications**

Add scheduled habit reminders to the capstone tracker.

**Feature:** Each habit can have a daily reminder at its `reminderTime`. From the habit detail (or list) screen the user can set or clear a per-habit reminder time. When a reminder is set, a `OneTimeWorkRequest` is enqueued under a unique name keyed on the habit's id (e.g. `"reminder_${habit.id}"`); when it fires it posts a notification and re-enqueues itself for the next day. When a reminder is cleared or the habit is deleted, the unique work for that habit is cancelled by name.

**Android 13+ permission gate:** Before any reminder can be scheduled, the app checks for `POST_NOTIFICATIONS`. If not granted, the app requests the permission using the Activity Result API. If the user denies the permission, the reminder UI is disabled or a message explains that notifications are blocked and offers a path to system settings.

> [!example] Capstone acceptance criteria
> - [ ] Setting a reminder for a habit enqueues a unique `OneTimeWorkRequest` keyed on the habit's id; the notification fires near the chosen `reminderTime`.
> - [ ] Clearing or disabling a habit's reminder cancels that habit's unique work by name â€” other habits' reminders are unaffected.
> - [ ] On API 33+ (emulator), `POST_NOTIFICATIONS` is requested before the first notification is posted; the permission request uses `rememberLauncherForActivityResult`, not `onRequestPermissionsResult`.
> - [ ] Denying `POST_NOTIFICATIONS` disables the reminder UI or surfaces a "go to settings" message â€” no crash.
> - [ ] The notification appears in the shade and tapping it opens the app at the correct screen.
> - [ ] After a device reboot, reminders resume â€” WorkManager re-registers persisted work automatically.
