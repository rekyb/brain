# 4.2 DataStore

> [!abstract] TL;DR
> Jetpack DataStore is the coroutine/Flow-native replacement for `SharedPreferences`: reads come as `Flow`, writes are `suspend` calls inside an atomic `edit` block, and there is no synchronous API to misuse.

## Digest

### Why DataStore replaces SharedPreferences

`SharedPreferences` had two persistent problems. `apply()` defers the disk write to a background thread but still blocks the main thread during `onStop()` / `onPause()` to flush queued writes â€” a source of real-world ANRs on slow devices. And all the `getX()` calls are synchronous: you either read preferences before the first frame (blocking startup) or race against the UI initialising without them.

DataStore eliminates both. Reads are `Flow<T>` â€” you collect them just like any other state source, and the composable is notified when the value changes. Writes are `suspend` functions inside an atomic `edit` block, routed to a dedicated IO dispatcher and fully flushed before the coroutine resumes. There is no "write queued but not yet committed" state.

### Preferences DataStore setup

Add the dependency:

```kotlin
// build.gradle.kts
implementation("androidx.datastore:datastore-preferences:1.1.x")
```

The idiomatic singleton declaration is a top-level extension property on `Context`:

```kotlin
val Context.dataStore: DataStore<Preferences> by preferencesDataStore(name = "settings")
```

In practice, expose it through a Hilt module so repositories can inject it cleanly and the store is replaceable in tests:

```kotlin
@Provides
@Singleton
fun provideDataStore(@ApplicationContext context: Context): DataStore<Preferences> =
    context.dataStore
```

### Reading a key as Flow

Define keys with the type-safe factory functions â€” the return type is encoded in the key:

```kotlin
private val WEEK_STARTS_MONDAY = booleanPreferencesKey("week_starts_monday")
```

Expose the value as `Flow` from a repository:

```kotlin
val weekStartsMonday: Flow<Boolean> =
    dataStore.data
        .map { preferences -> preferences[WEEK_STARTS_MONDAY] ?: false }
        .catch { e ->
            if (e is IOException) emit(false) else throw e
        }
```

The `.catch` handles a corrupted or missing backing file on first access â€” without it, an `IOException` would terminate the flow.

### Writing a key

```kotlin
suspend fun setWeekStartsMonday(value: Boolean) {
    dataStore.edit { preferences ->
        preferences[WEEK_STARTS_MONDAY] = value
    }
}
```

`edit` is a `suspend` function that acquires a lock, runs your lambda, and flushes atomically before resuming. Unlike `SharedPreferences.edit().putBoolean(â€¦).apply()`, there is no risk of the write being lost before the process dies.

### Proto DataStore

Preferences DataStore keys are typed by value â€” `booleanPreferencesKey("week_starts_monday")` yields `Preferences.Key<Boolean>`, so reading a Boolean key with an Int accessor is caught at compile time. What the type system cannot catch is a typo in the key-name string itself (`"week_starts_mondy"` compiles fine), and there is no versioned schema. Proto DataStore replaces the string-keyed approach entirely: it serialises a Protocol Buffer message defined in a `.proto` file, giving you a generated, versioned, fully typed schema. The tradeoff is the additional build step and `.proto` toolchain. For a handful of boolean/string/int settings, Preferences DataStore is the right choice; reach for Proto DataStore when settings grow complex enough that a typed schema and explicit versioning pay for themselves.

### What changed since 2021

- **`SharedPreferences` is effectively deprecated for new code.** Since DataStore hit 1.0 stable (late 2021) Google's guidance has been to migrate existing stores and use DataStore for all new development.
- **No synchronous read path.** There is no `getBoolean(key, false)` â€” the first value arrives as the first emission of the `Flow`, which is fast but always async. Design your initial UI state to handle "not yet loaded" (e.g., a sensible default in the `stateIn` initial value).
- **No `apply()` / `commit()` distinction.** There is a single write path: `suspend edit { }`. The only "wait" you need to reason about is the coroutine suspension point.

## Drill

Work in the capstone project.

**Part A â€” define the key and repository**

Create a `SettingsRepository` interface with `weekStartsMonday: Flow<Boolean>` and `suspend fun setWeekStartsMonday(value: Boolean)`. Implement it by injecting `DataStore<Preferences>`. Include the `.catch` guard on the `data` flow. Register the `@Binds` mapping in a Hilt module.

**Part B â€” surface in the ViewModel**

In a settings ViewModel (or extend an existing one), collect `settingsRepository.weekStartsMonday` into a `StateFlow` using `stateIn` with `WhileSubscribed(5_000)` and an appropriate initial value. Expose an `onWeekStartsMondayChanged(Boolean)` handler that calls the repository's `suspend` function in `viewModelScope`.

**Part C â€” wire to the UI**

Add a `Switch` (or `Checkbox`) composable to a settings screen, or a settings section of the existing main screen. Bind its checked state to the `StateFlow` value; call the event handler on toggle. Force-stop and relaunch the app â€” confirm the setting survives.

> [!example] Success criteria
> - [ ] The setting persists across a force-stop and relaunch.
> - [ ] No `SharedPreferences`, `getSharedPreferences`, or `PreferenceManager` reference exists anywhere in the project.
> - [ ] `weekStartsMonday` is exposed as `Flow<Boolean>`; the ViewModel reads it via `stateIn`.
> - [ ] The write path calls `dataStore.edit { }` inside a `suspend` function â€” no `apply()` or `commit()` exists.
> - [ ] A corrupted or missing DataStore file does not crash the app (the `.catch` guard is present).
