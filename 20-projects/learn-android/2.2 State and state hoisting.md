# 2.2 State and state hoisting

> [!abstract] TL;DR
> Compose state is explicit and observable: `remember` + `mutableStateOf` keeps values across recompositions, and **state hoisting** ‚Äî moving state up to the caller ‚Äî keeps composables reusable and testable.

## Digest

### remember and mutableStateOf

In Compose, a local variable in a composable function is recreated every recomposition ‚Äî its value is lost. `remember` is the escape hatch: it executes its lambda *once* (on first composition) and returns the same object on every subsequent recomposition.

To make that remembered value *reactive* ‚Äî i.e., cause the composable to recompose when it changes ‚Äî wrap it in `mutableStateOf`:

```kotlin
@Composable
fun Counter() {
    var count by remember { mutableStateOf(0) }
    Button(onClick = { count++ }) {
        Text("Count: $count")
    }
}
```

The `by` delegation (`import androidx.compose.runtime.getValue / setValue`) lets you read and write `count` as if it were a plain `Int` rather than a `State<Int>` wrapper.

### rememberSaveable

`remember` survives recomposition but not configuration changes (screen rotation, process death). Use `rememberSaveable` to also survive those ‚Äî it uses the saved instance state bundle under the hood:

```kotlin
var name by rememberSaveable { mutableStateOf("") }
```

Saveable state works automatically for types that have a `Saver` (primitives, strings, `Parcelable`). For custom types, you write a custom `Saver` or annotate with `@Parcelize`.

Use `rememberSaveable` for user-entered values and scroll positions. Use plain `remember` for derived or transient state (animation targets, dropdown expanded state) that should reset on rotation.

### Unidirectional data flow

Compose enforces a discipline called **unidirectional data flow (UDF)**: state flows *down* (from parent to child as parameters), and events flow *up* (from child to parent as lambda callbacks). A composable should never own state it doesn't need to own.

```
ViewModel
    ‚îÇ state (StateFlow)
    ‚ñº
Screen composable
    ‚îÇ data parameters
    ‚ñº
Child composable ‚îÄ‚îÄ‚îÄ‚îÄ events (lambdas) ‚îÄ‚îÄ‚îÄ‚ñ∫ Screen composable ‚îÄ‚îÄ‚ñ∫ ViewModel
```

This makes it straightforward to predict what will render for any given state, and to test composables by simply calling them with different arguments.

### State hoisting

**State hoisting** is the pattern of taking state out of a composable and moving it up to its caller. The result is a **stateless composable** ‚Äî one that accepts its current value and an event callback, owns nothing internally:

```kotlin
// Stateless ‚Äî accepts state and events as parameters
@Composable
fun Counter(count: Int, onIncrement: () -> Unit) {
    Button(onClick = onIncrement) {
        Text("Count: $count")
    }
}

// The caller owns the state
@Composable
fun CounterScreen() {
    var count by remember { mutableStateOf(0) }
    Counter(count = count, onIncrement = { count++ })
}
```

A stateless composable is:
- **Reusable** ‚Äî the caller decides what the count is and what incrementing means.
- **Testable** ‚Äî you call it with specific values; no internal state to set up.
- **Previewable** ‚Äî every preview just passes hardcoded arguments.

A composable should hold state internally only when nothing outside cares about it (e.g., an animation that's purely cosmetic).

### What changed since 2021

- **No `ViewModel.observe` / `LiveData.observe` in the UI layer.** That wiring is gone; the ViewModel exposes `StateFlow`, the composable collects it (see [[2.7 Side effects and lifecycle]]). The UI no longer holds `Observer` lambdas.
- **State is a first-class Kotlin value.** In XML/Fragment UI you mutated views imperatively; here you mutate a `MutableState` and Compose redraws what needs to change.
- **State hoisting is the idiomatic pattern** ‚Äî replacing the old `TextWatcher`/`setOnClickListener` on view references. The mental model is now: data down, events up.

## Drill

Work in the Compose project from the previous chapter.

**Part A ‚Äî stateful counter**

Write a `Counter` composable that manages its own count using `remember { mutableStateOf(0) }`. It should show the current count and a button to increment it. Embed it directly in `setContent { }` and confirm it works.

**Part B ‚Äî hoist the state**

Refactor `Counter` into a **stateless** composable: `Counter(count: Int, onIncrement: () -> Unit)`. The stateless version must contain zero calls to `remember` or `mutableStateOf`. Create a `CounterScreen` composable that owns the state and passes it down to `Counter`. Replace the `setContent { }` body with `CounterScreen()`.

**Part C ‚Äî survive rotation**

Change the state holder in `CounterScreen` from `remember` to `rememberSaveable`. Rotate the emulator (Ctrl+F12) and confirm the count survives.

> [!example] Success criteria
> - [ ] Part A: tapping the button increments the visible count without a page reload.
> - [ ] Part B: `Counter(count, onIncrement)` contains no `remember` or `mutableStateOf` ‚Äî verified by inspection.
> - [ ] Part B: `CounterScreen` owns the `count` state and passes it to `Counter`.
> - [ ] Part C: count value is preserved after a configuration change (rotation).
> - [ ] No `View` references, no `LiveData`, no XML in any file touched.
