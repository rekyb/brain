# 2.1 The Compose mental model

> [!abstract] TL;DR
> Jetpack Compose replaces XML layouts with Kotlin functions: UI is described as `UI = f(state)`, and the framework rerenders only the parts that changed when state changes.

## Digest

### Declarative vs imperative

In 2021, you built a screen by inflating XML, holding references to Views, and mutating them in response to events — `textView.text = name`, `button.isEnabled = isReady`. The UI was an object graph you mutated in place. This is **imperative** UI: describe *how* to change the screen.

Compose is **declarative**: you describe *what* the screen looks like for any given state, and let the framework figure out the mutations. There are no View references, no `setText`, no `setVisibility`. Instead you write a function that takes data and emits UI:

```kotlin
@Composable
fun Greeting(name: String) {
    Text("Hello, $name!")
}
```

When `name` changes, Compose re-executes the function and reconciles the output — updating only what's different. You never touch the result directly.

### @Composable functions

`@Composable` is a compiler annotation that enables the Compose runtime to track the function's output and manage recomposition. A few rules fall out of this:

- Composable functions can only be called from other composable functions (or a `setContent { }` block).
- They must be **fast and side-effect-free** in their body — no network calls, no disk I/O, no launching coroutines directly (those belong in effect APIs covered in [[2.7 Side effects and lifecycle]]).
- They can call non-composable Kotlin freely; the annotation only gates *emitting UI*.

By convention, composable functions that emit UI are named in PascalCase like classes (`ProfileCard`), not camelCase like functions.

### Recomposition: UI = f(state)

Compose tracks which state values a composable read during its last execution. When any of those values change, Compose **recomposes** — re-executes — only the affected composables, not the entire tree. This is called **smart recomposition** and is what makes the model efficient.

```kotlin
@Composable
fun Counter(count: Int, onIncrement: () -> Unit) {
    Column {
        Text("Count: $count")          // recomposes when count changes
        Button(onClick = onIncrement) {
            Text("Increment")           // never recomposes — no state reads here
        }
    }
}
```

`Text("Increment")` is stable — it reads nothing that changes — so Compose skips it on subsequent recompositions. The framework's default behaviour is to skip composables whose inputs haven't changed.

### Previews

Compose ships a `@Preview` annotation that renders composables directly in Android Studio's design panel, without a running device or emulator. A preview is itself a composable:

```kotlin
@Preview(showBackground = true, name = "Light mode")
@Composable
fun ProfileCardPreview() {
    ProfileCard(name = "Reky", onFollow = {})
}
```

You can stack multiple `@Preview` annotations on the same function (different themes, font scales, locales). Previews live in the same file as the composable or in a separate `*Previews.kt` file — they're stripped from the release build.

### What changed since 2021

- **No XML layouts.** The entire `res/layout/` tree is gone. `setContentView(R.layout.activity_main)` is replaced by `setContent { YourRootComposable() }`.
- **No View references.** `findViewById`, `ViewBinding`, and `DataBinding` are all unnecessary. State drives the UI; you never hold a pointer to a widget.
- **Kotlin-only.** Compose has no Java API; it requires Kotlin and leverages compiler plugins. The Compose compiler is a separate Kotlin compiler plugin configured in `build.gradle.kts`.
- **`@Preview` replaces the XML layout editor** for iterating on UI. The live-edit feature (recent Android Studio versions) can even hot-reload composable changes into a running emulator.
- **Fragments are optional.** Compose screens are composable functions. Navigation between them is handled by `NavHost` (see [[2.6 Navigation in Compose]]), not the Fragment back stack.

## Drill

Work in an Android project with Compose configured (the scaffold from Phase 0 works). You are not allowed to create any XML layout files.

**Task:** Build a `ProfileCard` composable that displays:
- A circular avatar placeholder (use `Box` with a `background` and a fixed size — no image loading library needed).
- A display name (text, larger than body).
- A subtitle (e.g. "Android Developer", smaller/secondary style).
- A "Follow" button that calls an `onClick` lambda.

Wire it into `setContent { }` in `MainActivity` with sample data. Add a `@Preview` annotated composable that renders `ProfileCard` with hardcoded values so it shows in the design panel without a device.

> [!example] Success criteria
> - [ ] `ProfileCard` is a `@Composable` function with no `View` or XML references anywhere in the file.
> - [ ] The preview renders in Android Studio's design panel with the correct layout (avatar + name + subtitle + button).
> - [ ] The app runs on an emulator and shows the card (no crashes).
> - [ ] No `res/layout/` files exist; `setContentView` is gone from `MainActivity`.
> - [ ] `@Preview` uses `showBackground = true` and displays meaningful sample data.
