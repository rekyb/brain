# 3.4 Repository pattern and data layer

> [!abstract] TL;DR
> The repository is the data layer's public API — an interface the ViewModel depends on, implemented once (in-memory now, Room in Phase 4), exposing `Flow` so the UI updates reactively without any polling or manual refresh.

## Digest

### Repository as single source of truth

A repository's job is to be the **single source of truth** for a data domain: it decides where data comes from, reconciles multiple sources if needed (local cache + remote API), and exposes a stable, typed interface upward. Neither the ViewModel nor any composable accesses a DAO, Retrofit service, or DataStore key directly — everything goes through the repository.

This constraint pays off immediately: when the data source changes (you swap an in-memory list for a Room DAO in Phase 4), only the repository implementation changes. The ViewModel, its tests, the composables — none of them notice.

### Interface + implementation

Always define the repository as an interface:

```kotlin
interface HabitRepository {
    fun getHabits(): Flow<List<Habit>>
    suspend fun addHabit(name: String)
    suspend fun deleteHabit(id: Long)
}
```

Implement it separately. For Phase 3, a `MutableStateFlow`-backed in-memory store is enough:

```kotlin
class HabitRepositoryImpl @Inject constructor() : HabitRepository {

    private val _habits = MutableStateFlow<List<Habit>>(emptyList())

    override fun getHabits(): Flow<List<Habit>> = _habits.asStateFlow()

    override suspend fun addHabit(name: String) {
        _habits.update { current ->
            current + Habit(id = System.currentTimeMillis(), name = name)
        }
    }

    override suspend fun deleteHabit(id: Long) {
        _habits.update { it.filterNot { habit -> habit.id == id } }
    }
}
```

In Phase 4 the `MutableStateFlow` backing store is replaced by a Room DAO. The interface stays identical — the ViewModel never changes.

The ViewModel depends only on `HabitRepository` (the interface); Hilt injects `HabitRepositoryImpl` via the `@Binds` module from chapter 3.3.

### Exposing Flow

Repositories expose `Flow`, not `List`. This is the reactive contract: when the underlying data changes, all active collectors automatically receive the updated value. No polling, no callbacks, no `notifyDataSetChanged`.

The ViewModel collects from the repository inside `viewModelScope`:

```kotlin
init {
    viewModelScope.launch {
        repository.getHabits().collect { habits ->
            _uiState.update { it.copy(habits = habits) }
        }
    }
}
```

For a cleaner ViewModel, use `stateIn` to convert the repository's cold `Flow` directly into the ViewModel's hot `StateFlow`:

```kotlin
val uiState: StateFlow<HabitListUiState> =
    repository.getHabits()
        .map { habits -> HabitListUiState(habits = habits) }
        .stateIn(
            scope = viewModelScope,
            started = SharingStarted.WhileSubscribed(5_000),
            initialValue = HabitListUiState(isLoading = true),
        )
```

`SharingStarted.WhileSubscribed(5_000)` keeps the upstream active for 5 seconds after the last collector disappears — long enough to survive a configuration change without restarting the repository query.

### Mapping data to domain models

The data layer may use different model types than the rest of the app. A Room entity carries database annotations (`@Entity`, `@ColumnInfo`); your domain `Habit` data class should have none of those. The repository is responsible for translating:

```kotlin
// Domain model — no framework annotations, no database concepts
data class Habit(val id: Long, val name: String, val isCompleted: Boolean)

// Phase 4: repository maps entity ↔ domain at the boundary
private fun HabitEntity.toDomain() = Habit(id, name, isCompleted)
private fun Habit.toEntity() = HabitEntity(id, name, isCompleted)
```

For Phase 3 the in-memory implementation works directly with domain objects — the mapping layer materialises when Room entities arrive in Phase 4.

### What changed since 2021

- **Repositories now expose `Flow<T>` instead of `LiveData<T>`.** The ViewModel collects inside `viewModelScope`; the composable collects from the ViewModel's `StateFlow`. The entire chain is pure Kotlin — no Android framework types in the data layer.
- **`stateIn` / `shareIn`** (coroutines 1.4+) make it practical to share a single cold Flow across multiple collectors without restarting the upstream — previously you'd manage this with `MediatorLiveData` or channels.
- **Interface-first design is the expected norm**, because Hilt's `@Binds` makes it trivially cheap to swap implementations. In 2021 you might have skipped the interface; the architecture guide now calls it out explicitly.
- **Room returns `Flow<List<Entity>>`** from DAO queries (since Room 2.2), so you get live-updating database results for free — the repository just exposes them up the chain.

## Drill

Work in the capstone project.

**Part A — define the interface**

Create a `HabitRepository` interface in a data layer package. Declare at minimum: `getHabits()` returning `Flow<List<Habit>>` and `addHabit(name: String)` as a `suspend` function. Think ahead to what else the capstone will need (delete? toggle completion?) and add those declarations now.

**Part B — implement in-memory**

Write `HabitRepositoryImpl`. Back it with a private `MutableStateFlow<List<Habit>>`. Implement `getHabits()` to expose that state as a read-only flow. Implement `addHabit` using `.update { }` to append to the list. Annotate the constructor with `@Inject`. Register the interface→impl binding in the Hilt module from 3.3 (if you haven't already).

**Part C — back the ViewModel**

Remove any hardcoded data from `HabitListViewModel`. Replace it by collecting from `repository.getHabits()` (use `viewModelScope.launch { ... .collect { } }` or `stateIn`). Ensure `onEvent(AddHabit(...))` calls `repository.addHabit(...)` in a `viewModelScope` coroutine. Run the app: adding a habit from the UI should update the list without a screen restart.

> [!example] Success criteria
> - [ ] `HabitListViewModel` imports only `HabitRepository` (the interface) — `HabitRepositoryImpl` appears nowhere in the ViewModel file.
> - [ ] `getHabits()` returns `Flow<List<Habit>>`; the ViewModel never reads a list or DAO value directly.
> - [ ] Adding a habit from the UI updates the displayed list reactively — no manual refresh triggered.
> - [ ] Rotating the screen preserves the habit list (ViewModel survives configuration change).
> - [ ] The repository file contains no Compose or ViewModel imports.

## Capstone step

Wire the full reactive stack for the habit tracker:

- `HabitListScreen` (Compose) collects `uiState` with `collectAsStateWithLifecycle()` and sends events via `onEvent()`.
- `HabitListViewModel` (`@HiltViewModel`) holds `StateFlow<HabitListUiState>` backed by the repository, handles events in `viewModelScope`.
- `HabitRepositoryImpl` (injected via Hilt `@Binds`) holds an in-memory `MutableStateFlow<List<Habit>>` as the data source.

**Acceptance criteria:**

- The screen launches and displays a placeholder or empty state.
- Entering a habit name and confirming adds it to the list without restarting the screen.
- The habit list survives a configuration change (screen rotation).
- All three layers are cleanly separated: the composable knows only `UiState` and events; the ViewModel knows only `HabitRepository` (interface) and `HabitListUiState`; the repository knows only `Habit` domain objects.
- Data persistence is deliberately deferred — the in-memory store resets on process death, and that is expected. Persistence comes in [[4.1 Room]].
