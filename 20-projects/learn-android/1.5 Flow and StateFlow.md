# 1.5 Flow and StateFlow

> [!abstract] TL;DR
> `Flow` is Kotlin's reactive stream — cold, composable, and cancellation-aware; `StateFlow` is its hot, always-valued sibling that replaced `LiveData` as the ViewModel-to-UI communication layer.

## Digest

### Cold Flow vs hot StateFlow

A **cold** `Flow` doesn't produce values until someone collects it, and each collector gets its own independent execution:

```kotlin
val ticker: Flow<Int> = flow {
    var n = 0
    while (true) {
        emit(n++)
        delay(1_000)
    }
}
// Nothing happens until .collect { } is called.
// Two separate collectors each receive 0, 1, 2, ... independently.
```

A **hot** flow produces values regardless of collectors. `StateFlow` is hot, always holds exactly one value (the current state), and any late collector immediately receives the current value:

```kotlin
private val _count = MutableStateFlow(0)
val count: StateFlow<Int> = _count.asStateFlow()   // expose read-only

fun increment() { _count.update { it + 1 } }
```

`SharedFlow` is another hot flow without the "always holds one value" guarantee. It's useful for one-shot events (navigation commands, error toasts) where you don't want a late collector replaying the last event.

### Building a Flow

The `flow { }` builder lets you emit values lazily with `emit()`. Use `flowOf(...)` for a fixed sequence or `List.asFlow()` to convert a collection:

```kotlin
val numbers: Flow<Int> = flowOf(1, 2, 3, 4, 5)
val fromList: Flow<String> = listOf("a", "b", "c").asFlow()

val timed: Flow<Int> = flow {
    for (i in 1..10) {
        emit(i)
        delay(100)
    }
}
```

### Operators

Flow operators are lazy — they don't execute until collection:

```kotlin
flow.map { it * 2 }
flow.filter { it > 0 }
flow.take(5)
flow.debounce(300)          // drop emissions within 300 ms windows — useful for search input
flow.distinctUntilChanged() // suppress duplicate consecutive values
flow.flatMapLatest { q ->   // cancel the previous inner flow when a new value arrives
    searchApi(q)
}
```

`combine` merges two flows, emitting a combined value whenever either updates:

```kotlin
val combined: Flow<String> = combine(nameFlow, ageFlow) { name, age ->
    "$name ($age)"
}
```

### Collecting

`collect` is a `suspend` function that runs in a coroutine and suspends until the flow completes or is cancelled:

```kotlin
viewModelScope.launch {
    habits.collect { list ->
        _uiState.update { it.copy(habits = list) }
    }
}
```

In Compose, use `collectAsStateWithLifecycle()` (from `lifecycle-runtime-compose`) rather than `collectAsState()` — it pauses collection when the composable is not in the foreground, saving resources. You will use this in [[2.7 Side effects and lifecycle]].

### StateFlow as the LiveData replacement

`LiveData` was lifecycle-aware but Android-only and hard to test without Robolectric. `StateFlow` is pure Kotlin — testable without a device, composable with Flow operators, and lifecycle awareness is handled at the collection site.

```kotlin
class HabitViewModel : ViewModel() {
    private val _habits = MutableStateFlow<List<Habit>>(emptyList())
    val habits: StateFlow<List<Habit>> = _habits.asStateFlow()
}

// In Compose:
val habits by viewModel.habits.collectAsStateWithLifecycle()
```

Use `update { }` for atomic state mutations — it does a compare-and-set, safe for concurrent updates:

```kotlin
_habits.update { current -> current + newHabit }
```

Avoid direct `.value = newValue` in shared mutable state; prefer `update { }`.

### What changed since 2021

- **`Flow` replaces `LiveData`** for ViewModel→UI communication in new code. `LiveData` is not removed but not recommended for new development; the guides now start with `StateFlow`.
- **`collectAsStateWithLifecycle()`** (androidx.lifecycle 2.6+) is the Compose-idiomatic way to collect a `Flow` — it is lifecycle-aware. Plain `collectAsState()` collects even in the background.
- **`StateFlow.update { }`** (coroutines 1.5+) is the safe atomic update method. Prefer it over direct `.value =` assignment.
- **`SharedFlow`** was introduced for events and broadcasts, replacing `Channel` and the deprecated `BroadcastChannel` for most use cases.
- **`stateIn` / `shareIn`** convert a cold `Flow` into a hot `StateFlow`/`SharedFlow`, letting multiple collectors share a single upstream without restarting it.
- **Room returns `Flow<List<T>>`** from DAO queries automatically — you get live-updating database results for free, piped straight into `StateFlow` via `stateIn`.

## Drill

Work in **Kotlin Playground** or a scratch `.kt` file. You will need `kotlinx-coroutines-core`.

**Part A — cold Flow pipeline**

Create a `Flow<Int>` that emits the integers 1 through 10 with a 100 ms `delay` between each. Apply two operators in sequence:

1. Keep only even numbers (`filter`).
2. Multiply each by 10 (`map`).

Collect the transformed flow inside `runBlocking` and print each value as it arrives. Print the total elapsed time after collection completes.

**Part B — MutableStateFlow counter**

Model a simple counter. Start from this structure — implement only the function bodies:

```kotlin
val counter: MutableStateFlow<Int> = MutableStateFlow(0)

fun increment() { TODO() }
fun decrement() { TODO() }
fun reset()     { TODO() }
```

Inside a `runBlocking` block:

1. Launch a coroutine that collects `counter` and prints every new value as it arrives.
2. From the main coroutine, call `increment()` three times, `decrement()` once, and `reset()` once. Insert a `delay(50)` between each call so the collector has time to print.
3. After `reset()`, cancel the collector coroutine.
4. Call `increment()` one more time and confirm the collector prints nothing (it was cancelled).

> [!example] Success criteria
> - [ ] Part A: only values 20, 40, 60, 80, 100 appear in the output; total elapsed time is ~500 ms.
> - [ ] Part B: the collector prints a new value for each state change in order.
> - [ ] Part B: the final value printed before cancellation is `0` (after `reset()`).
> - [ ] Part B: the post-cancellation `increment()` produces no output from the collector.
> - [ ] `update { }` used for all mutations in Part B, not direct `.value =` assignment.
> - [ ] No `LiveData` imports anywhere.
