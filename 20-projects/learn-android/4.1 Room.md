# 4.1 Room

> [!abstract] TL;DR
> Room is Android's SQLite abstraction: annotate an entity and DAO, build a database singleton, and get reactive `Flow`-returning queries and coroutine-safe writes with no raw cursor management.

## Digest

### The three pillars: Entity, DAO, Database

Room works through three annotated classes.

`@Entity` maps a Kotlin data class to a database table. Each property becomes a column; annotate with `@ColumnInfo` to customise names or `@Ignore` to skip a field.

```kotlin
@Entity(tableName = "habits")
data class HabitEntity(
    @PrimaryKey(autoGenerate = true) val id: Long = 0,
    val name: String,
    val isCompleted: Boolean = false,
    val streak: Int = 0,
    val reminderTime: Int? = null, // minutes-of-day; null = no reminder (use a TypeConverter to map to/from LocalTime)
)
```

`@Dao` is an interface whose functions Room implements at compile time. Write queries as SQL in `@Query`; write mutations as `@Insert`, `@Update`, or `@Delete`.

```kotlin
@Dao
interface HabitDao {
    @Query("SELECT * FROM habits ORDER BY id ASC")
    fun getAll(): Flow<List<HabitEntity>>

    @Insert(onConflict = OnConflictStrategy.REPLACE)
    suspend fun insert(habit: HabitEntity)

    @Delete
    suspend fun delete(habit: HabitEntity)
}
```

Return `Flow<T>` for queries you want to observe: Room re-emits whenever the relevant table changes, so the UI always reflects current data without polling. Return nothing (or `Long`/`Int`) from `suspend` mutations тАФ the `suspend` modifier routes them off the main thread automatically.

`@Database` ties everything together. Extend `RoomDatabase`, list the entities, set a version number, and expose abstract DAO accessors.

```kotlin
@Database(entities = [HabitEntity::class], version = 1, exportSchema = true)
abstract class AppDatabase : RoomDatabase() {
    abstract fun habitDao(): HabitDao
}
```

Build the singleton in a Hilt module so it's injected rather than hand-constructed:

```kotlin
@Provides
@Singleton
fun provideDatabase(@ApplicationContext context: Context): AppDatabase =
    Room.databaseBuilder(context, AppDatabase::class.java, "app.db")
        .build()

@Provides
fun provideHabitDao(db: AppDatabase): HabitDao = db.habitDao()
```

### Migrations

Room tracks schema versions. When you change the schema, increment `version` and supply a `Migration` that expresses the SQL delta:

```kotlin
val MIGRATION_1_2 = object : Migration(1, 2) {
    override fun migrate(db: SupportSQLiteDatabase) {
        db.execSQL("ALTER TABLE habits ADD COLUMN priority INTEGER NOT NULL DEFAULT 0")
    }
}
```

For straightforward structural changes тАФ adding a nullable column, adding a new table тАФ Room 2.4+ supports **auto-migrations**: declare `@AutoMigration(from = 1, to = 2)` in the `@Database` annotation and Room generates the migration SQL at compile time. Manual migrations are still required for column renames or row-data transforms.

During development, `fallbackToDestructiveMigration()` drops and recreates the database on a schema mismatch тАФ fast, but destroys user data. Restrict this to debug builds.

### Type converters

SQLite stores primitives, strings, and blobs. For custom types (enums, `Instant`, complex objects), write a `@TypeConverter` pair and register it on the database class:

```kotlin
class Converters {
    @TypeConverter
    fun fromInstant(value: Instant?): Long? = value?.toEpochMilli()

    @TypeConverter
    fun toInstant(value: Long?): Instant? = value?.let { Instant.ofEpochMilli(it) }
}

@TypeConverters(Converters::class)
@Database(...)
abstract class AppDatabase : RoomDatabase() { ... }
```

Keep entities flat where you can тАФ type converters obscure query-ability and complicate migrations.

### What changed since 2021

- **KSP replaces kapt.** Room's annotation processor now runs on Kotlin Symbol Processing, which is substantially faster. In `build.gradle.kts`, replace `kapt("androidx.room:room-compiler:тАж")` with `ksp("androidx.room:room-compiler:тАж")`.
- **`Flow`-returning DAO queries.** Up to 2021, `LiveData<T>` was still the common observable return type even in Room 2.2+ projects; `Flow<T>` is now standard тАФ no Android framework types leak into the data layer, and the flow integrates directly into the coroutine/StateFlow chain from Phase 3.
- **Auto-migrations** (Room 2.4+) handle straightforward schema changes without hand-written SQL.
- **Kotlin Multiplatform** support is progressing (Room 2.7+). KMP is out of scope here, but it is why KSP was a prerequisite тАФ Room's internals are moving toward pure-Kotlin.

## Drill

Work in the capstone project.

**Part A тАФ entity and DAO**

Create `HabitEntity` as a Room `@Entity` mirroring the canonical `Habit` domain model: `id` (auto-generated `Long` primary key), `name`, `isCompleted`, `streak`, and `reminderTime` (stored as `Int?` minutes-of-day with a TypeConverter). Create a `HabitDao` interface with: `getAll()` returning `Flow<List<HabitEntity>>`, a `suspend fun insert(habit: HabitEntity)`, and a `suspend fun delete(habit: HabitEntity)`.

**Part B тАФ database and Hilt wiring**

Create `AppDatabase` extending `RoomDatabase`, listing `HabitEntity` in `entities` at `version = 1`. In a Hilt module, provide the database singleton via `databaseBuilder` and expose `HabitDao`. Make sure `exportSchema = true` and point the KSP argument at a `schemas/` directory so the schema JSON is tracked.

**Part C тАФ swap the data source**

Remove the `MutableStateFlow<List<Habit>>` backing store from `HabitRepositoryImpl`. Delegate `getHabits()` to `habitDao.getAll()`, mapping `HabitEntity тЖТ Habit` with an extension function. Delegate `addHabit` / `deleteHabit` / any other mutations to the corresponding `suspend` DAO functions.

The ViewModel and its interface should require zero changes.

> [!example] Success criteria
> - [ ] Habits added in one session are still present after a force-stop and relaunch.
> - [ ] `HabitDao.getAll()` returns `Flow<List<HabitEntity>>`; the repository maps entities to domain `Habit` objects before exposing them upstream.
> - [ ] `HabitRepositoryImpl` contains no `MutableStateFlow` тАФ the reactive stream originates from Room.
> - [ ] `HabitListViewModel` is unchanged from Phase 3 (it depends only on the `HabitRepository` interface).
> - [ ] The Hilt module provides `AppDatabase` as `@Singleton`; `HabitDao` is derived from that singleton.
