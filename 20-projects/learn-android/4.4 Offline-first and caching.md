# 4.4 Offline-first and caching

> [!abstract] TL;DR
> Offline-first means the UI always observes the local database — the network is only a writer behind it. Data flows network → Room → UI, never network → UI directly.

## Digest

### The single-source-of-truth pattern

An offline-first feature has one rule: **the UI observes the database, not the network response.** The repository never routes a network result straight to a `MutableStateFlow` or directly into UI state.

The data flow is always:

1. The ViewModel (or a lifecycle trigger) asks the repository to refresh.
2. The repository calls the `suspend` network function.
3. On success, the repository writes the result into Room.
4. Room's `Flow`-returning DAO emits automatically because the table changed.
5. The ViewModel's `stateIn`-bound `StateFlow` delivers the updated value to the composable.

The composable never touches a network response type. It only ever sees domain models, emitted by the Room-backed `Flow` from step 4.

### Why it matters offline

When step 2 fails — no connectivity, timeout, server error — step 3 is never reached. Step 4 therefore emits nothing new. The UI continues to display whatever Room last persisted, which is correct: stale data is shown with an error indicator, not a crash or a blank screen.

The error is communicated separately: the repository's refresh function returns `Result<Unit>` (or throws and is caught in the ViewModel), and the ViewModel surfaces it as an `error: String?` field in `UiState`. The Room-backed Flow and the error signal are independent — one carries data, the other carries failure state.

### Cache invalidation

The simplest policy is time-based expiry. Store a `lastFetchedAt` timestamp (epoch millis) alongside the cached data — in a Room column, or in a DataStore key if the timestamp is per-feature rather than per-row. Before making a network call, check whether the stored value is younger than your threshold; if so, skip the fetch and let Room serve from cache.

More sophisticated strategies (ETags, `Cache-Control` headers, server-side cursors) add complexity without proportional benefit at small scale. Keep the policy simple until a product requirement demands more.

### When to reach for Paging 3

Paging 3's `RemoteMediator` formalises exactly the network → Room → UI pattern described here, but applied page by page. It is worth reaching for when:

- The list has hundreds or thousands of items that shouldn't all be in memory at once.
- The server API uses cursor- or offset-based pagination.
- You need a "loading next page" indicator as the user scrolls toward the end.

For the capstone app — tens of habits, a single daily quote — Paging 3 is unnecessary overhead. Know it exists; revisit it when the list is genuinely large and paginated.

### What changed since 2021

The offline-first pattern itself predates modern Android, but several things make it substantially less painful now:

- **Room's `Flow`-returning DAOs** (Room 2.2+) eliminate manual invalidation and `notifyDataSetChanged`. The UI just collects; Room notifies on every table write.
- **Coroutines** replace `AsyncTask` / `Executor` for the write step. A single `suspend` function in the repository fetches from the network and inserts into Room — no callback nesting, no thread bookkeeping.
- **Paging 3** (2020) introduced `RemoteMediator` as a first-class integration point for network-backed Room lists, replacing the fragile `BoundaryCallback` from Paging 2.

## Drill

This is a **design drill** — write in Markdown; no implementation code is required.

In a scratch file in your project notes (or directly in the capstone's `docs/` folder), sketch the offline-first design for the optional daily-quote feature:

1. **Data flow** (prose or ASCII): describe every hop from the network call to the composable that displays the quote. Name the classes involved at each hop (API interface, repository, DAO, ViewModel, composable).
2. **Failure handling**: describe what the repository does when the network call throws. What does the ViewModel surface in `UiState`? What does the user see?
3. **Cache policy**: state your chosen freshness threshold (e.g., "fetch at most once per day") and where you store the `lastFetchedAt` value — a DataStore key, a Room column, or something else.
4. **Room schema**: name the entity that stores the quote and list its key columns.

> [!example] Success criteria
> - [ ] The design places Room — not a `MutableStateFlow` or the raw network response — as the source the composable observes.
> - [ ] The network result is written to Room before any new data reaches the ViewModel.
> - [ ] The failure path does not crash the app — the design describes an explicit error state in `UiState`.
> - [ ] A cache-invalidation policy is described with a concrete threshold (not just "check if cached").
> - [ ] The Room entity for the quote is named and its key columns are listed.

## Capstone step

With Phase 4 complete, the capstone has three persistence layers working together.

**Habits — Room**

`HabitRepositoryImpl` reads from and writes to `HabitDao`. All CRUD operations are backed by the Room database. A habit created in session N is still present after a force-stop and cold relaunch in session N+1.

**Settings — DataStore**

At least one user preference (e.g., "week starts on Monday") is stored in Preferences DataStore and exposed as `Flow<T>`. It persists across restarts and is reflected in the settings UI immediately on relaunch.

**Daily quote — optional offline-first feature**

Implement a "Quote of the Day" card (or screen) that applies the pattern from this chapter:

- Fetches from a real public API using a Retrofit `suspend` endpoint.
- Writes the result into a Room entity on a successful fetch.
- Exposes the stored quote via the DAO's `Flow`-returning query — the composable observes Room, not the network response.
- Applies the cache policy from the design drill: if a quote was already fetched within the freshness window, the app skips the network call on relaunch.
- Displays the last-known quote (from Room) while refreshing or when offline.
- Shows an error indicator — not a crash, not a blank screen — when the network call fails and no cached data exists.

**Acceptance criteria (habits and settings — required):**

- Habits survive a force-stop and relaunch.
- The user preference persists across restarts.
- The app does not crash when launched without a network connection.

**Acceptance criteria (optional quote feature, if implemented):**

- A quote is visible in airplane mode after at least one prior successful fetch.
- The first launch with no connectivity shows an error indicator rather than an empty screen.
- Logcat shows the Retrofit network call being skipped on a subsequent launch when the cached quote is still fresh.
